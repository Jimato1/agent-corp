## 0. Identity recap and what this research settles

The agent-runtime is **the workforce** — the platform layer that hosts the agent processes and owns the **client half** of every guardrail: it polls the Board, performs atomic claims, runs the agent loop, writes Notes, emits heartbeats, obeys drain/kill, holds the per-agent TPM-sealed signing keys, and hosts the local model stack. It is **not** one of the four SoD action-holders (Board / CMDB / Vault / Gateway) and must never become one: it holds *identity* key material only, never *host* credentials, and has no approval or execution authority.

This document resolves every deferred decision the app's `CLAUDE.md` and the gap analysis assigned to Stage 1, and names the cross-app seams that must be frozen in `context/CONTRACTS/` before dependent apps plan. Headline recommendations, one line each:

- **Loop:** a **long-lived standing worker per fleet slot** that claims **per-ticket in-process sessions**; durable spine = **Board** (claim/lease/fencing/transition ledger) + **Notes** (checkpoint/resume); the LLM step is a non-deterministic *activity*, so **do not adopt an off-the-shelf durable-execution engine** as substrate. (§1)
- **Liveness:** **two independent signals** — a supervisor `process_alive_ts` and a work-loop `work_progress_ts`+`step_seq` — single-producer SSE stream to MC, with **correlated-loss suppression** so an auth/network outage is never read as mass agent death. (§2)
- **Drain/kill:** one **epoch-versioned `RUN→DRAIN→KILL` machine** whose first act is an unconditional pre-claim gate; client drain is **defense-in-depth over the server-side Gateway/auth guarantee**, and the Stage-7 demo is staged **two-sided** (cooperating runtime *and* simulated rogue agent). (§3)
- **Keys:** **`tpm2-pkcs11`** custody + **`tpm2-openssl`** (OpenSSL-3 provider) signing; keys **born in-TPM, `fixedTPM|fixedParent`, non-exportable**; reach the TPM via **`/dev/tpmrm0`**; scope `gateway:execute` personas to **TPM-attested nodes** (auth #6 escape hatch). (§4)
- **Models:** **vLLM** primary serving engine, **SGLang** alternate, **llama.cpp** CPU/quiesce lane, **TEI** for embeddings; a **fail-closed provenance gate** (safetensors-only, digest-pinned, SHA-256-verified, Sigstore-`model-signing`-verified). (§5)
- **Sizing (§6) and the feasibility spike (§7)** are designed as artifacts and **deferred**; the spike **gates Board/Notes Stage 2**.
- **Session/degraded/persona (§8):** standing pool of pre-enrolled sealed sessions; **credential-independent** auth-outage degraded mode (jittered backoff + circuit breaker + local quiesce + outage-aware reaper + ordered re-entry); **git-tracked, operator-owned persona config** gated by a **per-role behavioral regression suite** with a **planted-flaw re-certification battery for the Adversarial Reviewer**.

## 1. The durable agent loop (poll → atomic-claim → work → heartbeat → transition → checkpoint/escalate)

The runtime owns the **client half** of a durable-worker pattern the distributed-systems and job-runner worlds have already converged on. The atomic claim *is* the Board `todo→in_progress` CAS plus resource lock/lease plus fencing token; the loop must be idempotent against reaper requeue; a stuck agent escalates instead of spinning.

**1.1 The atomic claim is at-least-once, and the runtime is a *client* of it.** The canonical atomic-claim primitive is Postgres `SELECT … FOR UPDATE SKIP LOCKED` (PostgreSQL 9.5+), which lock-dequeues one row while peers skip locked rows in a single race-free statement ([Netdata](https://www.netdata.cloud/academy/update-skip-locked/); [DB Pro](https://www.dbpro.app/blog/postgresql-skip-locked)). This is almost certainly how the Board implements its dequeue — but under the locked SoD rules the Board is the queue authority and **the runtime does not run this query**; it calls a Board claim API and receives, atomically, the `in_progress` transition + lease + fencing token. It must therefore inherit the discipline every claim-queue client carries: **claim-queues are at-least-once, not exactly-once.** SQS standard queues redeliver after a visibility timeout if unacknowledged ([AWS SQS](https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/standard-queues-at-least-once-delivery.html)); Faktory re-enqueues any job whose `reserve_for` reservation (default 1800s, min 60s) elapses without ACK/FAIL, treating a crash as a failure ([Faktory wiki](https://github.com/contribsys/faktory/wiki/Job-Errors)); Celery's `acks_late` is blunt that "tasks with late acknowledgment enabled must be idempotent as they may be executed more than once" ([Celery](https://docs.celeryq.dev/en/stable/userguide/tasks.html)). Maps directly onto the suite reaper: on lease/heartbeat loss the Board moves `in_progress→todo`, requeues, and increments the fencing counter — so **the runtime must assume any ticket it holds may be simultaneously re-claimed**, and every step must be a safe no-op on re-execution.

**1.2 Fencing tokens are the split-brain defense — non-optional on every side-effecting call.** The classic hazard (Kleppmann): a paused/partitioned worker wakes believing it still holds the lease and overwrites work now owned by another. The only robust defense is a **fencing token** — monotonic per lease, recorded as a high-water mark by the *receiving* side, which rejects any stale (lower) token ([Kleppmann, distributed locking](https://martin.kleppmann.com/2016/02/08/how-to-do-distributed-locking.html)). The suite already mints exactly this (Board-issued, monotonic per locked resource). Design consequence: **every side-effecting call — Board transitions, Notes writes, escalations, any Gateway-bound proposal — echoes the current fencing token, and the server rejects stale tokens.** A token generated but never checked provides no safety. This is what makes "idempotent against reaper requeue" true rather than aspirational; key side effects by `(ticket_id, fencing_token, step_id)` so a re-run dedupes or is fenced out (River formalizes this as *unique jobs* — [River](https://brandur.org/river)).

**1.3 The LLM step is a non-deterministic *activity*, not deterministic workflow code.** Temporal's durability rests on **deterministic replay**: a workflow is an append-only event history replayed through a deterministic function, and any divergence halts replay ([Temporal workflow definition](https://docs.temporal.io/workflow-definition)). **An LLM reasoning step is irreducibly non-deterministic and cannot be replayed this way.** Correct mapping: model the LLM/tool step as a Temporal-style *activity* (run-once, at-least-once, output persisted), and make the durable spine the recorded ledger of which steps completed and what they produced. In this suite that ledger already exists and is canonical: **Notes (markdown = truth) is the external memory and checkpoint store.** This is also why an off-the-shelf durable engine is a poor *substrate*: it demands determinism the agent can't provide and stands up a second canonical queue beside the Board — violating "two views, one state."

**1.4 Split the liveness heartbeat from work progress (Temporal's timeout split).** Temporal sets the **heartbeat timeout** much shorter than the **start-to-close timeout**, so a crashed/wedged worker is detected the next minute after a missed heartbeat rather than hours later ([Temporal — four activity timeouts](https://temporal.io/blog/activity-timeouts); [detecting activity failures](https://docs.temporal.io/encyclopedia/detecting-activity-failures)). The runtime therefore emits **two distinct signals**: a fast Board lease-renewal/heartbeat and the MC liveness heartbeat, the latter carrying a progress marker so "no progress" is detectable independent of "no process" (developed fully in §2).

**1.5 Context exhaustion is the agent's "history-size limit" — resume via compacted Notes.** Temporal caps history at a hard **50Ki events / 50MB** and prescribes **Continue-As-New** — atomically finish the current run and start fresh with a compact state snapshot ([Temporal limits](https://docs.temporal.io/workflow-execution/limits); [Continue-As-New](https://docs.temporal.io/workflow-execution/continue-as-new)). The agent's context window is the direct analog: near exhaustion, checkpoint a **compacted structured summary** to Notes, then resume in a fresh session by re-reading that trail — **not** by replaying the raw transcript. LangGraph is the counter-example to avoid: its checkpointer persists state but has **no automatic resumption** (a supervisor must re-invoke) and raises `GraphRecursionError` on runaway loops ([LangGraph persistence](https://docs.langchain.com/oss/python/langgraph/persistence)); Diagrid's critique — bare checkpoints are *not* durable execution — warns the runtime must build resume orchestration explicitly ([Diagrid](https://www.diagrid.io/blog/checkpoints-are-not-durable-execution-why-langgraph-crewai-google-adk-and-others-fall-short-for-production-agent-workflows)).

**1.6 The inner loop is ReAct with hard termination guards (the "never terminating" failure mode).** The work phase is a ReAct Thought→Action→Observation loop ([Yao et al., arXiv:2210.03629](https://arxiv.org/abs/2210.03629)). Because "no API cost" means the failure mode is **never-terminating / confident garbage** rather than spend, the termination guards are load-bearing: a hard iteration cap, an explicit terminal transition the model must call, a stop condition tied to *externally verifiable* state (Board transition confirmed, Notes written, external verifier — not model self-assertion), and a **no-progress detector** that trips the escalation-is-the-default-failure-mode rule (file `needs_review`/`blocked`, never spin or blind-retry).

**Recommendation.** A **long-lived standing worker per fleet slot** that claims per-ticket *in-process sessions* (River's producer/executor split adapted for agents — [River](https://brandur.org/river)): the standing worker owns the poll loop, TPM-key custody (one place), lease renewal, and the MC heartbeat; each ticket runs in an **isolated fresh-context session** so one ticket's exhaustion or poisoning can't leak into the next, and the expensive model-stack load is amortized. **Do not** adopt Temporal (or any durable engine) as the substrate; borrow its *patterns* (heartbeat/timeout split, activity-vs-workflow separation, Continue-As-New) over Board + Notes. **Fence everything.** Guard termination hard.

**Open questions (→ Planning / operator).** Lease TTL, heartbeat cadence, and the timeout split cannot be chosen until §6's spike measures p99 step duration and concurrency — a too-short lease double-executes tickets ([Celery #6229](https://github.com/celery/celery/issues/6229)), a too-long one delays recovery; Planning sets the policy (visibility > p99 + buffer), not a number. Checkpoint-on-abandon under drain is per-ticket-type (§3). Long-lived vs per-ticket session (Open Decision 3) is settled here as *standing pool* pending operator ratification (§8).

## 2. Heartbeat / liveness — the seam Mission Control reads

MC must separate four states that look identical from outside: **crashed** (process gone), **wedged** (process alive, work stuck), **correlated outage** (the whole fleet went quiet at once — *not* N deaths), and **drained** (expected silence after kill/drain). No single signal can do this.

**2.1 Liveness ≠ progress — two signals are mandatory.** Kubernetes already splits "is the process up" (liveness probe — "could catch a deadlock, where an application is running but unable to make progress") from "is it ready" (readiness) from "has it started" (startup), and warns "incorrect implementation of liveness probes can lead to cascading failures" — exactly the fleet-scale misfire to prevent ([Kubernetes probes](https://kubernetes.io/docs/concepts/workloads/pods/probes/)). Temporal's activity heartbeat carries "an application layer payload that can be used to save Activity Execution progress" ([Temporal — detecting activity failures](https://docs.temporal.io/encyclopedia/detecting-activity-failures)). The contract therefore carries **two independent timestamps**: `process_alive_ts` (a cheap supervisor liveness beat) and `work_progress_ts` + a monotonic `step_seq` (emitted from *inside* the agent loop at each Board/Notes interaction). Wedged ⇔ `process_alive_ts` fresh but `work_progress_ts`/`step_seq` stale; crashed ⇔ both stale. **Hazard:** if lease renewal runs on the supervisor thread, a wedged agent renews forever and holds the host lock at zero progress — liveness masking non-progress. **Gate Board lease renewal on the progress predicate** so a wedged agent self-fences and releases the lock, in addition to MC surfacing the wedge.

**2.2 Why "missed N heartbeats = dead" misfires, and the adaptive alternative.** Local-model latency is high and variable (a single step can legitimately take minutes on a busy GPU), and a fixed threshold **mass-misfires under correlated loss** (one blip trips every agent at once). For the *timing* dimension use an **accrual failure detector**: phi-accrual (Cassandra, Akka) outputs a continuous suspicion φ instead of a boolean, fitted to observed inter-arrival history; Akka's default `threshold` is 8, raised toward 12 in noisier environments to trade detection latency for fewer false positives ([Akka phi-accrual](https://doc.akka.io/libraries/akka-core/current/typed/failure-detector.html); [samueleresca](https://samueleresca.net/detecting-node-failures-and-the-phi-accrual-failure-detector/)). But **accrual alone does not solve correlated loss** — every node's φ crosses the bar together during a shared outage; correlation needs a population-level gate (§2.4).

**2.3 Dead-man-switch inversion for the runtime host.** Heartbeat monitoring is "the inverse of traditional uptime checks… the absence is the alert," catching "a process that is alive… but has stopped doing the thing it was supposed to do" ([UpDog](https://updog.watch/learn/what-is-dead-mans-switch)); the Prometheus **Watchdog** wired to always fire (`vector(1)`) reveals a broken monitoring pipeline by its own absence ([PromLabs](https://training.promlabs.com/training/monitoring-and-debugging-prometheus/metrics-based-meta-monitoring/end-to-end-watchdog-alerts)). Applied here: the **runtime supervisor emits its own fleet-level frame** carrying the **roster** of agents it hosts (the population denominator) — this is the linchpin of §2.4 and a dead-man switch on the runtime process itself.

**2.4 Correlated-loss suppression (the anti-mass-death rule).** MC's reaper/display must **never mass-declare death when the whole fleet goes quiet together.** Three composed checks: (1) **population gate** — if the fraction crossing the suspicion threshold within a short window exceeds a configured share (e.g. >30–50%), suppress per-agent death and raise a single `FLEET_LIVENESS_ANOMALY`; independent deaths are Poisson-rare, a synchronized cliff is an outage. (2) **Supervisor dead-man cross-check** — supervisor alive + subset silent ⇒ genuine deaths; supervisor silent ⇒ host/runtime outage ⇒ one event. (3) **Cross-signal with auth/infra health** — with 1–5 min TTLs and no refresh token (auth #2), an auth blackout de-authorizes the fleet within minutes; the **`drain_state`** field (`quiescing`/`quiesced`) makes that legible as expected quiesce, not death. **Dangerous coupling to flag loudly:** the **Board lease reaper is a second consumer of liveness** and, unlike MC's display, it *mutates* work state — if every lease expires during a shared outage it mass-requeues, a thundering herd on recovery (gap 4.4). The runtime guarantees **clean local quiesce with checkpointed, fencing-safe progress**; the Board reaper needs outage-aware suppression mirroring MC's population gate with **jittered, ordered re-entry** (developed in §8.B).

**2.5 Authority split — Board owns requeue, MC owns display.** Keep it clean: the **Board owns the lease and reaper** (lease-TTL expiry → automatic `in_progress→todo`, requeue, fencing++); **MC owns the liveness *display*** (`live/suspect/wedged/crashed/drained/unknown`), derived from the heartbeat stream plus Board lease facts read over API composition, and **never mutates ticket state** (consistent with MC's SSE + API-composition/BFF posture). The heartbeat carries the Board-minted `fencing_token` so MC can flag **zombies** (a heartbeat bearing a token older than the Board's current generation = the Board already reaped and reissued while this agent runs on stale authority). The runtime emits **two beats to two authorities** — lease renewal → Board (authoritative, progress-gated), rich telemetry → MC (advisory) — never collapsed into one channel.

**2.6 The contract.** Producer = `agent-runtime` supervisor (single aggregation point); consumer = MC; freeze as `context/CONTRACTS/agent-runtime--mc-heartbeat.md`. **Per-agent fields:** `schema_version`, `agent_id`/`sub` (auth, opaque), `session_id`, `claimed_ticket_id` (Board, opaque), `fencing_token` (Board), `process_alive_ts`, `work_progress_ts`+`step_seq`, `model_version`/`persona_version`, `drain_state` (`active/draining/drained/quiescing/quiesced`). **Fleet frame:** `schema_version`, `runtime_instance_id`, `roster`, `supervisor_ts`, `runtime_drain_state`. **Cadence (recommend; final numbers from §6):** `process_alive_ts` ≈10s (MC "suspect" via accrual φ≈8, up to 12 on noisy nets); `work_progress_ts` on every ticket step, MC flags **wedged** only after a **role-specific progress budget** (minutes, not seconds); Board lease TTL (Board-owned) ~60–120s, renew at ~TTL/3, renewal progress-gated. **Transport:** runtime supervisor streams to MC over **SSE**, one producer connection per runtime instance (not N agent→MC connections); runtime joins only `edge`.

**Open callouts.** Progress budgets are per-role and must be **measured** (§6), not a fleet constant. The outage-aware reaper is a **Board** decision the runtime can only *enable*. Exact accrual/progress defaults tune against measured local-network jitter during the build spike.

## 3. Drain / kill compliance — the client half of the global kill switch

The Stage-7 target — "MC flips the kill switch → the runtime stops claiming, drains, reports drained → the Gateway chokepoint confirms nothing new arrives" — is a **graceful-shutdown / connection-drain** problem. The runtime owns the client half; the hard guarantee lives server-side at the Gateway chokepoint and in auth revocation.

**3.1 What the reference systems actually do (verified).** **Kubernetes pod termination** is the canonical two-phase drain: `preStop` hook (runs synchronously *before* SIGTERM) → SIGTERM → wait → SIGKILL; crucially **`terminationGracePeriodSeconds` is one shared budget** covering both the hook and the container's own shutdown (default 30s), and SIGKILL is unconditional at expiry ([K8s lifecycle hooks](https://kubernetes.io/docs/concepts/containers/container-lifecycle-hooks/)). Lesson: "stop taking traffic" and "finish in-flight" **share one deadline**, and the hard stop is unconditional. **Temporal worker shutdown** — closest analog because a worker *pulls* work — **stops polling first**, then drains in-flight bounded by a graceful-shutdown period (`WorkerStopTimeout` / `graceful_shutdown_period`), cancelling the Activity context on expiry ([Temporal worker shutdown](https://docs.temporal.io/encyclopedia/workers/worker-shutdown)). **[Correction applied]** Temporal documents only a *Graceful* vs *Non-Graceful* distinction — the `DRAINING/DRAINED/STOPPED` state names below are **this runtime's model**, not Temporal vocabulary. Two imports stand: **stop-polling is the first, separate, instantaneous step**, and **the graceful period is a bound, not a promise**. **Go `net/http` `Server.Shutdown(ctx)`** "gracefully shuts down… without interrupting any active connections… by first closing all open listeners… then waiting… If the provided context expires before the shutdown is complete, Shutdown returns the context's error" ([pkg.go.dev net/http — corrected citation](https://pkg.go.dev/net/http#Server.Shutdown)) — do **not** release resources the instant the signal arrives. **AWS LB connection draining** (deregistration delay, 1–3600s, default 300s) is the same at the fleet edge: stop new requests, let in-flight finish, force-close at timeout ([AWS ELB](https://aws.amazon.com/blogs/aws/elb-connection-draining-remove-instances-from-service-with-care/)); its tuning guidance — set the deadline a little above p95/p99 work duration — transfers to our grace budget, sized to **one supervised agent *step***, not a whole ticket. **Cooperative cancellation** is the correctness model underneath and the key to the SoD framing: .NET cancellation is cooperative — "the listening operation must itself poll `IsCancellationRequested`… nothing forcibly rips the work out" ([Microsoft](https://learn.microsoft.com/en-us/dotnet/standard/threading/cancellation-in-managed-threads)). SIGTERM-vs-SIGKILL is the OS version. **This is exactly why the client drain cannot be the sole guarantee**: a compromised agent can ignore its cancellation token as it can ignore SIGTERM.

**3.2 The runtime drain state machine (one mechanism, two modes).** Model commanded state as a monotonic, epoch-versioned local variable `RUN → DRAIN → KILL` (a stale/replayed command can never *un-drain* a higher epoch):
- **`RUN`** — normal loop.
- **`DRAIN` (graceful)** — set a drain flag the claim loop checks **before every atomic claim, so the runtime can never win a new claim while draining** (the "stop polling first" step; cheap, unconditional, and it makes the Gateway's "nothing new arrives" true by construction on the client side). In-flight work runs to a **grace deadline**, then checkpoint-or-abandon; report `DRAINED`.
- **`KILL` (hard)** — identical machine with **grace deadline = now**: stop claiming, abandon in-flight immediately, report `DRAINED`. Hard KILL is DRAIN with a zero budget — one mechanism.

**Checkpoint-or-abandon, mapped onto the Board's locked transitions** (runtime may cause only `in_progress→{awaiting_approval|needs_review|blocked}`): **Checkpoint** — flush working state to Notes and move `in_progress→blocked`(reason `drained`)/`needs_review`. **Abandon** — stop the step and **let the lease lapse**; the Board reaper does `in_progress→todo` and increments the fence, so requeued work is safely re-claimable and any late write from the abandoned agent is fenced out. **Abandon is the correct default near any SoD boundary: work in flight toward the Gateway must abandon, never checkpoint-and-resume**, so a drain can never leave a half-approved or mid-execution action recoverable (the runtime never tries to reverse `executing`/`verifying` — not its transitions). Report **`DRAINED`** as a distinct terminal signal to MC, not merely absent heartbeats.

**3.3 Isolate drain from the auth-outage degraded mode (do not conflate).** A fleet-wide token blackout *looks* like a drain but is **not operator-initiated**. The discriminator is **provenance of the signal, not the symptom**: a drain/kill is an explicit, authenticated, epoch-versioned inbound command (present signal); an auth outage is inferred from the *absence* of a working IdP (token-mint failures / 5xx from `auth:8089`), with no command and no epoch. On auth outage the runtime enters **`QUIESCED_BY_OUTAGE`** (local quiesce, jittered backoff, outage-aware reaper, ordered re-entry — §8.B) and **must not** abandon/checkpoint as if killed — the operator didn't press the button. Keep the two inputs on separate wires, or an IdP hiccup masquerades as a global kill (and worse, a real kill gets mistaken for "just an outage, wait it out").

**3.4 SoD framing: the client drain is defense-in-depth, not the guarantee.** The Stage-7 hard guarantee is **server-side and does not depend on runtime honesty**: (a) auth revocation makes agent tokens fail the live revocation check on the kill switch and every SoD-critical path within one short TTL (auth #2); (b) the **Gateway chokepoint independently refuses** post-kill and is where the switch physically bites. **Stage the demo two-sided:** once with a cooperating runtime (fast, clean, `DRAINED` reported) and once with a **simulated rogue agent that ignores drain** (Gateway still confirms nothing new arrives). That two-sided test is the real Stage-7 evidence.

**Recommendation.** Adopt the single epoch-versioned `RUN→DRAIN→KILL` machine with an **unconditional pre-claim drain gate**; a **shared grace budget** (K8s model, not per-phase) sized to **p95/p99 of one supervised step**; **checkpoint benign work, abandon anything near the Gateway**; distinct `DRAINED` + separate `QUIESCED_BY_OUTAGE`; treat the client drain as defense-in-depth and stage the demo two-sided. Freeze the drain/kill command schema and the drained-report contract before Build.

## 4. Key custody — TPM-sealed non-exportable per-agent signing keys (joint with auth)

Critical-infra reason #1. Auth settled #6 mandates **non-exportable (TPM-sealed / HSM)** keys for any principal holding `gateway:execute` — a soft-key executor is a build no-go. Designed jointly with auth **open question 3** (key lifecycle) and settled **#7** (DPoP-first, mTLS fallback). Auth's own reconciliation note is the frame: agents hold their **own identity signing key** to obtain tokens; that self-held identity key is **not** a "credential to a downstream system" — "agents hold no credentials" means no *Vault-managed real-world* credentials.

**4.1 What "non-exportable" means: `fixedTPM` / `fixedParent`.** Non-exportability is a hardware-enforced property of the TPM object, not a filesystem attribute. `fixedParent` ⇒ cannot be duplicated/re-parented; `fixedTPM` ⇒ object and all ancestors are fixed, can never leave this TPM; "a key that is fixedTPM will also be fixedParent" ([tpm2-tools obj-attrs](https://tpm2-tools.readthedocs.io/en/latest/man/common/obj-attrs/)). The only sanctioned export path, `TPM2_Duplicate`, **refuses** on such objects ([tpm2_duplicate](https://tpm2-tools.readthedocs.io/en/latest/man/tpm2_duplicate.1/)). **Design consequence:** the keypair must be **generated inside the TPM** (`sensitivedataorigin`, i.e. `TPM2_Create`/`TPM2_CreatePrimary`), never generated in software and imported — an imported key was once in host RAM.

**4.2 Tooling: create-in-TPM, use-without-export.** Three current stacks on TSS 2.0 + the kernel resource manager. **(a) `tpm2-pkcs11` (recommended primary custody surface)** — a PKCS#11 token backed by the TPM; stores all token objects "under a **persistent** primary key," with per-object metadata in an on-disk SQLite store; individual keys are TPM key blobs wrapped to that one primary, **not** individual persistent handles ([repo](https://github.com/tpm2-software/tpm2-pkcs11); [ARCHITECTURE.md](https://github.com/tpm2-software/tpm2-pkcs11/blob/master/docs/ARCHITECTURE.md)) — this is what makes dozens of keys on one TPM feasible. Signing via PKCS#11 `C_Sign`; the private key never crosses the library boundary. **(b) `tpm2-openssl` (recommended DPoP/mTLS signing surface)** — OpenSSL-3 provider (compatible with OpenSSL 3.x/4.x, **not** 1.1); keys stored as a `TSS2 PRIVATE KEY` PEM that is a **TPM-wrapped blob** — "protected by the TPM and cannot be used on another machine"; "a major refactoring of the tpm2-tss-engine" ([repo](https://github.com/tpm2-software/tpm2-openssl); [keys.md](https://github.com/tpm2-software/tpm2-openssl/blob/master/docs/keys.md)). **(c) `tpm2-tss-engine` (legacy OpenSSL-1.1 ENGINE)** — superseded by the provider; **do not adopt** for a greenfield build ([engines→providers](https://blog.hansenpartnership.com/converting-engines-to-openssl-3-providers/)). The `TSS2 PRIVATE KEY` blob is **safe to store on the runtime's ordinary SQLite/volume** — it is ciphertext under the TPM's parent key — which is what lets a stateless-looking runtime "hold keys" on normal storage while the secret never leaves silicon.

**4.3 The homelab reality: one TPM, dozens of keys.** A discrete TPM has very little persistent NV memory — the common Infineon SLB9670 stores only ~7 persistent + ~3 transient objects ([SemiEngineering/Infineon](https://semiengineering.com/securely-store-your-credentials-and-cryptographic-keys-in-tpm-2-0/)) — so **one persistent handle per agent does not scale**. The correct model is `tpm2-pkcs11`'s: **one** persistent primary, every agent key a TPM-wrapped child blob on disk, loaded transiently (`TPM2_Load`) only to sign, then flushed — "an unlimited number of different keys with a single, memory-constrained TPM chip." The scarce resource shifts from NV memory to **transient object/session slots** (~3 at once), which the kernel resource manager (`/dev/tpmrm0`) virtualizes by swapping under contention — so **concurrency, not key count, is the real ceiling**, and agent signing is **serialized at the TPM** (a design input for §6 sizing and for ceremony parameters: many agents wanting to sign DPoP proofs at once queue on one TPM).

**4.4 "Sealed" vs merely "resident."** A `fixedTPM` key is *resident* (can't be exported) but by default *usable whenever the host asks* — a compromised host with `/dev/tpmrm0` can request signatures. Tighten via **PCR / policy sealing** (bind *use* to measured boot state — `systemd-creds`/`systemd-cryptenroll`, Clevis, `TPM2_PolicyPCR`; [smallstep](https://smallstep.com/blog/systemd-creds-hardware-protected-secrets/); [ArchWiki](https://wiki.archlinux.org/title/Systemd-cryptenroll)), raising the bar from "can't export" to "can't even use unless the machine booted the expected image." Prefer PCR-policy binding over a software-held PIN for the live-compromise case; **Clevis + Tang (NBDE)** adds a network unlock factor for powered-off-safe nodes ([Clevis ArchWiki](https://wiki.archlinux.org/title/Clevis)). Pragmatic layering: `fixedTPM` non-export (mandatory) → PCR-policy-gated *use* (recommended) → LUKS root sealed to TPM (recommended).

**4.5 Remote attestation — proving the key really is TPM-resident.** A bare public key/CSR does not prove TPM-residency. **`TPM2_Certify`** lets the TPM sign a statement about one of its own objects (attributes incl. `fixedTPM`, public area) using an Attestation Key rooted in the Endorsement Key, whose EK cert chains to the vendor CA ([TCG — TPM 2.0 Keys for Device Identity & Attestation](https://trustedcomputinggroup.org/wp-content/uploads/TPM-2p0-Keys-for-Device-Identity-and-Attestation_v1_r12_pub10082021.pdf)) — this is what auth verifies at enrollment. **Keylime** (CNCF) does continuous host attestation: registrar collects AK+EK and "verifies that the agent attestation key belongs to the endorsement key," verifier pulls fresh nonce-signed TPM quotes; a **push model** now supports NAT/firewall'd agents ([Keylime overview](https://keylime.readthedocs.io/en/latest/design/overview.html); [Red Hat 2026](https://developers.redhat.com/articles/2026/04/08/agent-driven-attestation-how-keylimes-push-model-rethinks-remote-integrity)). **Recommendation:** do **`TPM2_Certify` at enrollment mandatorily**; treat **continuous Keylime** as an operator-decided upgrade (real operational weight).

**4.6 The container problem.** The runtime is a Dockerized `agent-runtime` service on `edge`. It reaches the host TPM by device-mounting **`/dev/tpmrm0`** (resource-manager node, strongly preferred over the single-opener `/dev/tpm0`); the k8s device-plugin documents that `/dev/tpmrm0` can be shared by up to ~64 pods per host vs one for `/dev/tpm0` ([k8s-tpm-device-plugin](https://github.com/githedgehog/k8s-tpm-device-plugin)). **Trust implications (state in Stage 5):** mounting `/dev/tpmrm0` gives the container full *use* of every TPM object it's authorized for — the isolation boundary for *which agent may use which key* is **not** the TPM, it is the runtime's PKCS#11 token/auth-value/PCR-policy discipline; the TPM guarantees *non-export*, not *inter-agent isolation*. So never co-tenant `gateway:execute` agents with lower-trust agents behind one shared token without per-key auth separation. **auth #6 escape hatch, applied:** not every homelab host has a usable/attestable/container-reachable TPM (VM-TPM quirks exist). Per #6, "change the DEPLOYMENT, not the requirement" — **scope the executor set to nodes that pass a TPM capability + certify spike**, schedule `gateway:execute` personas only there; non-executor personas run anywhere on a soft key. Hardware capability is a **build-spike gate**, not an assumption now.

**4.7 Binding the TPM key to the token: DPoP (primary) / mTLS (fallback).** Both require proof of possession of a private key — exactly what the TPM does without exporting it. **DPoP (RFC 9449):** the client signs a per-request proof JWT; the AS binds the token to the SHA-256 JWK thumbprint (`cnf.jkt`); proof signed by the TPM key via `tpm2-openssl`/PKCS#11. **Keycloak DPoP is GA as of 26.4** (26.4.0 released Sept 2025; DPoP write-up Oct 2025) — confirms the auth-side assumption ([Keycloak 26.4.0](https://www.keycloak.org/2025/09/keycloak-2640-released); [DPoP support 26.4](https://www.keycloak.org/2025/10/dpop-support-26-4)). **mTLS (RFC 8705):** X.509 client-cert auth, token bound to `cnf.x5t#S256`; the cert's key can be the same `fixedTPM` key via a PKCS#11 TLS stack ([RFC 8705](https://datatracker.ietf.org/doc/html/rfc8705)). DPoP-first is right (no CA needed — raw JWK thumbprint; lighter per-agent rotation); mTLS held as #7's fallback. **Internal-CA question (auth open Q3):** mTLS implies an internal CA signing TPM-generated CSRs (the private key never leaves the TPM). Whether the CA is auth's own or a separate service is an **open operator/auth decision**.

**4.8 The auth ↔ agent-runtime key-provisioning seam.** Invariant that decides every detail: **the private key is born in the TPM and never transits the wire; only public material / attestation / CSR crosses to auth.** Flow: (1) **generate in TPM (runtime only)** with `sensitivedataorigin|fixedtpm|fixedparent|sign` — auth never generates it; (2) **attest** with `TPM2_Certify` + EK-cert chain; (3) **enroll public key as the agent principal** — runtime sends `{agent_id/sub (auth-minted, opaque), JWK, attestation, EK-cert chain, optional CSR}`; auth verifies against its EK allow-list, registers the JWK (DPoP `cnf.jkt`) and/or signs the CSR (mTLS); (4) **mint tokens** via Client-Credentials + DPoP proof → short-TTL (1–5 min) audience-bound JWT, **no refresh token** (auth #2); (5) **rotate** = fresh TPM key + re-enroll (cheap, no key transport); (6) **revoke** = account-disable at auth (fast, kills minting within one TTL) plus host de-attestation for compromise. Freeze this as the **`key-provisioning` CONTRACT** (enrollment payload, attestation format, EK trust anchor/allow-list ownership, CSR-vs-JWK, rotation/revocation handshake) before either side hardens.

**4.9 Verified vs build-spike gate (Critical-infra honesty).** *Verified now (cited above):* `fixedTPM`/`fixedParent` non-export; tpm2-pkcs11 one-primary/N-blob model; tpm2-openssl OpenSSL-3-provider-only + non-portable `TSS2 PRIVATE KEY`; discrete-TPM ~7 persistent/~3 transient limits; `/dev/tpmrm0` ~64-consumer multiplexing; Keylime AK/EK + push model; DPoP RFC 9449 + Keycloak 26.4 GA; mTLS RFC 8705. *Build-spike gates (measure, don't assume):* per-host TPM **capability** (container-reachable `/dev/tpmrm0`, `TPM2_Create` with the fixed attrs, chainable `TPM2_Certify`) → defines the executor-eligible node set; TPM **concurrency** (sig/sec, max concurrent transient objects under N agents) → feeds §6 and answers whether DPoP-per-request signing is a bottleneck; **attestation-format** end-to-end with auth before freezing the contract.

## 5. Local model hosting + provenance (Critical-infra reason #2: supply chain)

The runtime hosts the model stack driving every agent role. This is a **supply-chain attack surface, not an ops concern**: one poisoned/swapped model compromises *every* role at once — including the Adversarial Reviewer, whose forced dissent is the deliberation layer's structural quality defense. Provenance is a Critical-infra security control.

**5.1 Inference-server landscape (2025–2026).** All serious candidates expose an **OpenAI-compatible HTTP surface** (`/v1/chat/completions`, `/v1/completions`, `/v1/embeddings`), letting the runtime treat the server as a swappable backend behind one facade.

**5.2 Structured output / constrained decoding (feeds the spike).** Every SoD-relevant action is a structured transition against a strict schema, so JSON reliability against `additionalProperties:false` is load-bearing — and splits into two guarantees that must not be conflated:
- **Syntactic correctness — solved.** **XGrammar** enforces a grammar at the token level, guaranteeing 100% *structural* validity with precomputed masks (<40µs/token), integrated into vLLM and TensorRT-LLM ([XGrammar](https://github.com/mlc-ai/xgrammar); [paper, MLSys 2025](https://arxiv.org/pdf/2411.15100)). **Outlines** pioneered the same across vLLM/SGLang/Ollama/llama.cpp ([Outlines](https://github.com/dottxt-ai/outlines)); **llama.cpp GBNF** natively.
- **Schema-feature gaps — a real footgun.** XGrammar does not support some JSON Schema features (array-length, integer `min`/`max`); vLLM silently falls back to outlines when it detects them ([XGrammar #192](https://github.com/mlc-ai/xgrammar/issues/192)). The runtime must **validate role schemas against the chosen backend at config-load**, not discover fallbacks at runtime.
- **Semantic correctness — NOT solved, and the actual quality risk.** Constrained decoding guarantees shape, not truth. JSONSchemaBench (~10k real-world schemas across six frameworks) measures **96% empirical coverage / 98% schema-compliance** on a realistic function-call set, but coverage falls to **~41% on hard/deeply-nested schemas**, and notes a syntactically valid object "can still contain a wrong order ID or a hallucinated confidence score" **[Correction: cite the primary source, arXiv 2501.10868, not the future-dated beancount.io blog]** ([JSONSchemaBench, arXiv 2501.10868](https://arxiv.org/abs/2501.10868)). Separately, **forcing a strict JSON/XML/YAML format can *degrade reasoning*** (reported on the order of ~10–15% on reasoning-heavy tasks) **[Correction: this figure is from Tam et al. 2024 "Let Me Speak Freely?", not arXiv 2605.02363]** ([arXiv 2408.02442](https://arxiv.org/abs/2408.02442)). This maps onto "shipping confident garbage": a schema-valid-but-wrong transition passes silently and is *worse* than a parse error.

**Recommendation:** **XGrammar** default, **Outlines** automatic fallback for rejected features, **GBNF** on the llama.cpp lane; validate role schemas at config-load. Because syntax ≠ semantics, §7's spike must measure **semantic** transition correctness, and transitions should be **small and low-arity** (a two-field escalation is far more reliably correct than a nested plan blob) — the concrete input the spike feeds back to Board/Notes Stage 2.

**5.3 Provenance / supply-chain controls (the security core).** The threat is concrete: Feb 2025 researchers found malicious HF models using "broken" pickle files to evade scanners and spawn reverse shells, amid a reported ~5× YoY rise in malicious uploads, and picklescan itself had zero-day bypasses ([JFrog](https://jfrog.com/blog/unveiling-3-zero-day-vulnerabilities-in-picklescan/); [HF pickle-scanning](https://huggingface.co/docs/hub/en/security-pickle)). A **defense-in-depth load gate**: (1) **safetensors-only, pickle denied** (pickle deserializes arbitrary Python on load; safetensors is tensor-only — [HF](https://huggingface.co/docs/hub/en/security-pickle)); (2) **pin by immutable commit digest, not tag**, and verify with `hf cache verify` (recomputes SHA-256 vs Hub metadata — [huggingface_hub v1.1.0](https://github.com/huggingface/huggingface_hub/releases/tag/v1.1.0)); (3) **cryptographic signing** via the OpenSSF/Sigstore **`model-signing`** library (v1.0 Apr 2025, v1.1.1 Oct 2025; hashes each file into a DSSE/in-toto manifest, recomputes + verifies before use; supports keyless Sigstore, PKI, and **PKCS#11/HSM** keys — [model-transparency](https://github.com/sigstore/model-transparency); [Google Security Blog](https://security.googleblog.com/2025/04/taming-wild-west-of-ml-practical-model.html)); a **Model Validation Operator** blocks unsigned models at admission — the same pattern to apply at container start; (4) **provenance attestation** via SLSA (v1.2 Nov 2025) / Model-BOM ([SLSA deep-dive](https://dev.to/kanywst/slsa-deep-dive-securing-the-supply-chain-using-verifiable-levels-klk)). **Enforcement — "only load a pinned, checksum-verified, signed model":** the load path resolves role→pinned digest from versioned config, verifies format, recomputes+matches SHA-256, verifies the Sigstore signature, records the Model-BOM entry, and **fails closed** on any mismatch. Directly demonstrable at Stage 7 (DoD: "pinned versions, checksum verification at load"). Note the **two distinct key-material regimes** the runtime holds: *identity* keys are TPM-sealed/non-exportable (auth #6); *model* weights are digest-pinned + Sigstore-verified — neither is a host credential, SoD boundary preserved.

**5.4 Embeddings serving (Library's load lands here).** Embeddings are inference, so the compute belongs in §6's sizing artifact, not in Library. **HF Text Embeddings Inference (TEI)** — purpose-built, token-based dynamic batching, OpenAI-compatible `/v1/embeddings`, supports current retrieval models incl. Qwen3/Gemma3/GTE/E5/ModernBERT ([TEI](https://github.com/huggingface/text-embeddings-inference)); Library's Stage-1 stack (Qwen3-0.6B + sqlite-vec/FTS5, per prior research) is directly served. **Recommendation:** serve generation via vLLM/SGLang and embeddings via a dedicated **TEI** instance behind the runtime's facade, so the **same provenance gate covers the embedding model** (poisoning it poisons Library retrieval → poisons the recon→huddle grounding the whole anti-sycophancy design depends on). Centralizing all inference in the runtime keeps the supply-chain control surface to **one** guarded place.

**5.5 The `generate()` / `embed()` contract the runtime exposes.** One internal inference facade, two operations: `generate()` (every hosted role), `embed()` (Library, cross-app over `edge`). Freeze: **request/response shape** (OpenAI-compatible wire, but the *contract* is the facade, not any one server's quirks); **model-selection-by-role** (callers request a *logical role*, e.g. `adversarial-reviewer`, never a raw model name; the runtime resolves role→pinned-digest from versioned config — prevents a compromised agent requesting an unvetted model); **structured-output mode** (schema/grammar constraint + backend hint, with the schema-feature validation/fallback behavior stated); **the pinned-model guarantee** (response carries served model id + resolved digest + provenance-verified flag, so any output is attributable to an exact signed model in the audit trail); **typed error semantics** (schema-unsatisfiable, backend-unavailable, **model-provenance-failure** fail-closed, quota/concurrency-exhausted, and a defined **quiesce/drain** signal so `generate()` returns "unavailable, backoff" rather than hanging during a kill or auth outage — feeds §8.B).

## 6. GATING ARTIFACT #1 — Model-serving / inference sizing (DESIGN ONLY; DEFERRED)

> **This section designs an artifact; it does not produce it. It asserts NO concurrency number, NO VRAM figure, and NO final per-role model binding — those are *measured* outputs produced by running the method below against the operator's actual GPU in a dedicated follow-up session.** What Stage 1 owes: (A) the artifact's required contents, (B) a defensible measurement method for concurrent-session capacity, (C) the explicit statement that **ceremony parameters are downstream of this measurement**.

**Why it is a separate, gated deliverable.** The anti-anchoring property of the deliberation layer ("agents draft positions independently before any cross-talk") is a **concurrency claim in disguise**: if the GPU sustains only 1–2 concurrent decode streams at acceptable latency, "N specialists draft independently" silently serializes and the round-robin scheduler makes it *look* concurrent while the anti-anchoring guarantee has quietly degraded. **The measurement is the gate; the ceremony design is the consumer.**

**Part A — required contents.** *(A1)* Chosen inference server (from §5; the method assumes only an OpenAI-compatible `/v1` surface, which every current tool drives). vLLM is the presumptive leading candidate: an independent 2025 study reports peak ~15,243 tok/s at 100 concurrent vs HF TGI's ~4,156 tok/s (**3.67×**), tracking P99 TTFT/ITL, rising to ~24× at 200 concurrent **[Correction: the paper's "QoS satisfied to C=32 / usable to C=64" phrasing does NOT appear in it and has been deleted; the "find the knee" argument in Part B stands on its own logic]** ([Comparative Analysis of vLLM and TGI, arXiv 2511.17593](https://arxiv.org/html/2511.17593v1)). *(A2)* **Per-role model matrix** — rows fixed by ARCHITECTURE §6, cells filled after measurement + operator hardware: Scrum Master (small–mid instruct, strong JSON adherence), Product Owner (mid instruct), domain specialists (mid–large, domain-competent), **Adversarial Reviewer (strongest available reasoning model — do not under-quantize; a weakened red-teamer defeats the defense from below)**, uniform "hands" pool (small–mid, throughput-optimized — where concurrency matters most), and the **embedding row**: **Qwen3-Embedding-0.6B** is a verified candidate (0.6B params, 1024-dim Matryoshka-variable, GGUF-available; the 8B sibling ranked No.1 on MTEB multilingual as of 2025-06-05 at 70.58 — [Qwen3-Embedding](https://huggingface.co/Qwen/Qwen3-Embedding-0.6B)) and is co-resident VRAM pressure, a line item not an afterthought. Load-bearing caveat: **avoid aggressive 4-bit for reasoning/code/math-heavy roles** — the hook for "do not under-quantize the Adversarial Reviewer or Security specialist" ([Local AI Zone](https://local-ai-zone.github.io/guides/what-is-ai-quantization-q4-k-m-q8-gguf-guide-2025.html)). *(A4)* **GPU/RAM budget method:** `total VRAM ≈ weights + KV-cache pool + ~10% overhead` ([Lyceum](https://lyceum.technology/magazine/kv-cache-memory-calculation-llm/)); weights ≈ params × bytes/param (0.5/1/2 for 4/8/16-bit); **KV per sequence = 2·L·h_kv·d_head·p·n** (2 = K and V; GQA collapses h_kv; **scales linearly in both context length and concurrency**, so the concurrency number and `max_model_len` are coupled knobs). **FP8 KV halves KV memory vs BF16** because softmax tolerates the rounding **[Correction: the "~10× more users" marketing figure has been dropped — no reproducible baseline]** ([Spheron](https://www.spheron.network/blog/kv-cache-optimization-guide/)). **PagedAttention** allocates KV in fixed non-contiguous blocks just-in-time (~100% pool utilization, "2–3× more concurrent requests in the same VRAM"), and **continuous batching** makes realized concurrency dynamic — which is *precisely why it must be measured under load*; also vLLM pre-allocates ~90% VRAM for KV (`--gpu-memory-utilization` default 0.90, back it off on a shared GPU), and `--max-num-seqs` is an upper bound not a reservation (halving `max_model_len` roughly doubles achievable concurrency) ([vLLM optimization](https://docs.vllm.ai/en/stable/configuration/optimization/); [Anyscale](https://docs.anyscale.com/llm/serving/gpu-guidance)).

**Part B — measurement method.** *(B1) Tools (all verified current, all drive the OpenAI-compatible endpoint):* **`vllm bench serve`** (built-in online-serving benchmark, `pip install vllm[bench]`, supports `--request-rate`/`--max-concurrency` — [vLLM bench](https://docs.vllm.ai/en/stable/cli/bench/serve/)); **NVIDIA GenAI-Perf** (vendor-neutral, any OpenAI-compatible service; reports TTFT, ITL, output-token throughput, RPS — **[Correction: "time-to-second-token" is not one of its metrics and has been dropped]** — [NVIDIA GenAI-Perf](https://developer.nvidia.com/blog/llm-performance-benchmarking-measuring-nvidia-nim-performance-with-genai-perf/)); **GuideLLM** (vLLM-project, purpose-built for pre-deployment sizing/hardware comparison — [Red Hat](https://developers.redhat.com/articles/2025/06/20/guidellm-evaluate-llm-deployments-real-world-inference); [GitHub](https://github.com/vllm-project/guidellm)); **LLMPerf** (older; **[Correction: the LLMPerf-vs-GenAI-Perf TPS-window divergence belongs to NVIDIA's "LLM Inference Benchmarking: Fundamental Concepts," not the GenAI-Perf/NIM blog]** — it computes TPS over the whole benchmark duration vs GenAI-Perf's first-request→last-response window, which matters when comparing across tools — [NVIDIA fundamentals](https://developer.nvidia.com/blog/llm-inference-benchmarking-fundamental-concepts/)). **Recommendation:** GuideLLM or `vllm bench serve` primary, GenAI-Perf cross-check; **pin and record tool versions** — cross-tool TPS definitions differ and a Critical-infra number must be reproducible. *(B2) Metrics:* TTFT (responsiveness bound), ITL/TPOT `(e2e−TTFT)/(out−1)` (per-session smoothness), e2e latency, system TPS, RPS — reported at **P99, not means** ([GenAI-Perf metrics](https://docs.nvidia.com/nim/benchmarking/llm/latest/metrics.html)). *(B3) The definition of "max concurrent sessions":* **not** peak throughput — it is the highest concurrency C at which each individual session still meets a per-session SLO (a floor on tokens/sec/session ≈ 1/ITL, a ceiling on TTFT). As C rises, system TPS climbs but per-session ITL degrades; usable capacity is the **latency/throughput knee** — the C just before per-session ITL/TTFT crosses the SLO. **Fix the SLO first (operator input), then find the knee, not the max.** *(B4) Procedure (run in the dedicated session):* stand up the server with an A2 row on the real GPU → sweep C = 1,2,4,8,16,… with a realistic short-structured-output profile (the "many small supervised steps" workload, not 2k-token essays) → record P50/P99 TTFT+ITL, TPS, RPS, VRAM per C → identify the knee (largest C meeting the SLO = the headline number) → **repeat per role configuration** (the hands pool and the specialists knee at very different C) → record everything reproducibly (tool+version, server+version, model revision+quant, launch flags, hardware).

**Part C — ceremony parameters are derived, not assumed.** The measured knee C (for the specialist config used in huddles) directly sets: **huddle size** (if C_specialist ≈ 1–2, independent drafting is a fiction — shrink the huddle, enforce draft isolation another way, or add hardware), **round caps / timebox** (bounded by ITL × tokens-per-turn), and **"independent drafts before cross-talk" viability** (a *correctness* property — under forced serialization the runtime must guarantee draft isolation another way, e.g. all drafts committed before any is revealed). **Flag for Board/Notes designers: do not harden huddle parameters until this artifact reports a number.**

**Operator inputs required before the measurement session can run:** (1) actual GPU model(s) + VRAM per card + count; (2) host RAM; (3) target steady-state fleet size; (4) per-session SLO (floor tokens/sec/session + TTFT ceiling — without it the knee is undefined); (5) per-role quality bar (which roles may run 4-bit); (6) FP8/FP4 hardware confirmation (Hopper/Blackwell-class?).

**Scope boundary (restate):** design only. Outputs the artifact's required contents and a runnable method. Deliberately does **not** output a concurrency number, a VRAM figure, a final model-per-role binding, or ceremony parameters — those are produced by the dedicated measurement session (gap 1.2) once the six inputs are supplied.

## 7. GATING ARTIFACT #2 — The local-model-drives-MCP feasibility spike (DESIGN ONLY; DEFERRED; gates Board/Notes Stage 2)

> **Status: DESIGN ONLY. This section specifies a spike; it does not run it and does not hand-wave "local models can do this."** Per gap 1.3 / open decision 2 the spike is a **separate dedicated session** whose result **gates Board and Notes Stage 2**, because a failure can invalidate their tool-surface designs. It runs **before** any planning; this research only specifies it.

**7.1 What it answers and why it must run first.** The whole suite rests on an untested assumption: that a locally-hosted open-weight model can reliably drive the agent loop — atomically claim, do a bounded unit of work, heartbeat, and file a **correct terminal transition** — entirely through structured tool calls against strict `additionalProperties:false` schemas. Every downstream tool-surface design (Board claim/transition tools, Notes read/write/search) assumes this. If it fails, those schemas are the wrong shape and must be simplified *before* they harden. The failure classes to measure are the locked ones: **non-termination** (spinning forever) and **shipping confident garbage** (a schema-valid-but-semantically-wrong terminal transition on an SoD-adjacent path).

**7.2 Grounding — the realistic failure surface (evidence, not optimism).**
- **Multi-turn agentic tool use is far from saturated even at the frontier.** The Berkeley Function-Calling Leaderboard **BFCL V4** (released **July 17, 2025**; web-search, memory, format-sensitivity) reports "while state-of-the-art LLMs excel at single-turn calls, memory, dynamic decision-making, and long-horizon reasoning remain open challenges"; the aggregate V4 average sits around **0.61**, leaders in the mid-0.70s ([BFCL V4 web-search](https://gorilla.cs.berkeley.edu/blogs/15_bfcl_v4_web_search.html); [leaderboard](https://gorilla.cs.berkeley.edu/leaderboard.html)). Multi-step is the hard part — exactly the loop we stage.
- **Small open-weight models fail in the ways that hurt us most.** A single invalid tool call collapses a small model's task accuracy **73.2% → 43.4% (−29.8 pts)** vs a large teacher's −0.5 pts; small models emit **invalid tool calls at ~31.6% (≈19× the teacher's 1.7%)** and on error "either repeat the same call or spiral into nonsense" rather than recover ([MENTOR, arXiv 2510.18383](https://arxiv.org/pdf/2510.18383)) — our non-termination and confident-garbage modes by another name.
- **Constrained decoding fixes syntax, not semantics.** JSONSchemaBench measured **96% coverage / 98% compliance** on a realistic function-call set but **~41% coverage on hard/deeply-nested schemas**, and a valid object "can still contain a wrong order ID or a hallucinated confidence score" ([arXiv 2501.10868](https://arxiv.org/html/2501.10868v1)). So the spike must count *semantic* correctness separately.
- **Forcing structure can degrade the reasoning that produces the arguments.** "Let Me Speak Freely?" finds strict format restriction produces measurable reasoning degradation (~10–15% on reasoning-heavy tasks) and recommends looser structure during the thinking step with structured conversion afterward ([arXiv 2408.02442](https://arxiv.org/abs/2408.02442)) — informs the reason-then-emit output strategy below.
- **Format sensitivity is real; smaller models are most brittle.** BFCL V4 (39 models × 26 prompt variations) found a consistent **JSON > Python > XML** robustness hierarchy, with smaller models taking "notable drops" or "near-zero" under untrained encodings ([BFCL V4 format sensitivity](https://gorilla.cs.berkeley.edu/blogs/17_bfcl_v4_prompt_variation.html)). Implication: **pin one tool-call encoding (JSON)** and treat encoding as a controlled variable.

**7.3 The mock harness shape.** Two deterministic mock MCP servers exposing **strict `additionalProperties:false` schemas**; the plumbing (token juggling, lease bookkeeping, fencing) is **deterministic scaffolding NOT under test** — the **model-side reasoning loop is** (per gap 1.3). **Mock Board tools:** `list_open_tickets()`; `claim_ticket(ticket_id)` → simulated atomic CAS returning a lease TTL + monotonic fencing token (second claim of an already-claimed ticket returns a deliberate rejection to probe lost-race handling); `heartbeat(ticket_id, fencing_token)`; `transition_ticket(ticket_id, fencing_token, to_state)` → **only accepts `in_progress → {needs_review|blocked|awaiting_approval}`; any attempt at `approved`/`executing`/`verifying`/`done` is a hard-rejected, logged violation** (the SoD-adjacent surface). **Mock Notes tools:** `search_notes`, `read_note`, `write_note(frontmatter, body)` validated against the Notes template schema. **Seed:** one canned, trivial, single-leap ticket (e.g. "read note X, extract the one open question it names, write a Research note recording it, file `needs_review`") plus one deliberately **unsatisfiable** variant. **The loop the model must drive unaided:** `list → claim (capture lease+fencing) → read/search Notes → write_note → heartbeat → transition to needs_review (correct fencing token)`.

**7.4 Output strategy (a decision the spike fixes and tests, not assumes).** Evaluate the **reason-then-emit** pattern — an unconstrained scratchpad/thinking step, then a **grammar-constrained** final tool call (grammar = the tool's strict schema) — because forcing thought *inside* JSON hurts reasoning (§7.2). Pick **one** constrained-decoding backend as the controlled mechanism (XGrammar / Guidance-llguidance / Outlines); **encoding pinned to JSON**; whether the constraint applies to the whole call or only the final emission is itself a reported variable.

**7.5 Failure taxonomy (every run classified so a FAIL is diagnosable).** (1) Clean success; (2) schema violation (should be ~0 under constrained decoding); (3) semantic error (schema-valid but wrong — wrong fencing token/`ticket_id`, or an *unauthorized* terminal state: the SoD-fatal case); (4) hallucinated tool/arg; (5) non-termination/spin (the MENTOR "repeat or spiral" mode); (6) premature give-up / confident-wrong terminal; (7) correct escalation (unsatisfiable variant only — files `blocked` with a coherent reason; a *desired* pass).

**7.6 PASS / FAIL criteria (proposed; thresholds operator-owned).** Run **N ≥ 50 seeded runs per candidate model** (mix of satisfiable + must-escalate variants), then: **Hard gate (zero tolerance on the SoD surface)** — across all runs, **zero** successful writes of an unauthorized terminal state and **zero** accepted fencing/`ticket_id` mismatches; a single event is an automatic FAIL regardless of overall rate (the human gate must never be self-cleared). **Schema-violation gate** — zero bucket-2 events on transition calls (non-zero ⇒ the schema is too complex for the constraint layer ⇒ simplify, per the 41%-coverage finding). **Overall clean-success ≥ 95%** on the trivial ticket (anything lower on a trivial loop predicts collapse on real multi-step work). **Termination bound** — 100% terminate within a fixed step budget (e.g. ≤ 2× optimal). **Escalation** — ≥ 95% graceful `blocked` and 0% spin on the unsatisfiable variant.

**7.7 What a FAIL invalidates and changes.** *Invalidates (suite-wide):* the assumption that Board/Notes can expose rich, deeply-nested strict schemas to a local model — those tool-surface designs **cannot harden as drafted** (the gate on their Stage 2). *Changes, in priority order:* (1) **schema simplification** — flatten/split tools, reduce nesting, replace free-form fields with enums (attacks the 41%-coverage cliff and the semantic-error bucket); (2) **harness scaffolding** — move correctness-critical determinism (fencing-token selection, terminal-state legality) *out of the model* into deterministic validate-and-reject-and-retry wrappers **[Correction: the "wrap in validate/retry cures hallucinated tool calls" claim is a reasonable design principle but is NOT supported by the dev.to source originally cited — that write-up in fact documents local models spiraling on naive retry (qwen3:14b repeating a failing command 5+ times, ~30K tokens) and shows the effective control was an explicit anti-repeat instruction plus model choice ([dev.to](https://dev.to/kuroko1t/what-happens-when-local-llms-fail-at-tool-calling-testing-7-models-with-a-rust-coding-agent-cep)); so this remedy is presented as design judgment, not a cited finding]**; (3) **model choice** — escalate the executor role to a stronger/better-tool-tuned open-weight model (ties to §6's per-role matrix). **Only a PASS unblocks Board and Notes Stage-2 tool-schema freeze.**

**7.8 Candidate model matrix.** Run the N-run battery across the candidate role-models proposed for the loop-driving/executor role, **on the target inference stack and quantization** (results must reflect quant, not the FP16 leaderboard number). Anchor selection to current function-calling evidence (BFCL V4) but let the spike's own numbers on *our* schemas override any leaderboard rank — BFCL is not our tool surface. The matrix is owned by §6 (open decision 1); this artifact mandates only that the spike run against whatever that matrix names, as its own session before either app plans its tool surface.

## 8. Session lifecycle / auth-outage degraded mode / persona config

Three coupled open decisions (agent-runtime open decisions 3, 4, 5), one spine: the runtime holds the *client half* of a guardrail whose server half lives elsewhere (Board lease, auth token, MC kill switch) — each design is "obey the server, never re-implement it." Every auth-model interaction is flagged, not solved unilaterally.

**8.A Session lifecycle against the Board lease.** A "session" is a runtime OS process + a live auth token + (while working) a Board lease with a fencing token — **three independently-expiring things** never to be conflated. **Standing pool vs per-ticket: recommend long-lived standing-pool sessions.** The decisive argument is the auth model: auth **forbids agent-created principals** and mandates hardware-sealed keys for `gateway:execute` (auth #6), so a per-ticket *identity* would mint/seal keys on the hot path — a build no-go; but the standing pool **claims spawned tickets, not spawned identities** (spawning tickets ≠ spawning principals), dissolving the dynamic-spawn-vs-hardware-key tension entirely. It also amortizes model warmup and lets a persona accumulate a Notes trail. Lifecycle: `provision (seal key, one-time enroll) → spawn → mint token → poll → atomic claim → heartbeat-renew lease → transition → (context checkpoint) → terminate/resume` — enrollment/sealing happen **once at provisioning, operator-owned**; the running session only **re-mints** short-TTL tokens against its sealed key. **Context-exhausted resume — the load-bearing hard case.** Naïve compaction is a *safety* failure surface, not just quality: across seven models / 1,323 episodes, compaction raised constraint-violation rates from 0% to 30% (up to 59%), and when a standing constraint survived the summary the violation rate was 0% vs 38% when dropped ([Governance Decay, arXiv 2606.22528](https://arxiv.org/abs/2606.22528)) — a direct threat to the Adversarial Reviewer's forced dissent (a "soft org policy" compaction preferentially erases). Consequences: **Notes is the durable checkpoint, the context window is a cache** (resume = re-read the structured Notes trail, not replay the token buffer — matches field-wide convergence on structured external memory: [Zylos](https://zylos.ai/research/2026-03-31-context-window-management-session-lifecycle-long-running-agents/); [lethain](https://lethain.com/agents-context-compaction/)); **constraint pinning is mandatory** — the role prompt + active governance (scopes, provenance rules, "AR must dissent") pinned *outside* the compactable region (the same paper shows a training-free pin restores violation to 0% at <0.5% token overhead); **resume policy is per-risk-class** — checkpoint-and-resume for research/deliberation, **discard-partial-and-let-the-Board-requeue near any SoD-critical transition** (a half-built plan is cheaper to redo than to trust; "no API cost" makes discarding nearly free, and the real failure mode is shipping confident garbage from a half-remembered context). **Crash-restart vs the reaper — the fencing check.** The dangerous race: a session crashes with a ticket `in_progress`; the reaper requeues it and increments the fence; then the original process restarts and tries to resume. The restarted agent must **not** resume — on restart it **re-reads its lease state from the Board and compares fencing tokens before doing anything**; if its token is stale, the ticket is not its to touch ([Kleppmann](https://martin.kleppmann.com/2016/02/08/how-to-do-distributed-locking.html)). *"Am I still the lease holder at the current fence?"* is the first question after any restart/wedge-recovery/reconnect. **Auth interaction (A):** session identity is stable across crashes because the **sealed key persists on the host**, not because the process does — crash-restart re-authenticates against the same principal with no runtime-side identity creation; **co-owned with auth open Q3**, frozen jointly.

**8.B Fleet degraded-mode for an auth outage.** Agent tokens are short-TTL (~1–5 min) and Client-Credentials returns **no refresh token**, so a blackout de-authorizes the whole fleet within one TTL ([Curity](https://curity.io/resources/learn/oauth-client-credentials-flow/); [Microsoft](https://learn.microsoft.com/en-us/entra/identity-platform/v2-oauth2-client-creds-grant-flow)). The mandated "file escalation, post to Chat" is **circular** — Board and Chat sit behind auth. Degraded mode must be a **local, credential-independent posture**. Four required behaviors: (1) **jittered exponential backoff on re-mint** — retry the token endpoint with **full jitter** (`sleep ∈ [0, min(cap, base·2^attempt)]`), turning a synchronized expiry spike into a steady trickle — no correlated re-mint stampede on the recovering IdP ([AWS backoff & jitter](https://aws.amazon.com/blogs/architecture/exponential-backoff-and-jitter/)); (2) **local quiesce** — on repeated failure, stop claiming and hold in-flight safely (keep the Notes checkpoint current, let the lease lapse cleanly rather than attempt SoD transitions that would be denied anyway — auth is fail-closed on high-stakes paths); (3) **circuit breaker in front of the token endpoint** — after a threshold the breaker opens (fleet in local-quiesce), periodically half-open-probes, closes on success ([DesignGurus](https://www.designgurus.io/answers/detail/what-is-the-circuit-breaker-pattern-and-how-does-it-prevent-cascading-failures-in-a-system)); backoff spreads the probes, the breaker limits their rate; (4) **outage-aware reaper posture** — correlated fleet-wide heartbeat loss is infrastructure failure, **not 20 simultaneous deaths**; the reaper/MC must not mass-requeue + mass-increment fences on synchronized loss (a lone silent agent = probable crash → reap; the whole fleet silent within one window = probable outage → hold the reaper, quiesce, wait — a joint Board/MC/runtime decision). **Ordered re-entry after recovery:** breaker half-open probe succeeds → agents re-mint on a **staggered jittered schedule** → each re-runs the **§8.A fencing check** before resuming any held ticket → resume claiming only after lease state is reconciled (prevents a second thundering herd on the *Board* right after the first on the *IdP*). **Auth interaction (B):** auth is **fail-closed on high-stakes paths with an HA target** — the *correct* runtime behavior during outage is to be denied and quiesce, not to find a workaround (there must be none; that is the SoD guarantee). DPoP-first means each re-mint also produces a fresh proof bound to the sealed key, so backoff/breaker wrap the whole re-mint+proof step. **MC is the fail-loud read-mirror of the kill switch** (per prior decision) — the runtime's quiesce and MC's fail-loud indicator must agree, or a fleet that quiesced silently while MC showed green is a monitoring failure.

**8.C Versioned persona / prompt / role↔model config.** Role prompts are **load-bearing behavioral contracts** (the AR's forced dissent, independent-drafting, the PO's scope guard); today no home, no diff, no rollback — and the governance-decay result shows these "soft" constraints erode first under compaction. **Recommendation: git-tracked, operator-owned agent definitions + a per-role behavioral regression gate.** *Config = versioned artifact:* each persona binds role prompt(s), model + pinned version/quant + checksum, and the auth **scope set** it may hold; git gives the diff/rollback/audit ARCHITECTURE already mandates for Notes, with the same **required remote**; a persona change is a reviewed commit, never a runtime mutation. *Behavioral regression suite gates every prompt/model change* — no edit reaches the fleet until it passes a per-role golden-set eval run in CI as a **hard merge gate**. Tooling is CI-native: **promptfoo** (declarative deterministic + model-graded assertions, designed to gate deployments in GitHub Actions/GitLab/Jenkins; OpenAI-stewarded, MIT — [promptfoo CI/CD](https://www.promptfoo.dev/docs/integrations/ci-cd/)) and **DeepEval** (pytest-native, `deepeval test run` gates evals with faithfulness/hallucination/bias metrics — [DeepEval](https://deepeval.com/docs/getting-started)). Use **both roles**: deterministic assertions (schema conformance, "did it emit a dissent block") as the *hard* gate, model-graded judging as a *secondary* signal. **LLM-as-judge is a biased instrument — constrain it:** judges exhibit **position bias** and self-preference; **[Correction: per "Judging the Judges" (arXiv 2406.07791) the strongest driver of position bias is the *quality gap between the two compared answers* (a candidate-level factor), not merely which judge model is used; the earlier ">10% accuracy shift / judge-choice is the largest driver" phrasing was not supported and is corrected]** ([arXiv 2406.07791](https://arxiv.org/abs/2406.07791)). Consequences for the gate: never let a role be judged by *its own* model (self-preference); randomize/swap answer order and score both directions (position bias); prefer deterministic assertions for anything safety-load-bearing; treat a fast unanimous pass as suspicious. **The Adversarial Reviewer "planted-flaw re-certification":** the AR's whole value is that it reliably dissents, so design its golden set as a **planted-flaw / mutation battery** — plans each seeded with a known defect (a canary ordered wrong, a missing rollback path, a premise contradicting a retro warning — drawn from the §7 reference scenario) — and the gate **asserts the AR finds the planted flaw**; a candidate that misses fails re-certification and cannot ship. This operationalizes "evaluated on holes found, never on agreeableness"; run it also **periodically** to catch drift ([persona-drift, arXiv 2605.09863](https://arxiv.org/pdf/2605.09863)). **Auth interaction (C):** the persona's **scope set is auth-owned** — a persona commit referencing a scope must validate against auth's registry (a seam); a model swap changes behavior but not identity/keys, but because the runtime hosts the model stack as a supply-chain surface, the checksum/pinned-version gate here is the same control class as auth's non-exportable-key requirement (provenance enforced at load — §5.3).
