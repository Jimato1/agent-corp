# Stage 1 Research — `board` Coordination Board (Standard risk class)

The Board is an **atomic-claim work queue first and a kanban board second** — a durable queue whose defining mechanic is that exactly one agent wins a piece of work (and locks the real host behind it) at the same instant, with an Agile-ceremony deliberation layer projected over that same state as the human face. Everything else (lifecycle columns, epic/sub-issue hierarchy, WIP limits, ceremony swimlanes) is a second projection of one table, never a bolt-on. This document is dated **2026-07-01**; fast-moving specifics (the MCP spec revision, SDK patch versions, library option names, Wazuh/Postgres/SQLite version floors) are deliberately **not trusted from memory** — they are flagged inline and collected in the consolidated **Verify-at-build list** so they are re-confirmed against pinned versions at build time rather than inherited as assumptions.

## Executive summary

- **Build a durable atomic-claim queue, not a Jira clone.** The claim semantics come from the durable-queue family (SKIP LOCKED / visibility-timeout leases); the epic/sub-issue/column/WIP vocabulary comes from the human-tracker family; neither can be added afterward. Adopt Linear's shallow shape (`parent_id` self-reference + a separate typed link relation) over Jira's rigid three-tier hierarchy ([Linear conceptual model](https://linear.app/docs/conceptual-model); [Atlassian issue hierarchy](https://community.atlassian.com/forums/Jira-questions/Issue-hierarchy-epics-features-stories-tasks-sub-tasks-how-to/qaq-p/2139946)).
- **The atomic claim is a status-guarded compare-and-swap.** On SQLite (the suite default): `UPDATE tickets SET status='in_progress', claimed_by=? WHERE id=? AND status='todo' RETURNING *` inside `BEGIN IMMEDIATE` under WAL — collision-proof because SQLite serializes writers ([SQLite WAL](https://sqlite.org/wal.html); [SQLite RETURNING](https://sqlite.org/lang_returning.html)). It ports verbatim to Postgres `FOR UPDATE SKIP LOCKED` if parallel writers are ever needed ([Postgres SELECT](https://www.postgresql.org/docs/current/sql-select.html)).
- **Load-bearing correction: `SKIP LOCKED` does not exist in SQLite.** Do not design the claim around it. SQLite's one-writer-at-a-time model *is* the safety property, not a defect — two competing `UPDATE`s are serialized by construction; losers get `SQLITE_BUSY` and retry ([SQLite WAL](https://sqlite.org/wal.html)).
- **Lock the host, not just the ticket.** Model each host as a first-class resource row claimed by one atomic conditional `UPDATE … WHERE claimed_by IS NULL OR lease_expires_at < now()` with a TTL lease — backend-agnostic across SQLite and Postgres, self-healing on crash. Reserve Postgres advisory locks as optional defense-in-depth only ([Postgres advisory locks](https://www.postgresql.org/docs/current/functions-admin.html); [SQLite file locking](https://sqlite.org/lockingv3.html)).
- **A claim is a lease, guarded by a fencing token — not just a timeout.** An expired lease is not proof of death; a slow-but-alive agent and a reaper-released copy can run at once ([SQS visibility timeout](https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/sqs-visibility-timeout.html)). Safety comes from a per-host monotonic `lock_generation` the **Gateway** enforces, rejecting any lower token ([Kleppmann, *How to do distributed locking*](https://martin.kleppmann.com/2016/02/08/how-to-do-distributed-locking.html)).
- **Enforce dependency readiness inside the claim query.** Store edges as a directed `ticket_dep` table and fold a `NOT EXISTS` anti-join into the atomic claim (partial-indexed on SQLite, SKIP LOCKED on Postgres); reject cycle-closing edges at insert with a `WITH RECURSIVE` (UNION) reachability probe. Jira/GitHub links are advisory — copy their model, not their non-enforcement ([Jira issue-linking model](https://developer.atlassian.com/cloud/jira/platform/issue-linking-model/); [SQLite WITH](https://www.sqlite.org/lang_with.html)).
- **Standing tickets are first-class template+trigger rows; the three kickoffs are one spawn function called three ways.** Human = authenticated create; scheduled = a dumb `node-cron` `noOverlap` job that only calls the Board's spawn function; event = a Wazuh `custom-` Integrator webhook verified by raw-body HMAC with a synthetic dedup key. Idempotency lives in a UNIQUE `spawn_key`, not in the cron layer ([node-cron](https://nodecron.com/scheduling-options.html); [Wazuh integration reference](https://documentation.wazuh.com/current/user-manual/reference/ossec-conf/integration.html)).
- **Model the ceremony as an orthogonal, event-sourced phase column — no embedded statechart engine.** An append-only `ceremony_events` log is the source of truth; `ceremony_phase` is a rebuildable projection with named guards. Make never-termination and sycophancy structurally impossible via hard round caps (2–3 rounds, 2–4 agents), a designated decider, and a required-dissent guard — never "debate until consensus," since identical models collapse onto confident-wrong answers ([Stately/XState guards](https://stately.ai/docs/guards); [FREE-MAD, arXiv:2509.11035](https://arxiv.org/pdf/2509.11035)).
- **Ship two siblings over one service layer.** An idempotent, role-scoped MCP tool set (`claim_next`/`claim`/`release`/`heartbeat`/`transition`/`ceremony_transition`/`query`…) over **Streamable HTTP** against the current stable spec `2025-11-25`, paired with a poll/SSE kanban — both calling the identical internal API and audit log, neither downstream of the other ([MCP spec 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25)).
- **Deferred decision #1 — tie-break authority (RESOLVED):** a DACI/RAPID-shaped split — Scrum Master = process Driver, Product Owner = single **Recommender-of-record** (not autonomous final say), Adversarial Reviewer = scoped veto — plus a **reversibility-gated human backstop**. Convergence machinery runs **server-side** (the Board holds the wall-clock timebox + round counter, not the SM agent); reversibility is **derived from a task-type table + CMDB tier, never reasoned**; the no-human auto-path is gated at the **per-server execution ticket** by CMDB tier, not at the huddle. The huddle is explicitly **not** a segregation-of-duties chokepoint ([Scrum Guide 2020](https://scrumguides.org/scrum-guide.html); [Bain RAPID](https://www.bain.com/insights/rapid-decision-making/); [Wang et al. 2025](https://arxiv.org/html/2509.23055v1)).
- **Deferred decision #2 — ceremony scaling (RESOLVED):** a Scrum-Master triage gate routes into three lanes (straight-to-execute / lightweight / full ceremony) via a **deterministic decision table over signals, not an LLM score**. Safety-critical signals come from non-agent authorities (host tier from CMDB, destructiveness from a human-owned task-type registry); the `awaiting_approval` gate fires on the OR of (registry=destructive) ∨ (CMDB=ask), decoupled from triage. Triage binds at the **epic level** with children inheriting the parent's lane as a floor; fail-safe is structural (missing/stale signal, >1 host, any critical tier, or timeout → Full ceremony) ([Atlassian change types](https://www.atlassian.com/itsm/change-management/types); [Amazon 2016 letter](https://www.aboutamazon.com/news/company-news/2016-letter-to-shareholders)).
- **`done` should confirm externally where a verifier exists.** For infra tickets, Wazuh's active→solved transition — not agent self-report — is the completion signal; the same Integrator channel that opens work later confirms it ([Wazuh vulnerability detection](https://documentation.wazuh.com/current/user-manual/capabilities/vulnerability-detection/how-it-works.html)).

## 1. Comparable systems — both faces (human trackers + durable work queues)

The Board is a **durable work queue first and a kanban board second**. Both faces must sit over one state, so this survey treats them as two projections of the same table: the queue face defines the *claim* mechanic (exactly one agent wins a host), and the tracker face defines the *lifecycle/hierarchy/visualization* the operator sees. The load-bearing thesis: the atomic-claim semantics come from the durable-queue family; the epic/sub-issue/column/WIP vocabulary comes from the human-tracker family; neither can be bolted on afterward.

### (a) Human trackers — what the UI/data-model face should borrow

**Findings.** Jira's *native* hierarchy is exactly three levels — Epic → standard issue (Story/Task/Bug) → Sub-task — and extending upward (above Epic) requires the paid Premium tier or marketplace apps; sub-tasks notably cannot be independently sprinted ([Atlassian Community: issue hierarchy](https://community.atlassian.com/forums/Jira-questions/Issue-hierarchy-epics-features-stories-tasks-sub-tasks-how-to/qaq-p/2139946)). Linear models a deliberately shallow tree — Roadmap → Projects → Cycles → Issues, with **sub-issues limited to one level** and separate *issue relations* (blocks/blocked-by) for cross-links; its default workflow states are Backlog → Todo → In Progress → Done → Canceled, and states are grouped/ordered per-team ([Linear conceptual model](https://linear.app/docs/conceptual-model), [Linear workflow states](https://linear.app/docs/configuring-workflows)). Kanban theory contributes the two visualization primitives the operator console needs: **WIP limits** — a hard cap on items allowed in a workflow stage, applied per column (and optionally per swimlane, cell, or user) to surface bottlenecks ([Atlassian WIP limits](https://www.atlassian.com/agile/kanban/wip-limits), [Businessmap WIP](https://knowledgebase.businessmap.io/hc/en-us/articles/360019128471-What-Is-a-WIP-Limit-and-How-to-Set-It-Up)) — and **swimlanes**, horizontal lanes that separate *classes of work* (priority, type, SLA) rather than pipeline stage ([Kanban Tool swimlanes](https://kanbantool.com/kanban-swimlanes)).

**Recommendation.** Adopt Linear's shallow shape, not Jira's ceremony: a single generic `ticket` row with a `parent_id` self-reference (Epic/standing → children) plus a separate typed `link` relation for blocks/depends-on — this directly serves "epics that spawn children on a trigger" without Jira's rigid three-tier types. Map our locked lifecycle (`todo → in_progress → (awaiting_approval →) needs_review → done`, plus `blocked`) onto kanban *columns*, and reuse **swimlanes for the ceremony phase** (triage/recon/planning/adversarial/backlog/execute/retro) and **WIP limits as first-class guardrails** — the per-agent and global WIP caps ARCHITECTURE.md already mandates are literally the kanban column-limit primitive, so build them into the queue, not just the UI.

> **Gotcha:** Jira/Linear WIP limits are advisory UI nudges. Ours must be *enforced at claim time* (a claim that would breach a per-host or global WIP cap must fail atomically), because the failure mode here is a runaway agent, not an overloaded human.

### (b) Durable queues — what the CLAIM mechanic should borrow

**Findings.** The dominant Postgres-native pattern is `SELECT ... FOR UPDATE SKIP LOCKED` (added in PostgreSQL 9.5): a worker locks the next available row and concurrent workers *skip* locked rows instead of convoying behind them, and because the lock is transaction-scoped, a worker crash rolls back and returns the job automatically ([Neon: queues with SKIP LOCKED](https://neon.com/guides/queue-system)). **River** (Go/Postgres) is built explicitly on this, enqueues jobs *transactionally* with app data so a job is visible only if its transaction commits, and advertises tens of thousands of jobs/sec ([River GitHub](https://github.com/riverqueue/river), [riverqueue.com/docs](https://riverqueue.com/docs)). **Graphile Worker** pairs SKIP LOCKED for fetch with `LISTEN/NOTIFY` for sub-millisecond wake-ups instead of polling ([Graphile Worker docs](https://worker.graphile.org/docs)). **pgmq** implements SQS-style semantics on Postgres — a **visibility timeout** makes a read message invisible to other consumers until it's deleted/archived or the VT expires and it reappears, again via FOR UPDATE + SKIP LOCKED ([pgmq docs](https://pgmq.github.io/pgmq/latest/), [pgmq GitHub](https://github.com/pgmq/pgmq)). **Oban** (Elixir) is the closest cross-DB analog and — critically for us — as of v2.23 officially supports **PostgreSQL 14+, SQLite3 3.37+, and MySQL 8.4+** behind one API, retaining completed rows for inspection with an explicit state machine (`scheduled → available → executing → completed/cancelled`) ([Oban GitHub](https://github.com/oban-bg/oban), [Oban hexdocs](https://hexdocs.pm/oban/Oban.html)). The Redis family teaches the *at-least-once* hazard: plain Sidekiq uses `BRPOP`, which *removes* the job, so a crash mid-job loses it; Sidekiq Pro's `super_fetch`/reliable fetch atomically `LMOVE`s the job to a private backup list so it survives, and Faktory has workers talk to a broker rather than raw Redis for the same reason ([Sidekiq Reliability wiki](https://github.com/sidekiq/sidekiq/wiki/Reliability)). **Temporal** is the outlier — event-sourced *durable execution*: workers poll task queues by RPC and the server replays an append-only event history to resume a crashed workflow exactly where it stopped ([Temporal task queues](https://docs.temporal.io/task-queue), [Temporal workflow execution](https://docs.temporal.io/workflow-execution)).

**Recommendation.** Model the claim as a **`SELECT ... FOR UPDATE SKIP LOCKED` + `UPDATE ... RETURNING` CTE on Postgres**, but treat SQLite (our default store) as the primary target: SQLite has **no `FOR UPDATE`/`SKIP LOCKED`**, so the equivalent is a single atomic `UPDATE tickets SET status='in_progress', claimed_by=? WHERE id=(SELECT id ... LIMIT 1) AND status='todo' RETURNING *` under WAL mode with `BEGIN IMMEDIATE`; `RETURNING` exists in SQLite since 3.35.0 ([SQLite RETURNING](https://sqlite.org/lang_returning.html)), and WAL lets readers not block the single writer ([SQLite WAL](https://sqlite.org/wal.html)). Borrow **two ideas beyond the raw claim**: pgmq's **visibility-timeout/lease** so a claimed-but-silent agent's host lock auto-expires and requeues (this is our answer to never-terminating agents), and Oban's **retain-completed-rows** so the Board doubles as the audit log PROCESS.md requires. Skip Temporal-grade event-sourcing — it's the right *idea* (crash-resumable, externally-confirmed done) but far too heavy for a single-operator homelab; a lease + explicit lifecycle column captures 90% of the value.

> **Correction (load-bearing):** The famous SKIP LOCKED pattern **does not port to SQLite** — do not design the claim around it. SQLite serializes writers (one writer at a time), so our concurrency story is "many agents contend, one wins the `UPDATE`, losers get `SQLITE_BUSY` and retry," not "N workers pull N different rows in parallel." For ~20 hosts and a handful of local agents this is entirely adequate; the atomic-claim *correctness* is identical, only the throughput ceiling differs.

> **Gotcha:** SQLite's `RETURNING` emits rows in **arbitrary order** ([SQLite RETURNING](https://sqlite.org/lang_returning.html)), so ordering (severity/priority *within* a ticket) must live in the inner `SELECT ... ORDER BY ... LIMIT 1`, never on the `RETURNING` of a multi-row write.

## 2. Atomic claim — exactly-one-worker dequeue

**Findings.** The anti-collision primitive is the whole ballgame: "if two agents ever act on one host, the failure started here." There are three families of correct patterns, and the right one depends on the backing store.

*Postgres (the graduation target).* Postgres solves the "convoy" problem directly with `SELECT ... FOR UPDATE SKIP LOCKED`. The official docs state that with `SKIP LOCKED`, "any selected rows that cannot be immediately locked are skipped. Skipping locked rows provides an inconsistent view of the data, so this is not suitable for general purpose work, but can be used to avoid lock contention with multiple consumers accessing a queue-like table" ([Postgres SELECT, Locking Clause](https://www.postgresql.org/docs/current/sql-select.html)). This is exactly the queue case. `SKIP LOCKED` was introduced in PostgreSQL 9.5 ([PG 9.5 release notes](https://www.postgresql.org/docs/9.5/release-9-5.html)); every supported server has it. The canonical claim is a single statement: `UPDATE tickets SET status='in_progress', claimed_by=$1 WHERE id = (SELECT id FROM tickets WHERE status='todo' AND host_lock_free ORDER BY priority, created_at FOR UPDATE SKIP LOCKED LIMIT 1) RETURNING *`. Because the row lock is tied to the transaction, a crashed worker's lock is released automatically on rollback — no orphaned claims. One documented gotcha: `NOWAIT`/`SKIP LOCKED` "apply only to the row-level lock(s) — the required `ROW SHARE` table-level lock is still taken in the ordinary way" ([Postgres SELECT](https://www.postgresql.org/docs/current/sql-select.html)), which is harmless here.

*SQLite (the default store) has no row locks and no `SKIP LOCKED`.* SQLite locks the **whole database** for writes, not rows, and permits exactly one writer at a time. The WAL docs are explicit: "readers do not block writers and a writer does not block readers... since there is only one WAL file, there can only be one writer at a time" ([SQLite WAL](https://sqlite.org/wal.html)). This is not a limitation for the Board — it is a feature. A single-writer database *cannot* double-claim, because the two competing UPDATEs are serialized by construction. The correct SQLite claim is a **conditional UPDATE guarded by status**, which is a compare-and-swap: `UPDATE tickets SET status='in_progress', claimed_by=? WHERE id=? AND status='todo' RETURNING *`. If the row was already claimed, `status` is no longer `'todo'`, zero rows change, and `RETURNING` yields nothing — the loser sees an empty result and moves on. `RETURNING` has been supported since SQLite 3.35.0 (2021-03-12) and is modeled on Postgres, but only on top-level INSERT/UPDATE/DELETE — not inside triggers, not on virtual tables, and its row order is unpredictable ([SQLite RETURNING](https://sqlite.org/lang_returning.html)). For a single-row claim, ordering is moot; if you need "claim next," do a `SELECT` to pick the id, then the guarded `UPDATE ... WHERE id=? AND status='todo'` and check rowcount/`RETURNING`.

> **Gotcha (load-bearing):** `PRAGMA busy_timeout` does **not** reliably prevent `SQLITE_BUSY` for a transaction that starts read-then-upgrades-to-write. A `BEGIN DEFERRED` (the default) that reads first and later writes can hit `SQLITE_BUSY` immediately with no retry, because SQLite will not invoke the busy handler when it would risk deadlock against another connection already holding the write lock ([SQLite forum: BEGIN IMMEDIATE + busy_timeout](https://sqlite.org/forum/info/daa5f8e2a9519c2ae16c44983e1545ba79c9a17c7ef98c971dc37e4efeeeff98); [Hubert, SQLITE_BUSY despite timeout](https://berthub.eu/articles/posts/a-brief-post-on-sqlite3-database-locked-despite-timeout/)). The fix is to open every claim transaction with **`BEGIN IMMEDIATE`**, which "causes the database connection to start a new write immediately, without waiting for a write statement" ([SQLite BEGIN](https://sqlite.org/lang_transaction.html)). It acquires the write lock up front, so contention is resolved by the busy handler (with backoff) rather than an instant error.

**Recommendation.** Ship the Board on **SQLite in WAL mode** with this exact claim recipe: (1) `PRAGMA journal_mode=WAL;` and `PRAGMA busy_timeout=5000;` on every connection; (2) route all writes through a **single writer connection** (or a tiny serialized write queue in-process) — SQLite is single-writer anyway, so embrace it rather than fight it; (3) begin the claim with `BEGIN IMMEDIATE`; (4) claim via the guarded compare-and-swap `UPDATE ... SET status='in_progress', claimed_by=?, claimed_at=? WHERE id=? AND status='todo' RETURNING id`; (5) treat "zero rows returned" as "another agent won — reselect." This is provably collision-free: database-level write serialization plus a status-guarded CAS means two agents physically cannot both flip the same `todo` row. The same guarded-CAS shape ports verbatim to Postgres, so the application code changes little on graduation.

**Recommendation (host lock).** The *resource* lock (one claim = one host) is a **separate CAS** in the same transaction: acquire a row in a `host_locks` table (`INSERT ... WHERE NOT EXISTS` / unique constraint on `host`) atomically with the ticket flip, so ticket-claim and host-lock commit or fail together. Never lock only the ticket row.

**Recommendation (when to graduate to Postgres).** Stay on SQLite until either (a) you need multiple concurrent *writer processes* pulling from one queue at high frequency (SQLite serializes them, so throughput caps at one write txn at a time), or (b) the Board's data is genuinely shared across apps. At that point switch to Postgres `FOR UPDATE SKIP LOCKED`, which lets N workers dequeue in parallel without a convoy. For a single-operator homelab coordinating ~20 hosts and a handful of local agents, SQLite's single-writer model is not the bottleneck and is the simpler, safer default.

## 3. Resource-level locking beyond the ticket (per-host mutex)

**Findings.** The load-bearing requirement is that two agents cannot act on one server *even via different tickets*, so the lock must live on the **resource (host)**, not the ticket row. This is a classic distinction between the queue's job-claim mechanism and a separate named mutex keyed by resource identity. The two canonical DB primitives are different:

- **`FOR UPDATE SKIP LOCKED`** is the *dequeue* primitive. Per the PostgreSQL manual, "any selected rows that cannot be immediately locked are skipped… can be used to avoid lock contention with multiple consumers accessing a queue-like table" ([PostgreSQL: SELECT](https://www.postgresql.org/docs/current/sql-select.html)). Because the row lock is tied to the transaction, a worker crash rolls back and returns the job automatically. But this locks the **ticket row**, not the host — perfect for "exactly one agent wins *this ticket*," useless for "no other ticket touches *this host*."

- **Advisory locks** are the *resource-mutex* primitive. PostgreSQL advisory locks are "application-defined resource locks… identified either by a single 64-bit key value or two 32-bit key values" and are explicitly meant to lock application-defined resources ([PostgreSQL: Advisory Lock Functions](https://www.postgresql.org/docs/current/functions-admin.html)). The critical gotcha is scope: `pg_advisory_xact_lock(key)` is auto-released at COMMIT or ROLLBACK, while session-level `pg_advisory_lock(key)` persists until `pg_advisory_unlock` or session end, and session locks *stack* (three acquires need three releases). `pg_try_advisory_lock` returns `false` immediately instead of blocking — the right shape for a poll-and-skip agent loop. Advisory locks are pure in-memory coordination: they enforce nothing about actual rows and vanish if the holding connection dies.

Prior art converges on the same shape. Temporal models a per-resource mutex as a **long-lived workflow per resource id** that serializes acquire/release signals — "lock/unlock a particular resource within a Namespace, [while] other Workflow Executions… wait until a locked resource is unlocked," explicitly recommended "for small, fixed pools of scarce resources where each permit is held for minutes, not milliseconds" ([Temporal mutex sample](https://github.com/temporalio/samples-go/blob/main/mutex/mutex_workflow.go)) — which is exactly ~20 homelab hosts patched over minutes. Airflow's **pools** cap parallelism over "an arbitrary set of tasks… often used [for] tasks that run on a GPU node" by handing out a fixed number of named slots ([Airflow: Pools](https://airflow.apache.org/docs/apache-airflow/stable/administration-and-deployment/pools.html)); a per-host pool of size 1 *is* a per-host mutex. Sidekiq Enterprise unique-jobs locks on a composite `(class, args, queue)` key with a **mandatory time limit** "so that if a process crashes, any locks it is holding won't last forever" ([Sidekiq Ent Unique Jobs](https://github.com/sidekiq/sidekiq/wiki/Ent-Unique-Jobs)) — the lease/TTL idea that any homegrown lock needs.

> **Correction (SQLite has no advisory locks).** The suite default is SQLite, which offers *no* equivalent of `pg_advisory_lock`. SQLite is single-writer: a write transaction takes a database-wide exclusive lock, so exactly one writer proceeds and the rest block ([SQLite File Locking](https://sqlite.org/lockingv3.html)). You therefore cannot port an advisory-lock design to SQLite — you must model the lock as **data**: an explicit `host` row with a `claimed_by`/`lease_expires_at` column, claimed via a single atomic conditional `UPDATE … WHERE claimed_by IS NULL` inside `BEGIN IMMEDIATE`. This is the only model that works identically on both backends, which is decisive for a SQLite-by-default suite.

**Recommendation.** Do **not** rely on advisory locks as the primary design, precisely because they don't exist in SQLite and evaporate on connection loss. Model the host lock as a **first-class `resource` (host) table** with columns `resource_id`, `claimed_by_ticket`, `claimed_by_agent`, `lease_expires_at`. Acquiring the lock is a single atomic statement — `UPDATE resource SET claimed_by_ticket=?, lease_expires_at=now()+ttl WHERE resource_id=? AND (claimed_by_ticket IS NULL OR lease_expires_at < now())` — whose affected-row count (1 = won, 0 = lost) is the mutex result. This is backend-agnostic: on SQLite wrap it in `BEGIN IMMEDIATE` so the read-and-claim is one write-locked step ([SQLite File Locking](https://sqlite.org/lockingv3.html)); on Postgres the row lock plus `WHERE` guard suffices, optionally hardened with `pg_advisory_xact_lock(hashtext(resource_id))` for defense in depth ([PostgreSQL: Advisory Lock Functions](https://www.postgresql.org/docs/current/functions-admin.html)).

**Lock/ticket lifecycle divergence.** Keep the two lifecycles *separate but linked*. A ticket transitions `todo→in_progress→…→done`; the host lock is acquired at claim and released on **any** terminal or abandonment transition (`done`, `blocked`, `needs_review` handoff, or lease expiry). Use a **TTL lease, not a bare boolean**, so a crashed agent's host frees itself — mirror Sidekiq's "mandatory time limit" ([Sidekiq Ent Unique Jobs](https://github.com/sidekiq/sidekiq/wiki/Ent-Unique-Jobs)). Renew the lease on heartbeat; a lapsed lease is reclaimable by the atomic UPDATE above. This dodges the advisory-lock trap where a lock silently persists (session-level) or silently drops (connection death) out of sync with the ticket.

**Contention ordering.** When several tickets target one host, they queue on the host lock; the loser gets 0 rows and re-polls. Ordering *within* that contention is by ticket priority/severity — never by splitting into per-CVE tickets that fight over the lock, consistent with "one claim = one host = clean lock." A per-host FIFO/priority index (`ORDER BY priority, created_at`) over waiting tickets gives deterministic, starvation-resistant ordering without a broker.

## 4. Leases, heartbeats, and auto-release of dead workers

**Findings.** A claim in this system is a **lease**, not a permanent assignment: an agent that dies mid-work must have its ticket *and its host lock* returned to the pool automatically, or the fleet silently loses a server. Every mature queue solves this the same way, and their designs converge on three moving parts — a lease timeout, a heartbeat that renews it, and a reaper that sweeps expired leases.

The canonical model is SQS's **visibility timeout**: on receive, a message becomes invisible to other consumers for a bounded window (default 30s, min 0, max 12h); if the worker does not delete it before the window expires, "the message becomes visible again in the queue and can be retrieved by the same or a different consumer" ([SQS visibility timeout](https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/sqs-visibility-timeout.html)). SQS is explicit that this is only mutual-exclusion *hinting*, not a guarantee: "because of the at-least-once delivery model, Amazon SQS doesn't guarantee that a message won't be delivered more than once within the visibility timeout period" — so a slow-but-alive worker and a reaper-released copy **can run simultaneously**. AWS's own remedy is to extend the lease with a heartbeat: "Implement a heartbeat mechanism to periodically extend the visibility timeout" via `ChangeMessageVisibility`, and set `VisibilityTimeout` to 0 to voluntarily release early ([ChangeMessageVisibility](https://docs.aws.amazon.com/AWSSimpleQueueService/latest/APIReference/API_ChangeMessageVisibility.html)).

Temporal formalizes the heartbeat side. An activity sends heartbeats to the service; "if the Temporal Service does not receive a Heartbeat within a Heartbeat Timeout time period, the Activity will be considered failed" and rescheduled. It recommends "a relatively short Heartbeat Timeout and a frequent Heartbeat" for long-runners, and — critically — **throttles** heartbeats to `heartbeatTimeout * 0.8` so the app can call heartbeat aggressively without flooding the backend ([Temporal detecting activity failures](https://docs.temporal.io/encyclopedia/detecting-activity-failures)). Heartbeats also carry a `details` payload so a rescheduled attempt resumes from progress rather than restarting.

The double-execution risk is the load-bearing danger, and Kleppmann's *How to do distributed locking* is the primary source: "if the GC pause lasts longer than the lease expiry period, and the client doesn't realise that it has expired, it may go ahead and make some unsafe change." His fix is a **fencing token** — "a number that increases every time a client acquires the lock" — that the *downstream resource actively checks*, rejecting any write whose token "has gone backwards" ([Kleppmann 2016](https://martin.kleppmann.com/2016/02/08/how-to-do-distributed-locking.html)). A lease alone never prevents two holders; only a monotonic generation counter enforced at the point of action does.

Library reapers show the reaper shape. River's **Rescuer** re-enqueues (or discards) jobs "active" past `RescueStuckJobsAfter`, "Defaults to one hour, or `JobTimeout` plus one hour," and runs *only on the leader*: "One client at a time runs maintenance services, determined by leader election" ([River maintenance](https://riverqueue.com/docs/maintenance-services)). Oban's **Lifeline** plugin does the same for jobs "left in an `executing` state indefinitely" after a node dies, moving orphans "back to available so they can run again," and pairs it with a `shutdown_grace_period` (default 15s) for clean drains ([Oban troubleshooting](https://oban.hexdocs.pm/troubleshooting.html)).

> **Correction:** an expired lease is *not* proof the worker is dead — it may be paused, swapped, or in a long model call. Never let auto-release be the *only* thing guarding the host. The reaper releases the *claim*; the fencing token stops the *stale actor*. You need both.

> **Gotcha:** in this suite the "resource" a fence must protect is the **Gateway** (the only thing that touches a host). The Board hands the claiming agent a per-host `lock_generation`; the Gateway records the highest generation it has executed for that host and refuses a lower one. That is what makes auto-release safe against a zombie agent — the Board reaping the lock is worthless if a paused agent can still drive an execution.

**Recommendation.** Model every claim as `claimed_by`, `lease_expires_at`, and a monotonically increasing per-host `lock_generation`, all set atomically in the same transaction that flips `todo→in_progress` and locks the host row. Size the loop from Temporal/SQS: **lease TTL ≈ 3–5× heartbeat interval**; for local agents whose steps run seconds-to-minutes, a 60–90s heartbeat with a 5-minute lease is a sane default. Agents heartbeat by bumping `lease_expires_at` (and may attach a progress note, à la Temporal `details`, so a re-claim resumes). A **single reaper sweep** (one row `UPDATE ... WHERE lease_expires_at < now()` returning ticket to the pool *and* releasing the host lock in the same statement) runs every ~30s; because agents run continuously, this is a queue's core reliability path, not an afterthought.

On storage: for Postgres use `SELECT ... FOR UPDATE SKIP LOCKED` for both dequeue and reaper so workers never convoy ([Postgres SELECT](https://www.postgresql.org/docs/current/sql-select.html)). For the SQLite default there is exactly **one writer at a time** in WAL — "there can only be one writer at a time" ([SQLite WAL](https://www.sqlite.org/wal.html)) — which actually makes the reaper *simpler*: a single `UPDATE` in one transaction with `PRAGMA busy_timeout` is already atomic; no `SKIP LOCKED` needed, and no leader-election machinery (River/Oban need it only for multi-node Postgres). Run the reaper as an in-process periodic task in the single Board container. Because never-terminating is *the* failure mode here, pair the lease with a hard **wall-clock max-attempts / max-lease-renewals cap**: a worker that keeps heartbeating but never completes must eventually be force-failed to `needs_review` (escalation), not renewed forever — a runaway loop can otherwise hold a host lease indefinitely by heartbeating.

## 5. Ticket dependencies / blocking (DAG ordering from a pool)

**Findings.** Ordered work can emerge from a claim pool with no central planner if each ticket carries its own prerequisite edges and the claim query simply refuses to hand out anything not yet *ready*. This is exactly how mature workflow engines model it, and their vocabulary is worth borrowing. Airflow expresses order as directed **upstream/downstream** edges; by default "Airflow will wait for all upstream (direct parents) tasks for a task to be successful before it runs that task," and this is tunable per-task via `trigger_rule` ([Airflow Tasks](https://airflow.apache.org/docs/apache-airflow/stable/core-concepts/tasks.html)). Crucially, Airflow does *not* poll a global planner — the scheduler marks a task `scheduled` only once "the Task's dependencies are met," i.e. readiness is derived from the states of direct parents ([Airflow Tasks](https://airflow.apache.org/docs/apache-airflow/stable/core-concepts/tasks.html)). Airflow also names the failure-propagation state we need: when a required upstream fails, the downstream enters `upstream_failed` — "an upstream task failed and the Trigger Rule says we needed it" ([Airflow Tasks](https://airflow.apache.org/docs/apache-airflow/stable/core-concepts/tasks.html)). Trigger rules such as `all_success` (default), `none_failed`, and `all_done` let a dependent choose whether a failed/skipped parent blocks it forever or merely gates it ([Astronomer trigger rules](https://www.astronomer.io/docs/learn/airflow-trigger-rules)).

Issue trackers model the same relation more loosely as a *link*, not an execution edge. Jira's "Blocks" link type has an outward description "blocks" and inward "is blocked by," stored as `outwardIssue`/`inwardIssue`; but Atlassian is explicit that "the semantics of each direction is only interpretable at the user interface level and not at the API level" — the link is a label, and **Jira does not prevent a blocked issue from starting** ([Jira issue linking model](https://developer.atlassian.com/cloud/jira/platform/issue-linking-model/)). GitHub shipped issue dependencies to GA on 2025-08-21 with "blocked by"/"blocking" relationships, up to 50 links per relationship type, full REST/GraphQL/webhook support — but it too is "primarily a visualization and organizational tool rather than an enforcement mechanism" ([GitHub changelog](https://github.blog/changelog/2025-08-21-dependencies-on-issues/)).

> **Correction / load-bearing gotcha:** Jira and GitHub links are *advisory*. Copying their data model is fine; copying their (non-)enforcement is not. The Board is a **queue first** — readiness must be *enforced inside the atomic-claim query itself*, or two agents will grab work whose prerequisites are unmet.

Deriving the ready set is cheap in SQL and needs no graph library. Store edges as one row per `(ticket_id, depends_on_id)` in a `ticket_dep` table. A ticket is ready when no incomplete prerequisite exists — a `NOT EXISTS` anti-join, which is the SQL spelling of "in-degree over unfinished parents is zero." This composes directly with the atomic claim: in Postgres, `SELECT ... FOR UPDATE SKIP LOCKED LIMIT 1` lets concurrent workers claim without blocking, since "any selected rows that cannot be immediately locked are skipped ... to avoid lock contention with multiple consumers accessing a queue-like table" ([Postgres SELECT](https://www.postgresql.org/docs/current/sql-select.html)). SQLite has no `SKIP LOCKED`, but its default single-writer serialization makes the claim `UPDATE ... WHERE id = (SELECT ... LIMIT 1) AND status='todo'` atomic on its own; a `WHERE NOT EXISTS(...)` readiness predicate rides along in the same statement. Readiness scans stay fast with a **partial index** on the hot subset, e.g. `CREATE INDEX ... WHERE status='todo'`, which stores entries only for open tickets and is supported since SQLite 3.8.0 ([SQLite partial index](https://www.sqlite.org/partialindex.html)).

Cycle detection matters because a dependency cycle makes a ready set unreachable (deadlock), and the SM/PO decomposition step could emit one by mistake. On write, validate that a new edge does not close a loop via a `WITH RECURSIVE` reachability walk. SQLite's docs are explicit: use `UNION` (not `UNION ALL`) so that "only add rows to the queue if no identical row has been previously added," which "prevent[s] the recursion from entering an infinite loop if the graph contains cycles," optionally bounded by `LIMIT` ([SQLite WITH](https://www.sqlite.org/lang_with.html)). Detection = "is `depends_on_id` already reachable from `ticket_id`?"; if yes, reject the edge.

**Recommendation.** Model dependencies as an explicit edge table `ticket_dep(ticket_id, depends_on_id, kind)` with `kind` defaulting to `finish_to_start`; keep edges *directed* and store a single canonical direction (blocker → blocked) to avoid Jira's inward/outward ambiguity. Define **ready** as: status = `todo`, host lock free, and `NOT EXISTS` any prerequisite whose status is not in the terminal-success set (`done`; plus externally-verified where applicable). Fold that predicate straight into the atomic-claim `UPDATE`/`SELECT FOR UPDATE SKIP LOCKED` so claimability and readiness are one indivisible decision — never a check-then-act race. Reject cycle-closing edges at insert time with a recursive-CTE reachability probe. For dependent fate when a prerequisite is cancelled/failed/reopened: default to **block, don't auto-fail** — a failed prerequisite leaves dependents un-ready (visibly `blocked` with reason), which the human/Scrum-Master resolves, rather than silently cascading `upstream_failed` across the tree; reopening a `done` prerequisite must *retract* readiness from its descendants (recompute, not cache). Expose readiness as a derived view/query, never a stored boolean — a cached `is_ready` flag will drift under concurrent state changes. This keeps ordering emergent (each ticket knows only its parents), enforcement real (in the claim query, unlike Jira/GitHub), and the whole thing library-free on stock SQLite.

## 6. Standing/epic tickets + the three kickoff types

**Findings.** The Board needs a *parent that manufactures children on a trigger*, plus three distinct ways new work enters the queue. The mainstream tracker model is instructive but also a cautionary tale. In Jira, a scheduled Automation rule (e.g. "run on the first day of the month") can create an Epic and then child issues, chaining `createdIssues` smart-values to set each child's parent ([Atlassian community — scheduled Epic→Task→Sub-task](https://community.atlassian.com/forums/Jira-questions/How-do-I-create-Automation-to-create-Epics-gt-Tasks-gt-Subtasks/qaq-p/2910952)). But practitioners repeatedly warn this is "complex and hard to maintain," you cannot parent a sub-task directly under an Epic (you must go Epic→Story→Sub-task), and native rules aren't real templates — hence the marketplace of template apps ([Atlassian community — recurring Epics without duplication](https://community.atlassian.com/forums/Automation-questions/How-to-automate-recurring-Epics-and-tasks-in-Jira-without/qaq-p/3121442)). The load-bearing lesson: model the *spawn* as first-class data (a stored template + a trigger record with a dedup key), not as an imperative rule graph.

For the **scheduled** kickoff, the cleanest in-DB option on Postgres is `pg_cron`: cron-syntax jobs stored in `cron.job`, run by a background worker, requiring `pg_cron` in `shared_preload_libraries` and Postgres 10+ ([citusdata/pg_cron](https://github.com/citusdata/pg_cron)). Critically for a queue, pg_cron "run[s] multiple jobs in parallel, but only one instance of each specific job at a time" — a later fire queues behind a still-running one rather than double-firing ([pg_cron README](https://github.com/citusdata/pg_cron)). Since the suite defaults to **SQLite**, pg_cron is unavailable; the equivalent is an in-process Node scheduler. `node-cron` exposes standard 5/6-field cron expressions and, per its docs, a `noOverlap: true` option that *skips* a run if the previous one is still executing (and a `distributed` mode for multi-instance) ([node-cron scheduling options](https://nodecron.com/scheduling-options.html)). Either way, overlap suppression at the scheduler is necessary-but-not-sufficient — see the correction below.

For the **event-driven** kickoff, Wazuh's real mechanism is the **Integrator** daemon (`wazuh-integratord`), configured via `<integration>` blocks in `ossec.conf`. A custom integration's `name` *must* begin with `custom-`, and blocks filter by `level`, `rule_id`, `group`, and set `alert_format` (use `json`), `hook_url`, and `api_key`, with `timeout`/`retries` defaults of 10s/3 ([Wazuh integration reference](https://documentation.wazuh.com/current/user-manual/reference/ossec-conf/integration.html)). The integrator invokes the script per matching alert with three positional args in a fixed order: `argv[1]` = path to the alert JSON file, `argv[2]` = `api_key`, `argv[3]` = `hook_url` ([Wazuh — integrate external software using Integrator](https://wazuh.com/blog/how-to-integrate-external-software-using-integrator/)). This is how a "new CVE" reaches the Board: a small `custom-` script POSTs the alert to the Board's inbound webhook. Directly relevant to the "done is confirmed externally" invariant: Wazuh's Vulnerability Detection module emits alerts when a vuln is *detected or resolved*, each alert carrying a **Status of "Active" or "Solved,"** and stores state in `wazuh-states-vulnerabilities-*` indices ([Wazuh — vulnerability detection how-it-works](https://documentation.wazuh.com/current/user-manual/capabilities/vulnerability-detection/how-it-works.html)) — so the same channel both *opens* work and later *confirms* it.

Inbound webhooks need dedup and authenticity. The canonical pattern: verify an HMAC signature over the **raw** body first, then dedup on a stable event id stored behind a UNIQUE constraint, short-circuiting before any state mutation ([Stripe webhook idempotency & dedup guide](https://www.hooklistener.com/learn/webhook-idempotency-and-deduplication)). Wazuh alerts have no Stripe-style `evt_` id, so a synthetic idempotency key must be derived (see recommendation).

> **Correction:** scheduler-level `noOverlap`/pg_cron single-instance guards prevent two *scheduler ticks* from overlapping, NOT duplicate *tickets*. If a weekly rescan already spawned an open child epic, the next tick must find-or-skip. Dedup belongs in the Board's data (a UNIQUE spawn key), not in the cron layer. Cron dedup and ticket dedup are different problems.

> **Gotcha:** Wazuh vuln alerts fire only on *state change* in later scans, not on initial inventory ([how-it-works](https://documentation.wazuh.com/current/user-manual/capabilities/vulnerability-detection/how-it-works.html)) — an empty webhook stream can mean "nothing changed," not "healthy." Baseline posture must be pulled via the Wazuh API/indexer, not inferred from webhook silence.

**Recommendation.** Model a **standing ticket** as a normal ticket row with `kind = 'standing'`, an embedded child-template (title pattern, type, default host-scope query), and a `trigger` spec (`{type: manual|schedule|event, cron?, event_filter?}`). Spawning writes a child plus a `spawn_key` (e.g. `parent_id:period` for schedules, `parent_id:cve:host` for events) under a UNIQUE index; re-spawns become no-ops. This makes "maintain & improve" a durable parent that idempotently emits per-cycle epics.

**Recommendation.** One concrete mechanism per kickoff, right-sized for a single-operator SQLite/Docker homelab: **(1) Human** — the ordinary authenticated create endpoint/MCP tool. **(2) Scheduled** — a tiny in-process `node-cron` job with `noOverlap: true` that only *calls the Board's own spawn function* (keep the cron dumb; keep dedup in the DB). Prefer this over pg_cron precisely because storage is SQLite; adopt pg_cron only if an app is forced onto Postgres. **(3) Event** — a Wazuh `custom-` Integrator script filtered to vulnerability rule groups that POSTs the alert JSON to a Board `/hooks/wazuh` endpoint; the endpoint verifies a shared-secret HMAC over the raw body, computes `spawn_key = sha256(agent.id + cve.id + status)`, and inserts idempotently. Heavyweight schedulers (Temporal Schedules, external cron sidecars) are over-scale here and are noted only as the escalation path if the homelab ever outgrows a single process.

## 7. Ceremony state machine + multi-agent deliberation / convergence / anti-sycophancy

**Findings — modeling the ceremony as a statechart.** The ceremony (`triage → recon → planning → adversarial review → backlog → execute → retro`) is a second, *orthogonal* state dimension riding alongside the ticket lifecycle (`todo → in_progress → …`), not a replacement for it. This is exactly the "orthogonal region" that statecharts (Harel; the model behind XState/SCXML) formalize: a ticket has both a `lifecycle` state and a `ceremony` state, transitioned independently. Transitions between ceremony phases should be **guarded** — a guard is a pure, synchronous predicate over `(context, event)` that is checked when an event fires; if it returns false the transition is not taken, and when several guarded transitions share an event they are tested in array order with the first true guard winning, plus an optional default fallback ([Stately/XState Guards](https://stately.ai/docs/guards)). For the Board the guards are things like "all recon sub-tickets are `done`" (recon→planning) or "adversarial reviewer has posted ≥1 dissent AND decider has voted" (review→backlog). XState also supplies composite guards (`and`/`or`/`not`) and `stateIn()` for cross-region conditions ([Stately/XState Guards](https://stately.ai/docs/guards)), which map directly onto "may only enter `execute` if lifecycle is past `awaiting_approval`."

> **Gotcha — do not embed a JS statechart engine in the Board.** XState is a *modeling reference*, not a runtime dependency you need. For a SQLite-backed queue the durable, auditable equivalent is an **event-sourced transition log**: an append-only `ceremony_events` table (ticket_id, from_phase, to_phase, actor/role, guard-that-passed, timestamp, note-link) is the source of truth, and the current `ceremony_phase` column is a rebuildable projection. This matches the monorepo invariant that databases are rebuildable indexes and gives a free git-diffable audit trail. LangGraph's persistence model validates the shape: it checkpoints graph state per-thread so runs can pause for **human-approval gates** and resume, exactly the `awaiting_approval`/`needs_review` pauses the Board needs ([LangGraph persistence / checkpointers](https://langchain-ai.github.io/langgraph/concepts/persistence/), [langgraph-supervisor](https://github.com/langchain-ai/langgraph-supervisor-py)).

**Findings — deliberation prior art.** Multi-agent debate is the grounding for the huddle: Du et al. show that having multiple LLM instances *propose then debate their reasoning over several rounds to a common answer* measurably improves factuality and reduces hallucination ("society of minds") ([Du et al. 2023, arXiv:2305.14325](https://arxiv.org/abs/2305.14325)). Role structure is the other half: MetaGPT encodes human **SOPs as role-scoped prompt sequences** (Product Manager → Architect → Engineer → QA/Reviewer) with explicit templates and intermediate verification checkpoints, and reports this structure — not free-form chat — is what reduces logic errors ([MetaGPT, arXiv:2308.00352](https://arxiv.org/abs/2308.00352)). Frameworks converge on a *manager/selector* pattern: AutoGen/AG2 GroupChat runs all agents over one shared thread with a `GroupChatManager` that selects the next speaker and can nest sub-chats ([AutoGen Group Chat pattern](https://microsoft.github.io/autogen/stable//user-guide/core-user-guide/design-patterns/group-chat.html)). The Board's Scrum Master role *is* that manager, but it owns process (turn order, timebox, calling the vote), never content — which is precisely the "coordination lives in the Board, not agent chat" invariant.

**Findings — convergence / termination (the load-bearing risk).** Because local agents cost nothing per token, the real failure is *never terminating* or *converging confidently on garbage*. The debate literature gives hard numbers: accuracy typically **plateaus at 2–3 rounds and 2–4 agents**, after which added rounds hurt both accuracy and compute ([literature review, arXiv:2506.00066](https://arxiv.org/pdf/2506.00066)). Critically, **consensus-based stopping is dangerous** — identical models socially reinforce a *wrong* answer ("wrong-consensus convergence"), with ~1 in 4 initially-disputed questions collapsing to unanimous-wrong by round 3, and >2 rounds amplifying spurious consensus ([FREE-MAD, arXiv:2509.11035](https://arxiv.org/pdf/2509.11035)). "Voting or Consensus?" finds decision protocol matters and studies consensus-driven conformity directly ([Kaesberg et al., arXiv:2502.19130](https://arxiv.org/pdf/2502.19130)). Takeaway: **cap rounds, prefer a designated decider or vote over "debate until agreement," and treat fast consensus as a red flag, not success.**

**Findings — anti-sycophancy.** Anthropic shows sycophancy is a *general* RLHF behavior: models match user beliefs, wrongly concede when challenged, and both human raters and preference models prefer convincingly-written sycophantic answers over correct ones a non-trivial fraction of the time ([Anthropic, *Towards Understanding Sycophancy*](https://www.anthropic.com/news/towards-understanding-sycophancy-in-language-models); [arXiv:2310.13548](https://arxiv.org/abs/2310.13548)). Two structural counters: (1) agents draft positions **independently before any cross-talk** to kill anchoring, and (2) a **forced dissenter / red-team** role that is structurally required to attack premises and is scored on holes found, never on agreeableness.

**Recommendation.** Model ceremony phase as an **orthogonal, event-sourced column on the ticket**: append-only `ceremony_events` (source of truth) + a projected `ceremony_phase` enum, with each transition gated by an explicit named guard recorded in the event row. Do **not** ship a statechart runtime; encode guards as plain predicate functions the Board evaluates and logs. Persist the huddle as a structured **note attached to the planning ticket** (per architecture), and have ceremony transitions reference note revisions so the debate is auditable and rebuildable.

For convergence, hard-code a **timebox + round cap (default 3 rounds, 2–4 role agents)**, a **designated decider** (Product Owner tie-breaks; deadlock escalates to a human `needs_review`), and a **non-termination guard** that fires escalation if rounds exceed cap or successive drafts stop changing without a decision. Do **not** auto-advance on consensus alone — treat unanimous fast agreement as a trigger for a mandatory red-team pass. For anti-sycophancy, enforce **independent first drafts** (each role writes its position to the note before it can read others'), make the **Adversarial Reviewer a required transition guard** (review→backlog cannot fire with zero recorded dissents), and score the reviewer on premises attacked. Right-sized: this is all queue-and-log mechanics in SQLite plus prompt scaffolding — no new container, no orchestration engine.

> **Critical composition note (carried from the §9 adversarial review):** the timebox / round-cap / non-termination guard must be a **server-side Board watchdog**, not a job the Scrum-Master agent runs. An agent-run timer can itself hang or mis-count — the system's core failure mode — so only the Board's deterministic service may enforce the clock and auto-escalate a stuck huddle to `needs_review`. Agents *request* the vote; the Board *enforces* it. See §9.

## 8. The two surfaces — MCP agent tools + human kanban, over one state

**Findings.** MCP is a JSON-RPC 2.0 protocol between *hosts* (LLM apps), *clients* (connectors), and *servers* (which expose Tools/Resources/Prompts); tools are "model-controlled" and each is identified by `name` with an `inputSchema` and optional `outputSchema` ([MCP spec index, 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25); [Tools](https://modelcontextprotocol.io/specification/2025-11-25/server/tools)). The **current stable spec revision is `2025-11-25`**, and its authoritative schema lives at `schema/2025-11-25/schema.ts` ([spec index](https://modelcontextprotocol.io/specification/2025-11-25)). A larger `2026-07-28` revision (stateless core, a formal **Tasks** extension for long-running work, MCP Apps for server-rendered UI) is in release-candidate; the SDK `main` branches implement it as v2 **beta**, but v1.x remains the supported production line ([TS SDK repo](https://github.com/modelcontextprotocol/typescript-sdk); [Python SDK repo](https://github.com/modelcontextprotocol/python-sdk)).

The spec defines exactly **two standard transports: `stdio` and Streamable HTTP**; clients SHOULD support stdio whenever possible ([Transports](https://modelcontextprotocol.io/specification/2025-11-25/basic/transports)). **Streamable HTTP replaces the old HTTP+SSE transport from `2024-11-05`** — one MCP endpoint handling POST and GET, optionally upgrading a response to an SSE stream; the standalone SSE transport is deprecated/legacy ([Transports](https://modelcontextprotocol.io/specification/2025-11-25/basic/transports)). Sessions are carried in an `MCP-Session-Id` header; HTTP servers MUST validate the `Origin` header (403 on mismatch) and SHOULD bind to `127.0.0.1` locally to prevent DNS-rebinding, and clients MUST send `MCP-Protocol-Version: 2025-11-25` on every request ([Transports](https://modelcontextprotocol.io/specification/2025-11-25/basic/transports)).

Tool design primitives the Board should use directly: tools return `content` (unstructured) and/or a `structuredContent` object validated against `outputSchema`; **two error channels exist** — JSON-RPC *protocol errors* (unknown tool, malformed request; models rarely recover) versus *tool-execution errors* returned as a normal result with `isError: true` and actionable text the model can self-correct on ([Tools](https://modelcontextprotocol.io/specification/2025-11-25/server/tools)). Tools carry an optional `annotations` object and an `execution.taskSupport` flag (`forbidden`/`optional`/`required`) for task-augmented calls; clients MUST treat annotations as **untrusted** unless from a trusted server ([Tools](https://modelcontextprotocol.io/specification/2025-11-25/server/tools)). Latest published SDKs: TS `@modelcontextprotocol/sdk` v1.29.x and Python SDK v1.28.x on the v1 line (exact patch to re-pin at build).

> **Correction:** Do not build on the deprecated HTTP+SSE transport or bare SSE endpoints — expose the Board's MCP server over **Streamable HTTP** (works cleanly behind Caddy/Traefik) and stdio for local subprocess agents. Anything you "remember" about an SSE transport predates `2025-03-26`.

> **Gotcha:** A claim-lost / already-claimed conflict is a *business* outcome, not a malformed request — return it as `isError: true` structured content (e.g. `{code:"CLAIM_CONFLICT", current_owner, ticket_version}`), **not** a JSON-RPC error, so the agent can re-poll the queue instead of treating it as a broken tool.

**Recommendation — MCP tool list** (verbs = queue first, kanban second): `board.claim_next` (atomic dequeue: filter by role/host-tier, returns one ticket + a host lock, or empty); `board.claim` (claim a specific ticket by id, idempotent on owner); `board.release` (hand a ticket back, freeing the host lock); `board.heartbeat` (renew the claim/host-lock lease; missed beats reap the claim); `board.transition` (lifecycle move with `expected_version` for optimistic concurrency); `board.update` (edit fields/priority); `board.create` (file ticket; the three kickoff types are the *source*, one tool); `board.get` / `board.query` (structured filters, paginated, `outputSchema`'d — never a context dump); `board.add_dependency` / `board.link_note`; and `board.ceremony_transition` (triage→recon→…→retro, enforcing timebox/round caps). Make every mutating tool **idempotent via a caller-supplied `op_id` or `expected_version`** so an agent that retries after a dropped SSE stream cannot double-act. Scope tools by role (Scrum Master vs hands) at the authz layer — agents are scoped users.

**Recommendation — human kanban.** Columns mirror the lifecycle (`todo → in_progress → awaiting_approval → needs_review → done`, plus a `blocked` swimlane); a parallel **ceremony ribbon** shows each planning ticket's ceremony state; cards show owner/host-lock, heartbeat freshness, priority, and epic parentage; column headers render **WIP limits** (per-agent and global) with over-limit highlighting. For a single-operator homelab, drive real-time updates with **one SSE stream from the same API** (poll fallback every few seconds) — websockets are overkill; the kanban needs server→client only.

**Recommendation — two views, one state.** MCP tools and the kanban are **siblings calling the identical service layer**; neither is downstream. Enforce it structurally: all reads/writes go through one internal API module over the single SQLite/Postgres store, the UI holds no business logic the MCP surface lacks, and both emit onto one audit/event log that also feeds the SSE stream — so a claim made by an agent lights up the operator's board within one poll interval and vice-versa.

## 9. DEFERRED DECISION #1 — Tie-break authority when the planning huddle deadlocks

> **Adversarial review (holes found; recommendation survives, amended).** The precedent scaffolding (Scrum Guide / RAPID / DACI) and the sycophancy citations were spot-checked and hold verbatim (Wang et al.: r=0.902 correlation between sycophancy and abandoning a correct answer; sycophancy "intensifies in later rounds"; a centralized judge cuts Disagreement-Collapse from ~81% to ~41%). The *composition* with this system's invariants is where the draft broke, on six points: (1) convergence machinery must not live in the SM agent — an agent timer can itself hang, so it moves **server-side** to a Board watchdog; (2) reversibility must be **derived, not reasoned** by the sycophancy-prone agents that would otherwise classify their own escape hatch; (3) reversibility is per-host-and-now (a patch with a rollback can still brick the NAS), so the auto-path is gated at the **per-server execution ticket**, not the huddle; (4) huddle-escalation and `awaiting_approval` are two **independent sequential** gates, not the same one; (5) forced dissent must be **mechanical** (AR files ≥1 recon-cited premise attack or the huddle is invalid; a veto clears only by AR or human); (6) over-escalation is contained not by widening the auto-path but by the segregation invariant — the huddle is **never the last signature** on a destructive action. The strengthened recommendation below folds in all six.

**Findings.** The candidate models map cleanly onto established decision-rights frameworks, so this is not greenfield. The **Scrum Guide (2020)** is unambiguous that the Product Owner is a *single accountable* decision-maker for content: "The Product Owner is one person, not a committee," those wanting a change "can do so by trying to convince the Product Owner," and "for Product Owners to succeed, the entire organization must respect their decisions" ([Scrum Guide 2020](https://scrumguides.org/scrum-guide.html)). The same guide gives the Scrum Master a *process*, not content, mandate: they are "accountable for the Scrum Team's effectiveness," ensure "all Scrum events take place and are positive, productive, and kept within the timebox," and coach "self-management" ([Scrum Guide 2020](https://scrumguides.org/scrum-guide.html)). This is exactly the split ARCHITECTURE.md proposes (SM owns process/timebox/the vote; PO owns goal + success metric/scope), and it is the canonical, citeable precedent for it.

Two decision-rights frameworks formalize "one namable decider." Atlassian's **DACI** names exactly one **Approver** — "the person with the final say" — while Contributors "have a voice, but not a vote," and the Driver merely facilitates ([Atlassian DACI](https://www.atlassian.com/team-playbook/plays/daci)). Bain's **RAPID** (from HBR "Who Has the D?") separates *Recommend* from *Decide*: exactly one **D** "makes the call," a small set of **A** roles hold "limited veto rights on clearly defined grounds (e.g., legal compliance … security)," and **I** roles inform but have "no veto rights" ([Bain RAPID](https://www.bain.com/insights/rapid-decision-making/)). Both frameworks exist specifically to *prevent* consensus-voting deadlock by pre-assigning a single accountable decider.

The escalation trigger has strong precedent too. Bezos's 2016 letter gives two load-bearing ideas: **"disagree and commit"** — "If you have conviction on a particular direction even though there's no consensus … 'I know we disagree on this but will you gamble with me on it?'" — and the **one-way vs. two-way door** distinction: reversible ("two-way door") decisions "can use a light-weight process," while consequential, irreversible ("one-way door") ones warrant heavy deliberation ([Bezos 2016 letter](https://www.aboutamazon.com/news/company-news/2016-letter-to-shareholders)). That directly ties tie-break weight to *reversibility*, which the Board already tracks via the `awaiting_approval` gate for destructive/irreversible task types.

> **Gotcha (the invariant killer):** an AI Product-Owner "decider" is exactly the failure mode the anti-sycophancy invariant warns about. A controlled study of multi-agent debate found a strong correlation (r=0.902) between an agent's sycophancy and its tendency to *abandon a correct answer* to align with the group, and — critically — "sycophantic behavior … actually intensifies in later rounds," so more debate makes false consensus *worse*, not better ([Wang et al., *Peacemaker or Troublemaker*, 2025](https://arxiv.org/html/2509.23055v1)). The same study found a centralized **judge** cut "disagreement collapse" from ~81% to ~41%, and that *heterogeneous* personas (a forced "troublemaker") beat uniformly agreeable ones. An always-agreeing AI decider therefore does not just fail to add value — it manufactures confident consensus, the exact "confident garbage" failure this suite is built to guard against.

**Strengthened recommendation.** Adopt a **DACI/RAPID-shaped split with a reversibility-gated human backstop**, not pure voting and not an autonomous AI final decider:

- **Process authority → Scrum Master agent (the Driver).** Owns the timebox, the round cap, and *calling* the vote. But — the load-bearing amendment — the SM agent may only *request* convergence; it does **not** hold the clock. **Convergence machinery lives server-side:** the Board keeps a wall-clock deadline + round counter on the huddle ticket and **auto-transitions a stuck huddle to `needs_review` regardless of any agent's activity.** An agent-run timer can itself hang or mis-count — the system's defining failure mode — so "deadlock never spins" is true *only* because the deterministic Board service, not an agent, enforces it.
- **Content authority → Product Owner agent (the Decider), but only as a *recommender-of-record*, never an autonomous final say.** The PO agent converges the huddle and writes a **decision record into the planning note** (goal, success metric, the dissent it overruled, and its rationale) — mirroring RAPID's "Recommend" plus DACI's single Approver. Because the anti-sycophancy invariant forbids trusting an agreeable AI as the last word, the PO's decision is *provisional* until it clears the existing human/tier gate.
- **The Adversarial Reviewer holds a scoped veto (RAPID's "A"), on narrow grounds only** — an unaddressed premise attack or a violated retro lesson. The amendment makes forced dissent **mechanical**: the AR must record **≥1 grounded premise-attack cited to a recon note**, or the Board marks the huddle **invalid → escalate** (this is the enforcement of "fast consensus is suspicious"). A veto clears only by the AR recording explicit acceptance **or** by the human — **never by the PO it checks** (that would be circular).
- **Reversibility is DERIVED, never reasoned.** The safety of any no-human path hinges on a ticket being correctly labeled reversible/low-blast-radius, and the sycophancy-prone agents must not classify their own escape hatch. Reversibility is derived mechanically from a **fixed task-type table AND the CMDB criticality tier** for the specific host — queried, not reasoned — with **default-deny** on any unknown type or missing tier.
- **Gate the auto-path at the per-server execution ticket, by CMDB tier — not at the huddle.** Reversibility is a property of *this host now* (a patch with a rollback can still brick the NAS; respect the "host that hung on reboot last time" retro lesson), and that per-host risk is unknowable until backlog decomposition creates the per-server execution ticket. So **huddle-escalation and the execution-ticket `awaiting_approval`/tier gate are two independent, sequential gates.** For reversible, low-tier per-host tickets the PO's "disagree and commit" record may flow to execution without a human (the two-way-door lightweight path that keeps the queue moving); anything irreversible or critical-tier hits the human/tier gate downstream regardless.
- **Escalation is automatic and reversibility-gated.** The Board escalates (files `needs_review` with the full transcript note) when *any* of: (a) round cap or timebox is hit without a PO decision; (b) the AR's veto is unresolved or the AR filed no grounded dissent; or (c) the downstream execution ticket is derived-irreversible or critical-tier.

**The backstop that de-fangs the rubber-stamp fear: the huddle is never the last signature.** Per the segregation-of-duties invariant, even a fully-compromised huddle emitting a malicious plan cannot cause a destructive action: each per-host execution ticket independently hits the **CMDB tier gate**, `awaiting_approval`, and the **Gateway** (approved-ticket + policy + Vault-brokered creds). A rubber-stamping AI decider is therefore *contained* for destructive work — its blast radius is wasted compute and bad plans, not root on a host. The huddle decision must flow **into** that tier/approval chain, never around it; the reversible auto-path is safe **only** because CMDB tier gates it downstream. **State it explicitly: the huddle is not a segregation-of-duties chokepoint.**

Net: process is agent-owned but **server-enforced** (deterministic, safe to automate); content is agent-*recommended* but human-*ratified whenever it is irreversible or contested*; deadlock never spins because the Board — not an agent — mechanically converts a stuck huddle into a human escalation. This satisfies the Scrum single-accountable-PO precedent while refusing to let a sycophancy-prone AI be the last signature on a one-way door.

## 10. DEFERRED DECISION #2 — Ceremony scales to complexity (triage routing threshold)

> **Adversarial review (holes found; recommendation survives, amended).** The core recommendation — tier planning rigor by risk, and never let triage buy a ticket out of the execution gates — survives; the ITIL and Bezos precedent is correctly read. Five holes in composition with local sycophantic agents + segregation-of-duties: (1) the reversibility signal does double duty — one mis-call both routes to straight-to-execute AND suppresses the human `awaiting_approval` gate, so the two must be **decoupled**; (2) "known-good runbook, minted on first run" quietly makes the **executing agent its own change authority over time** (the invoice-approver-cuts-the-check violation), worsened by host drift; (3) the fail-safe keyed on **self-reported LLM confidence** is exploitable — LLM confidence is measurably miscalibrated and down-tiering is the cheap path; (4) **per-child triage is blind to aggregate blast radius** and can route the canonical 20-host epic straight past ceremony; (5) an external verifier is **post-hoc** and must never offset irreversibility or tier in the "lightweight" lane. The strengthened recommendation replaces the LLM score with a deterministic table, sources safety-critical signals from non-agent authorities, and binds triage at the epic level.

**Findings.** The question ARCHITECTURE.md defers is whether a triage step should route trivial/reversible tickets *past* the full planning huddle, and by what threshold. Three independent bodies of precedent converge on the same answer — **yes, tier the process by risk** — and each contributes a usable dimension.

*ITIL change enablement* is the closest operational analogue and maps almost one-to-one onto skip-vs-full-ceremony. It defines three change types. A **standard change** is "low-risk, commonly repeated, and pre-approved"; because the procedure "has already gone through the risk assessment and approval process, it doesn't need to go through the process again," and is "a great candidate for automation" ([Atlassian, Change management types](https://www.atlassian.com/itsm/change-management/types)). Critically, a standard change is "pre-authorized *by a change authority*... low-risk, well-understood, and has a repeatable process" — the pre-approval is granted by an authority, not by whoever executes it ([SpocLearn, Standard Change in ITIL 4](https://www.spoclearn.com/blog/itil-4-definition-of-standard-change/)). A **normal change** has no pre-approved path and "should be scheduled, have its risk assessed, and be authorized," often sub-graded Major/Significant/Minor by risk ([IT Process Wiki, Change Management](https://wiki.en.it-processmaps.com/index.php/Change_Management)). An **emergency change** "must be implemented immediately" and uses expedited approval but still requires a post-implementation review ([Atlassian](https://www.atlassian.com/itsm/change-management/types); [IT Process Wiki](https://wiki.en.it-processmaps.com/index.php/Change_Management)). The load-bearing lesson: the *first* execution of a repeatable action earns full rigor and thereby *mints a pre-approved runbook*; later executions of the unmodified runbook skip it — **but only if the promotion is a change-authority act, not a self-tag.** Atlassian explicitly advocates "dropping the 'one size fits all' approach, and treating each change individually based on risk assessment" ([Atlassian](https://www.atlassian.com/itsm/change-management/types)).

*Reversibility* is Bezos's axis. Type 2 / two-way-door decisions are reversible: "Those decisions can use a light-weight process." Type 1 / one-way-door decisions are consequential and nearly irreversible and "must be made methodically, carefully, slowly." His warning is directly relevant to a system whose failure mode is over-deliberation: large organizations "tend to use the heavyweight Type 1 decision-making process on most decisions, including many Type 2 decisions," yielding "slowness, unthoughtful risk aversion... and consequently diminished invention" ([Amazon, 2016 Letter to Shareholders](https://www.aboutamazon.com/news/company-news/2016-letter-to-shareholders)).

*Agile right-sizing* supplies the caution that ceremony is not free: "When ceremonies continue on schedule but decisions do not change, they stop acting as control mechanisms and become procedural overhead," and right-sizing means "tailoring [process] to meet the needs of your team's unique situation" rather than adopting a maximal framework ([Mountain Goat Software, Just Right Size](https://www.mountaingoatsoftware.com/blog/the-just-right-size-for-agile-teams)). For local, no-API-cost agents this is doubly true: chattiness is free, so the true cost of full ceremony is *wall-clock latency and never-terminating deliberation*, not tokens — exactly the failure mode to guard against.

The fail-safe cannot rest on the model's own confidence: self-reported LLM confidence is structurally miscalibrated — GPT-4 assigned its top confidence to ~87% of answers including wrong ones, and "an agent that has already committed to an action has an incentive to justify rather than critique it" ([medRxiv, Overconfident AI?](https://www.medrxiv.org/content/10.1101/2024.08.11.24311810v1); [Medium/DSC, The Illusion of Confidence](https://medium.com/data-science-collective/the-illusion-of-confidence-why-asking-your-llm-are-you-sure-is-a-terrible-idea-84eb5859fc26)).

> **Gotcha:** "Reversible" and "known runbook" are not the same axis and must both be checked. A well-understood runbook applied to a Tier-0 host is still a one-way door. Do not collapse novelty and blast-radius into a single score.

**Strengthened recommendation.** Have the Scrum Master agent run a **triage gate** that supplies signal *values*, but harden it against the agent that runs it — the **lane is computed by a deterministic decision table, not an LLM score**. Five orthogonal signals feed the table:

1. **Reversibility** — is there a rollback path (two-way door) or is the action destructive/irreversible (one-way door)? Derived from the task-type registry + CMDB tier, never reasoned ([Amazon 2016 letter](https://www.aboutamazon.com/news/company-news/2016-letter-to-shareholders)).
2. **Blast radius** — number of hosts/resources touched; single-host < small batch < fleet-wide.
3. **Host criticality tier** — pulled from **CMDB**, which already holds tier + auto-vs-ask policy. Triage must *query*, not guess, this signal.
4. **Novelty** — is this an unmodified, previously-approved runbook (ITIL "standard change") or a first/modified execution ("normal change")? ([Atlassian](https://www.atlassian.com/itsm/change-management/types))
5. **External verifier present** — does an independent confirmer (e.g. Wazuh active→solved) exist? Its presence safely justifies a *lighter plan*, never a lighter approval.

The five hardening amendments that make this survive composition:

1. **Make the lane a deterministic decision table over signals, not an LLM judgment.** The routing function is a fixed rule; the agent supplies signal *values*, the Board computes the lane. **Straight-to-execute requires reversible AND known-runbook AND low tier AND external-verifier — all four, hard-coded.**
2. **Source the two safety-critical signals from non-agent authorities.** Host-criticality tier comes from **CMDB** (already a separate holder); task-type reversibility/destructiveness comes from a **human-owned task-type registry**, not the agent's per-ticket guess. Decisively, the **`awaiting_approval` gate fires on the OR of (registry says destructive) ∨ (CMDB tier policy says "ask") — fully decoupled from triage**, so a mis-triage can never suppress the human gate.
3. **The runbook / standard-change catalog is a change-authority-owned, host-state-versioned store the executing agent cannot write.** Promotion to "standard change" is a human/CMDB act (mirrors ITIL's change authority); catalog entries are invalidated on host drift. This preserves approver≠executor **across time** — closing the "agent self-mints its own pre-approval" hole.
4. **Triage binds at the epic/campaign level; spawned children inherit the parent's lane as a floor** and carry its plan slice — they are **never re-triaged downward** into straight-to-execute. This stops the 20-host fleet epic from being decomposed into 20 individually-trivial-looking tickets that each skip the ceremony the epic exists to force.
5. **The fail-safe is structural, not confidence-based.** Any missing/stale authority signal, any epic touching **>1 host or any critical-tier host**, or a triage **timeout** → **Full ceremony**. Any skip of full ceremony must be **ratified by an independent Adversarial Reviewer, never self-certified**. Deadlock/timeout escalation must **halt-and-notify the operator, never auto-proceed** — stall is the safe direction; silent proceed is not.

Routing outcomes (lanes computed by the table above):
- **Straight-to-execute** — reversible AND known-good runbook AND low tier AND external verifier exists. Emit the execution ticket directly; the `awaiting_approval` gate and CMDB tier policy still apply per segregation-of-duties.
- **Lightweight ceremony** — one mixed signal (e.g. known runbook but higher tier, or new but trivially reversible): a compressed pass — Product Owner scope check + a single Adversarial Reviewer challenge, capped at one round, no full standing huddle. **A post-hoc external verifier must never offset irreversibility or tier** into this lane.
- **Full ceremony** — irreversible OR fleet-wide OR any critical-tier host OR novel high-tradeoff work (the canonical Wazuh epic). Full triage→recon→planning→adversarial→backlog→execute→retro.

Critically, **the triage lane governs only the *planning* rigor, never the execution gates.** The `awaiting_approval` human/tier gate, CMDB policy, Vault brokering, and per-host mutex fire regardless of lane — triage can make planning cheaper but can *never* buy a ticket out of segregation-of-duties. Record the triage decision, the five signal values, **their source authority**, and the computed lane as structured fields on the ticket (and mirror into the huddle note) so the retro agent can audit mis-triage and tune thresholds over time.
