# RESEARCH.md — Notes (`notes`), Stage 1

> Compiled **2026-07-01**. This is the Stage-1 research artifact for the **notes** app: the suite's external memory and work product, where **markdown-on-disk is the canonical store** and the database is a rebuildable index. Risk class: **Standard**. The document surveys the feature landscape across ten research topics, records integration facts with inline citations, resolves the two decisions ARCHITECTURE.md explicitly deferred to research (editor: Milkdown vs TipTap; index/storage: SQLite FTS5 rebuildable index), and lists what must be re-verified at build time. Where a version or date could not be pinned from a primary source, it is pushed to the *Verify at build time* list rather than asserted. Every external fact carries an inline `[label](url)` citation.

## Executive summary

- **Storage model is a solved shape, copied from Obsidian + Dendron:** plain `.md` files in a folder are canonical; the DB is a rebuildable cache (Obsidian's own IndexedDB-metadata pattern). Notion, Confluence, and the new Logseq-DB are the anti-pattern — DB/block-canonical stores that force lossy API round-trips and kill free git diffing ([Obsidian data storage](https://obsidian.md/help/data-storage), [Notion data model](https://www.notion.com/blog/data-model-behind-notion)).
- **The one real hazard is concurrent writes:** an agent process and a human editor share one file. Obsidian's documented failure mode is an open editor buffer clobbering an external write ([Obsidian forum](https://forum.obsidian.md/t/data-loss-issue-sync-conflict-handling/113584)). Serialize all writes through one API, use atomic write-then-rename with mtime/hash preconditions, and live-refresh the editor on external change.
- **The FTS5 index is disposable by design.** SQLite FTS5 offers external-content and contentless modes plus a `'rebuild'` command that regenerates the whole index from content — exactly the "markdown is truth, DB is rebuildable" invariant ([SQLite FTS5](https://www.sqlite.org/fts5.html)). Recovery is `rm notes.db` → rescan files → rebuild.
- **DEFERRED DECISION 1 (editor) → Milkdown.** Both Milkdown 7.21.2 and TipTap 3.27.1 are MIT and ProseMirror-based, but Milkdown's canonical model *is* markdown (via remark), so round-trips are lossless by construction; TipTap's canonical model is ProseMirror JSON with markdown as an early-release serialization layer ([@milkdown/core](https://registry.npmjs.org/@milkdown/core/latest), [@tiptap/core](https://registry.npmjs.org/@tiptap/core/latest)).
- **DEFERRED DECISION 2 (index/storage) → SQLite FTS5 external-content table** over a shadow table regenerated from files, rebuildable via `INSERT INTO fts(fts) VALUES('rebuild')`, `porter unicode61 remove_diacritics 2` tokenizer ([SQLite FTS5](https://www.sqlite.org/fts5.html)).
- **Search is a tool, not a dump:** a `search_notes` MCP tool returns `bm25()`-ranked, `snippet()`-bounded hits + IDs (limit ~10, hard cap ~25), never full bodies; agents call `read_note(id)` on demand ([SQLite FTS5](https://www.sqlite.org/fts5.html)).
- **MCP surface:** build on `@modelcontextprotocol/sdk` 1.29.0 over **Streamable HTTP** behind the proxy; agents are scoped OAuth users with append-biased tools and a higher scope gating destructive overwrites ([MCP SDK](https://registry.npmjs.org/@modelcontextprotocol/sdk/latest), [MCP authorization 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization)).
- **Git audit is nearly free:** commit per logical agent action (sharing the index debounce boundary), author = acting agent, committer = Notes service, using pure-JS **isomorphic-git 1.38.6** (no git binary in the image) ([isomorphic-git](https://registry.npmjs.org/isomorphic-git/latest)).
- **Deliberation lives here, not in Chat:** each planning huddle is one `type: deliberation` markdown note with append-only per-turn headers — one replayable git diff, one reviewable artifact.
- **Conversion is the pdf app's job:** Notes calls out to a Pandoc 3.10 service (Typst as primary PDF engine, ~27x faster than XeLaTeX) that writes artifacts to Drive; Notes stays markdown-only ([Pandoc releases](https://pandoc.org/releases.html), [Typst benchmark](https://slhck.info/software/2025/10/25/typst-pdf-generation-xelatex-alternative.html)).

## 1. Comparable knowledge bases as agent-writable stores

The notes app's requirement is unusual: an external AI process writes files directly, concurrently with a human editing through a WYSIWYG UI, and markdown-on-disk is canonical with git as the audit trail. Judged against that bar, five reference tools split cleanly into "file-canonical" (safe to target) and "DB-canonical" (hostile to external writers).

**Obsidian — plain markdown files, file-canonical (closest fit).** Obsidian stores notes as Markdown plain-text files in a vault (a filesystem folder); "nothing is locked in a database," so any external tool can open, edit, or back up the files, and the app "automatically refreshes your vault to keep up with any external changes" ([Obsidian data storage](https://obsidian.md/help/data-storage)). The only DB is an IndexedDB **metadata cache** for Graph/Outline performance — rebuildable, explicitly not canonical. Frontmatter is first-class ("Properties," YAML in a `---` block, powering search/filters); `[[wikilinks]]` and a backlinks pane are native ([Obsidian Properties](https://help.obsidian.md/properties)). The one real risk: if a file is open with an unsaved buffer and an external process writes it, the app's later save can clobber the external edit; data-loss reports cluster around sync services racing the file ([Obsidian forum — sync conflicts](https://forum.obsidian.md/t/data-loss-issue-sync-conflict-handling/113584), [full file loss after crash](https://forum.obsidian.md/t/full-file-data-loss-after-obsidian-crash/71393)). Lesson: last-writer-wins on an open buffer is the hazard, not the on-disk format.

**Dendron — plain markdown files, file-canonical, developer-oriented.** Local-first, hierarchical, git-native ("a vault is just like a code repo") ([Dendron concepts](https://wiki.dendron.so/notes/c6fd6bc4-7f75-4cbb-8f34-f7b99bfe2d50/)). YAML frontmatter is autogenerated on note creation and carries an **immutable `id`** field — you may change every other field but not `id`; hierarchy is encoded in dot-delimited filenames (`proj.area.topic.md`); wikilinks/backlinks/hierarchical links supported ([Dendron FAQ](https://wiki.dendron.so/notes/683740e3-70ce-4a47-a1f4-1f140e80b558/)). The stable-`id` idea is directly transplantable so notes survive renames.

**Notion — block database, NOT file-canonical (avoid as a write target).** Every content unit is a **block** with a UUID v4 id, type, properties, content and parent pointers, stored as structured records in a source-of-truth database, not files ([Notion data model](https://www.notion.com/blog/data-model-behind-notion)). The API speaks custom JSON, not markdown, because markdown can't represent Notion block types; exporting to markdown is a lossy pipeline ([Notion block reference](https://developers.notion.com/reference/block)). Backend is PostgreSQL on RDS with SQLite/IndexedDB only as a client cache ([How Notion was built](https://howworks.ai/blog/how-notion-was-built) — secondary source, flagged). There is no file to commit, no diff, no way for an external process to write canonical state.

**Logseq — mid-migration, format is version-dependent (cautionary tale).** Logseq has split into "Logseq OG" (file-based markdown graph) and a new **DB version** storing to SQLite where **markdown is no longer the source of truth** — markdown export becomes a one-way backup ([Logseq db-version.md](https://github.com/logseq/docs/blob/master/db-version.md)). OG (markdown) releases remain on v0.10.x; dual-support builds begin at v0.11.x, still pre-release/beta ([Logseq releases](https://github.com/logseq/logseq/releases)). Note: the DB doc's "April 28th, 2026" is a documentation *as-of* timestamp, **not a GA date** — the DB version is still an invite-only rollout with no announced general availability ([Logseq db-version raw](https://raw.githubusercontent.com/logseq/docs/master/db-version.md)). The lesson is a warning: a tool that starts file-canonical and later moves canonical state into a DB breaks every external writer and the git trail at once. Our architecture makes markdown-canonical a *locked invariant* precisely to never take Logseq's DB turn.

**Confluence — XHTML storage in a DB, NOT file-canonical (avoid).** Pages are stored in "Confluence Storage Format," an XHTML/XML derivative with macro tags — not markdown — in the app database, reached via REST API ([Confluence Storage Format](https://confluence.atlassian.com/doc/confluence-storage-format-790796544.html)). Publishing markdown requires a converter (md → storage format) ([md2conf](https://github.com/hunyadi/md2conf)). Same disqualifier as Notion.

**Design lessons.** *Copy:* markdown files as canonical + rebuildable index cache (Obsidian); auto-refresh on external change (filesystem watcher); YAML frontmatter with a stable immutable `id` (Dendron); computed backlinks; optional filename-encoded structure. *Avoid:* DB/block-canonical stores; API/format round-trips to write content; long-lived unsaved editor buffers that last-writer-wins over disk; ever moving canonical state into a DB. **The load-bearing implication:** two writers share one file, so the app needs explicit write reconciliation — mtime/hash checks, serialized writes through one API, atomic write-rename — not just git recovery after the fact.

## 2. Markdown-as-truth with a rebuildable SQLite/FTS5 index and a file-watcher

The invariant maps cleanly onto SQLite FTS5's own supported design. FTS5 offers storage modes built for the case where real content lives elsewhere ([SQLite FTS5](https://www.sqlite.org/fts5.html)):

- **External-content** (`content='note', content_rowid='id'`): the FTS index references a separate SQLite shadow table for column values.
- **Contentless** (`content=''`): FTS5 stores only the inverted index, no copies of note bodies. Since **SQLite 3.43.0** there is also **contentless-delete** (`content='', contentless_delete=1`) adding DELETE / INSERT OR REPLACE / UPDATE support.

For a notes app the pragmatic layout is a plain SQLite `note` mirror table (path, frontmatter columns, body, hash, mtime) plus an FTS5 index. **External-content is the better fit** because you want SQL filters on frontmatter (status/type/tags) alongside MATCH, and FTS5's `snippet()`/`highlight()` need the column text — and because `'rebuild'` is **not available on contentless tables**.

**Recovery is a first-class feature.** `INSERT INTO notes_fts(notes_fts) VALUES('rebuild');` "deletes the entire full-text index and rebuilds it" from current content — the documented fix for any situation where index and content diverge ([SQLite FTS5](https://www.sqlite.org/fts5.html)). Full disaster recovery: rescan every `.md` → repopulate mirror table → rebuild; ultimate fallback is `rm notes.db` and reindex from files. The docs warn MATCH vs non-MATCH queries can contradict each other when index and content diverge, so a boot-time/periodic rebuild is cheap insurance.

**File-watcher (Node stack).** **chokidar 5.0.0** is the latest; it is **ESM-only** and requires **Node ≥ 20.19.0** (published `engines` field), released **2025-11-25** ([chokidar latest](https://registry.npmjs.org/chokidar/latest), [chokidar 5.0.0 release](https://github.com/paulmillr/chokidar/releases/tag/5.0.0)). If the service is CommonJS or on older Node, pin chokidar 4.x. Two options address atomic writes directly ([chokidar README](https://github.com/paulmillr/chokidar)): `awaitWriteFinish` (holds add/change events until file size stabilizes) and `atomic` (default on, 100 ms — filters temp-file artifacts, emitting `change` rather than `unlink`+`add` when a file is re-added within 100 ms).

**File-watcher (Rust stack).** **notify 8.2.0** (max stable; a 9.0.0-rc exists but is pre-release) uses native backends — inotify (Linux), FSEvents/kqueue (macOS), ReadDirectoryChangesW (Windows) — and points to a debouncer ([notify crate](https://crates.io/crates/notify), [docs.rs/notify](https://docs.rs/notify/latest/notify/)). Use **notify-debouncer-full 0.7.0** (ordered, correlates rename/move) ([crates.io](https://crates.io/crates/notify-debouncer-full)) or **notify-debouncer-mini 0.7.0** (one event per file per timeframe) ([crates.io](https://crates.io/crates/notify-debouncer-mini)).

**Debounce and atomic-write gotchas (esp. Windows).** One logical save = multiple filesystem events: editors write a temp file then rename it over the target, so a watcher sees CREATE(temp) → RENAME → sometimes a trailing WRITE; on Windows `ReadDirectoryChangesW` reports rename pairs and can drop/merge events under load ([file watchers lie](https://medium.com/@impactarchitecture/file-watchers-lie-debounce-throttle-and-coalescing-in-build-loops-8d91cb29f712), [Rust debouncing](https://oneuptime.com/blog/post/2026-01-25-file-watcher-debouncing-rust/view)). Practical guidance: debounce per-path (~50–100 ms is negligible latency while collapsing the burst); **watch the parent directory, not the single file** (an atomic rename replaces the inode, killing a file-level watch); reconcile by **content hash, not by event** so duplicate events are idempotent and missed events self-heal on the next touch.

**External edits and git checkouts.** The watcher alone is insufficient: a `git checkout`/`pull`/`stash`/branch switch rewrites many files with no reliable per-file event storm. Robust pattern: watcher = fast path, **full scan-and-reconcile = correctness backstop**, run (a) at process start, (b) after a git HEAD change (watch `.git/HEAD` and `.git/refs`, or hook your own commit path), and (c) on demand. Reconcile walks the tree, diffs (path, mtime, hash) against the mirror, applies inserts/updates/deletes, then rebuilds on detected drift. Because the DB carries no authority, an occasional full rebuild is always safe.

## 3. WYSIWYG-stores-markdown editor: Milkdown vs TipTap [DEFERRED DECISION]

Both editors are built on **ProseMirror** and both are **MIT-licensed** and free at their core. The decisive difference is *where markdown sits in the data flow*.

**Current versions and license (npm registry).** `@milkdown/core` **7.21.2** MIT ([registry](https://registry.npmjs.org/@milkdown/core/latest)); `@milkdown/crepe` (batteries-included editor) **7.21.2** MIT ([registry](https://registry.npmjs.org/@milkdown/crepe/latest)); `@tiptap/core` **3.27.1** MIT ([registry](https://registry.npmjs.org/@tiptap/core/latest)). TipTap's core is "free forever" and it recently open-sourced 10 formerly-Pro extensions under MIT ([TipTap pricing](https://tiptap.dev/pricing), [HN](https://news.ycombinator.com/item?id=44202103)).

**Is markdown native? — the architectural split.**
- **Milkdown: yes, natively.** Its pipeline converts Markdown string ⇄ remark AST ⇄ ProseMirror doc; remark handles both parse and serialize, so the document round-trips through markdown by construction, and GFM is a plugin over remark ([Milkdown architecture](https://milkdown.dev/docs/guide/architecture-overview), [DeepWiki](https://deepwiki.com/Milkdown/milkdown/1.2-architecture-overview), [Milkdown repo](https://github.com/Milkdown/milkdown)). Custom syntax is added at the AST level where it survives serialization.
- **TipTap: markdown is a serializer layer, and it is new.** Historically TipTap had no first-party markdown; teams used the community `tiptap-markdown` wrapping `prosemirror-markdown` ([tiptap-markdown](https://github.com/aguingand/tiptap-markdown)). The v3 line shipped an **official** `@tiptap/markdown` extension, first published in **3.7.0**, bidirectional and built on MarkedJS (CommonMark-compliant) ([bidirectional markdown](https://tiptap.dev/blog/release-notes/introducing-bidirectional-markdown-support-in-tiptap), [@tiptap/markdown](https://registry.npmjs.org/@tiptap/markdown)). But TipTap's own docs describe it as an early release that "can be subject to change or may have edge cases," and the **core model remains ProseMirror JSON/HTML** — markdown is produced by translating tokens ⇄ nodes, not by storing markdown ([basic usage](https://tiptap.dev/docs/editor/markdown/getting-started/basic-usage), [markdown-manager API](https://tiptap.dev/docs/editor/markdown/api/markdown-manager)).

**Round-trip fidelity by feature.** Basic blocks and inline marks are solid in both. Tables and task lists work in both, but in TipTap they are not in StarterKit — you add Table/TaskList extensions and enable `markedOptions: { gfm: true }`; Milkdown ships GFM via remark. Footnotes are not CommonMark, so TipTap's MarkedJS extension needs manual parse/serialize rules, while Milkdown can add them via a remark plugin at the AST layer (no official Milkdown footnote doc page was found — treat as a custom-plugin task in either editor). **Frontmatter and `[[wikilinks]]` are non-standard in both** — the important caveat for Notes: in Milkdown they are remark plugins + custom nodes staying inside the markdown pipeline (serialize cleanly); in TipTap you author a custom node plus `parseMarkdown`/`renderMarkdown` rules in the MarkdownManager, and anything unhandled risks being dropped or HTML-ified because the canonical store is JSON. TipTap preserves unrepresentable content as embedded HTML, so non-lossy round-trips depend on every custom node having a hand-written serializer (even empty-paragraph preservation needs `&nbsp;` tricks) ([basic usage](https://tiptap.dev/docs/editor/markdown/getting-started/basic-usage)).

**Ecosystem and commercial dimension.** TipTap has the larger ecosystem and a commercial platform; Collaboration, Comments, Snapshots, Conversion, and Content AI remain paid, subscription-gated add-ons ([TipTap pricing](https://tiptap.dev/pricing), [new pricing model](https://tiptap.dev/blog/release-notes/tiptaps-new-pricing-model-is-live)). For a self-hosted, no-cloud, local-agent org these are largely irrelevant. Milkdown is smaller, purely OSS/MIT, and markdown-first, with `@milkdown/crepe` as a batteries-included starting point ([Crepe guide](https://milkdown.dev/docs/guide/using-crepe)).

**Recommendation: Milkdown.** Notes' non-negotiable invariant is "markdown is the source of truth." The editor whose canonical document model *is* markdown via remark is the correct structural fit: round-trips are lossless by construction, GFM is built in, and custom syntax (frontmatter, wikilinks, footnotes) lives at the remark-AST layer where it serializes faithfully. **The tradeoff, stated plainly:** picking Milkdown trades TipTap's bigger ecosystem, larger contributor base, and commercial Content-AI/collaboration path for architectural alignment with the markdown-truth invariant and lower round-trip risk. For a Standard-risk, self-hosted, no-API-cost app where markdown canonicity is a build-failure-level invariant, that trade is clearly worth making. If instead you weighted ecosystem momentum and off-the-shelf collaboration/AI over markdown purity, TipTap would be the pick — but it would demand rigorous per-node serializer coverage and round-trip tests to satisfy the invariant.

## 4. YAML frontmatter schema

Frontmatter is the robust, diffable metadata carrier (Obsidian "Properties," Dendron autogenerated blocks). Proposed conventions (prose, not a schema): every note carries a YAML `---` block with:

- **`id`** — a stable, immutable identifier (Dendron's model) so links and backlinks survive renames/moves without mutating the human-visible filename ([Dendron FAQ](https://wiki.dendron.so/notes/683740e3-70ce-4a47-a1f4-1f140e80b558/)).
- **`type`** — the note template kind (research / planning / retro / deliberation / etc.), driving the per-type structured template.
- **`status`** — the ticket-lifecycle-aligned state (e.g. `todo / in_progress / needs_review / done`, plus ceremony states for deliberation notes).
- **`links`** — `[[wikilinks]]` to related notes (associative memory); **`tags`** — freeform classification.
- Timestamps and authorship as needed (created/updated, owning ticket id).

Frontmatter keys become SQL-filterable columns in the mirror table (status/type/tags), enabling scoped search. Note the export caveat (Topic 9): internal keys must be stripped/remapped before Pandoc conversion so they don't leak into PDFs/docx. Frontmatter is the more robust metadata carrier than filename-encoded structure, though Dendron shows dot-hierarchy filenames are a legitimate optional complement.

## 5. Wikilinks, backlinks, graph, and FTS5 search as an agent tool

**Links as associative memory.** Native `[[wikilinks]]` plus a computed backlinks index (both derivable from the files) give the associative-memory substrate. A stable `id` in frontmatter keeps links resolvable across renames. `list_backlinks(id)` returns notes linking *to* a note; the graph is the transitive closure of these links, all rebuildable from the markdown.

**FTS5 as a scoped SEARCH TOOL, not a context dump.** FTS5 ships the exact primitives ([SQLite FTS5](https://www.sqlite.org/fts5.html)):
- **`bm25(fts [, w0, w1, ...])`** — built-in Okapi BM25, hard-coded `k1=1.2`, `b=0.75`; the raw score is multiplied by -1 so **numerically lower = better** and ascending `ORDER BY` puts best hits first. Per-column weights boost title over body (`bm25(notes_fts, 10.0, 1.0)`).
- **`snippet(fts, col, start, end, ellipsis, max_tokens)`** — a short fragment (max_tokens 1–64) chosen to maximize query terms, with matches wrapped in your markup. This is the key primitive for returning just the relevant window (e.g. `snippet(notes_fts, -1, '**', '**', ' … ', 32)`).
- **`highlight(fts, col, start, end)`** — wraps matches in the *entire* column text; reserve for the UI when the whole note is already loaded, use bounded `snippet()` for tool output.
- **Tokenizers:** `unicode61` (default, `remove_diacritics 0|1|2`), `porter` (English stemming so "patching" matches "patch"), `ascii`, and `trigram` (substring/`LIKE`/`GLOB` matching, larger — useful as a secondary code-token index). Recommended default: `porter unicode61 remove_diacritics 2`.

**Shaping `search_notes`** — return hit rows, not documents. *Input:* an FTS5 MATCH `query` (supports `AND`/`OR`/`NOT`, `NEAR`, `"phrase"`, `col:term`, prefix `term*`) plus scoping params (`limit` default ~10, hard cap ~25, `offset`, and frontmatter filters `type`/`status`/`tag`/`linked_from` applied as SQL `WHERE`). *Output per hit:* `note_id`/`path`, `title`, one `snippet()` fragment (~32–64 tokens), and the `bm25()` score — never the full body, ordered by score. *Contract:* the agent gets enough to decide what to open next, then calls `read_note(id)` for full text of only the notes it needs. Always parameterize/validate the query — never string-concatenate raw agent input into SQL — and consider stripping/escaping FTS5 special characters unless you intend to expose the full grammar.

## 6. Structured note templates per type

Per-type templates are constrained outputs the agent fills in (ARCHITECTURE.md §5). Each `type` maps to a template with fixed section headers, e.g. **Research:** Objective / What I did / Findings / Open questions / Next step. Other types include **Planning**, **Retro** (reads git audit + external before/after, writes lessons that feed next-cycle recon), and **Deliberation** (Topic 8). `create_note(type, ...)` instantiates the correct template, enforcing structure at creation. Templates keep agent output decomposable and reviewable, raise the floor on structured work, and make the FTS index and human review predictable. The template set lives as service config, not hard-coded in the model.

## 7. Git-backed audit trail

Notes are files, so every agent edit is already a diffable, reversible unit — you just commit at the right granularity with the right authorship. Git records **author and committer as two separate identities** ("the author is the person who originally wrote the work, whereas the committer is the person who last applied the work"), which diverge naturally on rebase/amend ([Pro Git — viewing history](https://git-scm.com/book/en/v2/Git-Basics-Viewing-the-Commit-History)).

**Library options.** **isomorphic-git 1.38.6** is pure-JavaScript git — no native modules, no `git` binary in the image, runs in Node and browser; `git.commit()` takes explicit `author` and `committer` objects (each `name`, `email`, `timestamp` unix-seconds, `timezoneOffset`) and returns the commit SHA ([isomorphic-git latest](https://registry.npmjs.org/isomorphic-git/latest), [site](https://isomorphic-git.org/), [commit docs](https://isomorphic-git.org/docs/en/commit)). **simple-git 3.36.0** is a thin promise wrapper that shells out to the system `git` binary — full git fidelity and native hooks, but requires `git` in the image and spawns a process per commit ([simple-git latest](https://registry.npmjs.org/simple-git/latest), [npm-compare](https://npm-compare.com/isomorphic-git,nodegit,simple-git)). **Recommendation:** default to **isomorphic-git** (removes the git-binary dependency and makes per-agent author/committer a first-class function argument); keep **simple-git** as fallback for features the reimplementation lacks (native hooks, advanced merges — e.g. pre-commit frontmatter validation).

**Authorship attribution.** Set **author** to the acting agent (name = role/id, email = stable synthetic like `security-engineer@agents.local`) and **committer** to the Notes service (the process that applied it). This mirrors git's own semantics and gives a tamper-evident "who wrote vs who applied" split, so `git log --pretty="%an <%ae>"` and `git blame` attribute each edit correctly. On the CLI/simple-git path use `git commit --author="Agent Name <email>"` and/or `GIT_AUTHOR_*`/`GIT_COMMITTER_*` env vars per invocation ([Pro Git](https://git-scm.com/book/en/v2/Git-Basics-Viewing-the-Commit-History)).

**Commit granularity (a genuine design choice).** Commit-per-edit maximizes audit granularity but risks very high commit volume from continuous chatty agents; batched commits stay readable but can bury a bad edit. **Recommended middle ground:** commit per **logical agent action / tool-call**, not per filesystem-write event, tying the commit boundary to the same debounce boundary used for the index (one settled save = one commit) with ticket id + agent id + action in the message. Since the index reconciles from files regardless, commit cadence and index-update cadence share one debounced trigger.

## 8. Deliberation-thread-as-note

This construct is architecture-driven (ARCHITECTURE.md §6, Notes CLAUDE.md), so no external citations apply beyond markdown/frontmatter conventions. The Board's planning ceremony conversation — independent drafts → cross-talk → adversarial review → converged plan — persists as **one structured, threaded markdown note** attached to the planning ticket, giving the operator a reviewable, diffable record. Explicitly **not** the human-facing Chat app (Chat stays a one-way notification stream).

**Why one file, not many.** A single markdown file per planning ticket means each agent turn is one commit / one diff, so the whole huddle replays chronologically in `git log -p` (the non-negotiable audit property); the operator opens one note and reads the entire debate; and anti-sycophancy is structurally visible — independent drafts recorded *before* any cross-talk make anchoring detectable, and the forced red-team dissent is a required section whose absence is obvious in review.

**Structure.** Frontmatter carries `type: deliberation`, the owning `ticket`, `status` (triage/recon/drafting/cross-talk/adversarial/converged/escalated), `ceremony_stage`, `timebox` (convergence kill-switch deadline), `participants`, `decider`, `links` (wikilinks to recon notes), and `tags`. The body is phased: **Phase 1 — Independent drafts (no cross-talk)**, **Phase 2 — Cross-talk**, **Phase 3 — Adversarial review (REQUIRED)**, **Phase 4 — Convergence**. Turn conventions: one turn = one heading + one append via the append-only `append_to_note` tool (turns never overwrite each other; no agent edits another's turn); attribution in the header (`### <TurnType> — @agent-role · <ISO-8601 UTC> [· re: @target]`) for machine-parseable, minimal, additive diffs; phase ordering enforces the ceremony state machine (drafts before cross-talk kills anchoring; adversarial review mandatory before convergence); ground truth is linked via `[[wikilinks]]` not inlined (debate grounded in retrieved findings, keeping the thread compact and the FTS index clean); convergence writes the outcome into frontmatter (`status: converged`) so the Board can trigger backlog decomposition and the retro agent can later read the record, while a timed-out huddle sets `status: escalated` and files a `needs_review`. The construct reuses the same primitives as Topics 5–7: a `type: deliberation` template, append-only turns, wikilinks, and FTS5 search.

## 9. Pandoc md→PDF/docx conversion

**Version.** Latest is **Pandoc 3.10, released 2026-06-03** (recent cadence: 3.9.0.2 2026-03-19, 3.9.0.1 2026-03-17, 3.9 2026-02-03, 3.8.3 2025-12-01); 3.10 includes Docx and Typst improvements ([Pandoc releases](https://pandoc.org/releases.html), [GitHub releases](https://github.com/jgm/pandoc/releases)). **Pin an explicit version tag** in the container for reproducibility; do not track `latest`.

**PDF engine tradeoffs.** Pandoc shells out to a `--pdf-engine`; `-t latex` defaults to `pdflatex`, `-t html` defaults to **weasyprint** ([Pandoc MANUAL](https://pandoc.org/MANUAL.html)).
- **Typst (primary recommendation):** ~**27x faster** than XeLaTeX (356.5 ms ± 3.8 ms vs 9.653 s ± 0.121 s on a 4-page doc), single static binary (no multi-GB TeX Live), equivalent typographic quality, cleaner template language ([Typst benchmark](https://slhck.info/software/2025/10/25/typst-pdf-generation-xelatex-alternative.html)). Official `pandoc/typst` image exists ([Docker Hub](https://hub.docker.com/r/pandoc/typst)).
- **xelatex/lualatex/tectonic (LaTeX route):** highest fidelity, best complex-math/CJK/bidi, but multi-gigabyte TeX Live and slow; `tectonic` (in `pandoc/extra`) downloads packages on demand ([pandoc/extra](https://hub.docker.com/r/pandoc/extra)).
- **weasyprint:** Pandoc's default HTML PDF engine, CSS-styled, but Python + native deps.
- **wkhtmltopdf:** **abandonware** — repo archived 2023-01-02, last stable 0.12.6 (2020-06-11) on Qt 4.8.5; unpatched security risk, **avoid** ([Doppio writeup](https://doc.doppio.sh/article/wkhtmltopdf-is-now-abandonware), [wkhtmltopdf 0.12.6](https://github.com/wkhtmltopdf/wkhtmltopdf/releases/tag/0.12.6)).

**md→docx styling.** docx is not template-driven; supply a **`--reference-doc reference.docx`** whose content is ignored but whose styles + document properties (margins, page size, headers/footers, fonts) are applied. Generate a baseline with `pandoc --print-default-data-file reference.docx`, edit its Word styles, reuse it ([customizing-pandoc](https://github.com/jgm/pandoc/blob/main/doc/customizing-pandoc.md), [MANUAL](https://pandoc.org/MANUAL.html)). Ship **one org-branded `reference.docx`** as service config.

**Frontmatter/metadata mapping.** A YAML block maps into Pandoc template variables; `title`/`author`/`date` populate the title block and become docx/odt document properties and PDF metadata ([metadata blocks](https://pandoc.org/demo/example33/8.10-metadata-blocks.html), [variables](https://pandoc.org/demo/example33/6.2-variables.html)). Custom keys (status/type/wikilinks) are ignored by default templates — **strip or remap internal keys before invoking Pandoc** so they don't leak into exports.

**Docker + ownership.** Official images (`pandoc/minimal`, `pandoc/core`, `pandoc/latex`, `pandoc/typst`, `pandoc/extra`) are tagged by version (e.g. `pandoc/typst:3.10`) ([pandoc/dockerfiles](https://github.com/pandoc/dockerfiles), [pandoc/core](https://hub.docker.com/r/pandoc/core)). Run rootless with no network, mount only the I/O dir, treat each conversion as a short-lived subprocess. **Ownership recommendation:** conversion is a capability of the **pdf app** (ARCHITECTURE.md §4/§9 name pdf as the render tool). Notes calls pdf's render over the API; pdf writes artifacts to Drive by ticket. This keeps Notes markdown-only, centralizes engine/version/reference-doc ownership, and routes non-markdown outputs to Drive. Do **not** embed pandoc in Notes (duplicates the toolchain, splits style ownership, violates the Drive separation).

## 10. Agent MCP surface + human editor surface

**Two views, one state.** The MCP agent surface and the human WYSIWYG editor are siblings over one API over the same markdown files — neither downstream of the other.

**MCP SDK and transport (verify at build time).** Build on `@modelcontextprotocol/sdk` **1.29.0** (`engines.node >=18`) ([MCP SDK latest](https://registry.npmjs.org/@modelcontextprotocol/sdk/latest), [TS SDK repo](https://github.com/modelcontextprotocol/typescript-sdk)). Use **Streamable HTTP** transport (single endpoint, POST client→server, optional GET/SSE server→client) — it superseded the old HTTP+SSE transport, was introduced in the 2025-03-26 spec, and first landed in TS SDK 1.10.0 (2025-04-17) ([transports spec](https://modelcontextprotocol.io/specification/2025-03-26/basic/transports), [why SSE deprecated](https://blog.fka.dev/blog/2025-06-06-why-mcp-deprecated-sse-and-go-with-streamable-http/)). The current stable spec revision is **2025-11-25**; a **2026-07-28 release candidate** is in flight (announced 2026-05-21) ([MCP authorization 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization), [2026-07-28 RC](https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/)). Per ARCHITECTURE.md §9, pin and re-verify the SDK version and spec revision at build time.

**Tool set (mapped to read / write / search / link).** `read_note(id|path)`; `search_notes(query, filters, limit)` (Topic 5); `create_note(type, title, frontmatter, body)` (instantiates the per-type template); `append_to_note(id, section, content)` (append-only); `update_note(id, ...)` (full overwrite — higher-privilege, riskier); `link_notes(from_id, to_id)`; `list_backlinks(id)`.

**Append-only bias (agent safety).** Default the surface to additive tools (`create_note`, `append_to_note`, `link_notes`) so a runaway or confused agent cannot silently clobber external memory, and every addition is a clean small git diff. Reserve full overwrite for narrower authz scopes and the human UI. Because the store is git-backed, even an overwrite is diffable and revertible — but the default should bias to append.

**Scoped authz (agents are scoped users — Standard requirement).** Put authz on the MCP surface, not just the UI: the Notes server is an OAuth 2.1 resource server; the 2025-11-25 spec mandates OAuth 2.0 Protected Resource Metadata (RFC 9728) for discovery and supports machine-to-machine client-credentials flows suited to agent identities from the auth gateway ([MCP authorization 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization)). Validate the caller's token on every tool call, map the agent identity to a role/scope (e.g. `notes:read`, `notes:search`, `notes:append`, `notes:write`), gate destructive overwrite behind a higher scope than read/search/append, and audit-log every state-changing call (create/append/update/link).

**Human editor surface.** A Milkdown-based WYSIWYG editor (Topic 3) storing markdown, plus a review surface for `needs_review` artifacts and browse of `[[wikilinks]]`/backlinks. The editor must detect on-disk changes (filesystem watcher) and reconcile external agent writes rather than last-writer-wins from an unsaved buffer.

## Deferred-decision recommendations

- **Editor (Milkdown vs TipTap) → Milkdown** (`@milkdown/core` + `@milkdown/crepe` 7.21.2, MIT). Its canonical document model is markdown via remark, so round-trips are lossless by construction and custom syntax (frontmatter, wikilinks, footnotes) is added at the remark-AST layer where it serializes faithfully — directly satisfying the "markdown is source of truth" invariant. TipTap 3.27.1 now has a real first-party `@tiptap/markdown` extension and a bigger ecosystem, but its canonical store is ProseMirror JSON/HTML with markdown as an early-release serialization layer, pushing fidelity onto hand-written serializers and risking silent HTML fallback. Commercial Pro/Content-AI features are irrelevant to a self-hosted no-cost deployment.
- **Index/storage → SQLite FTS5 external-content table** over a shadow `note` mirror table (path, frontmatter columns, body, hash, mtime), tokenizer `porter unicode61 remove_diacritics 2`, fully rebuildable via `INSERT INTO notes_fts(notes_fts) VALUES('rebuild')`. External-content keeps `snippet()`/`highlight()` and SQL frontmatter filters working alongside MATCH and is the only mode where in-DB rebuild works (contentless cannot rebuild). Authoritative recovery = delete DB and reindex from files, honoring "markdown is canonical, DB is a rebuildable index."
- **Concurrent-write safety → serialize all writes (agent + UI) through one API over the files**, atomic write-then-rename with mtime/hash preconditions, editor detects on-disk change rather than last-writer-wins from an unsaved buffer. Git enables recovery but should not be the only defense.
- **Editor refresh → filesystem watcher** (chokidar 5.0.0 with `atomic:true` + `awaitWriteFinish`, watching directories not files; or notify 8.2.0 + notify-debouncer-full 0.7.0 in Rust), backed by a full scan-and-reconcile after any git HEAD change.
- **Git library → isomorphic-git 1.38.6** default (pure JS, no binary in the image, per-commit author/committer as first-class args), **simple-git 3.36.0** fallback for native hooks/advanced merges. Author = acting agent, committer = Notes service. Commit per logical agent action sharing the index debounce boundary.
- **Search exposure → one `search_notes` MCP tool** returning `bm25()`-ranked, `snippet()`-bounded hits + IDs (limit ~10, hard cap ~25), never full bodies; `read_note(id)` for full text on demand.
- **MCP transport → Streamable HTTP** on `@modelcontextprotocol/sdk` 1.29.0 behind the proxy; append-biased tools with a higher OAuth scope gating overwrite; audit-log every state change.
- **Conversion ownership → the pdf app** fronts a pinned Pandoc 3.10 service (Typst primary PDF engine, LaTeX/tectonic fallback for complex math; one org-branded `reference.docx` for docx; strip internal frontmatter keys), writing artifacts to Drive. Notes stays markdown-only.

## Open design questions

- Does the WYSIWYG editor need a CRDT/OT layer for true simultaneous human+agent edits of the same note, or is API-serialized single-writer-at-a-time sufficient? (Build-stage decision.)
- External-content vs contentless FTS5: is storing note bodies in the mirror table's disk cost acceptable, or is DB size tight enough to favor contentless (losing in-DB rebuild and `snippet()`)?
- Commit-per-action volume under continuous chatty agents: what cadence is acceptable before repo bloat/gc becomes a concern, and should reconciliation trigger periodic `git gc`?
- Do we need native git hooks (e.g. pre-commit frontmatter validation)? isomorphic-git does not run native hooks, which may force simple-git/CLI for that path.
- Should `search_notes` expose the full raw FTS5 MATCH grammar (NEAR, column filters, prefix, boolean) to agents, or a sanitized subset? Raw agent input must be validated/escaped regardless.
- Primary vs secondary tokenizer: is a secondary `trigram` index wanted for substring/code-token search inside notes, or is one `porter` index enough?
- Exact OAuth scope granularity for agent roles (`notes:read/search/append/write`) and how they map to the auth gateway's agent identities — needs coordination with the auth app's role model.
- Should deliberation-note turns be written directly by each agent via `append_to_note`, or mediated by the Scrum Master agent to enforce phase ordering? Affects concurrency/locking on the single file.
- Does the note-content mix need a LaTeX/tectonic layer alongside Typst frequently enough to justify bundling it, or is Typst-only sufficient?
- Should the conversion service expose an HTTP API wrapper or run as a one-shot subprocess per request (latency vs isolation under multi-agent concurrency)?
- Is weasyprint worth including as a CSS-styled PDF path for certain note types, given the Python/native dependency footprint?
