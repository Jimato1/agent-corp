# Stage 1 Research — `vault` Secrets Vault (Critical-infra)

This document is the Stage-1 research foundation for **`vault`**, the suite's secrets store and the single highest-value target in agent-corp: it holds the long-term plaintext credentials — including **root on ~20 homelab hosts** — that every other control (Board approval, CMDB policy, Gateway execution) exists to fence off. The Vault deliberately **breaks the suite's "two views over one state" rule**: the human operator gets a rich manage/rotate/audit UI, while the agent MCP surface is near-empty by design — an agent may reference a credential by *handle* only, and **only the Gateway may ever redeem that handle to plaintext**. It is one of the four segregation-of-duties (SoD) holders (Board = approval, CMDB = policy, Vault = credentials, Gateway = execution); its whole design assumes an agent will eventually try to exfiltrate secrets and must structurally fail. Every external fact is cited to a real URL. Because this is Critical-infra and several load-bearing facts move monthly (OpenBao/Vault versions, CVEs, the still-unlocked `auth` token model, the MCP authorization spec), version-, license-, and revision-specific items are flagged in a consolidated **Verify at build time** list rather than trusted from memory. Today is **2026-07-01**.

---

## Executive summary

- **ADOPT OpenBao — pin the 2.5.x stable line (≥ 2.5.5), not the 2.6.0 beta — do not build, do not ship BUSL Vault.** OpenBao is the MPL-2.0, Linux-Foundation/OpenSSF-governed fork of the exact Vault codebase whose primitives this app needs (KV v2, dynamic/leased secrets with revocation, deny-by-default ACL policies, the SSH secrets engine, response wrapping, pluggable audit devices), at parity for a single-operator homelab. The adversarial pass *strengthened* this: on the audit-device RCE (Vault CVE-2025-6000 / HCSEC-2025-14, CVSS 9.1) it was the **OpenBao team that found and led the coordinated fix** (OpenBao CVE-2025-54997, patched 2.3.2), refuting "a fork lags on patches." ([openbao.org](https://openbao.org/), [GitHub releases](https://github.com/openbao/openbao/releases), [OpenBao GHSA-xp75-r577-cvhp](https://github.com/openbao/openbao/security/advisories/GHSA-xp75-r577-cvhp))
- **Adopting an engine does not discharge the security stage.** Neither vanilla OpenBao nor Vault enforces the SoD inversion by itself; the "only the Gateway can redeem, agents structurally never can" property lives in **our** wrapper — the Vault app's API/MCP surface + ACL policies binding `vault:read-credential` to the Gateway identity alone. The engine gives us crypto, storage, leasing, and audit plumbing; the no-plaintext-to-agents proof is ours to write in Stage 5.
- **Gateway-only-redeem is enforced by path-scoped ACL on a deny-by-default backend, NOT by response wrapping.** Response wrapping is a single-use *transport/tamper-evidence* primitive; a wrapping token is a **bearer** credential any holder can unwrap. The agent's "handle" must be a powerless application-level reference (e.g. `cred://hosts/nas-01/root`), never a wrapping token. ([Vault policies](https://developer.hashicorp.com/vault/docs/concepts/policies), [sys/wrapping/unwrap](https://developer.hashicorp.com/vault/api-docs/system/wrapping-unwrap))
- **The redeem role must bind on a Gateway-*unique* claim, not on audience alone — and this is blocked on `auth`'s final token model.** Audience is a resource id, not a principal id; if all machine tokens carry `aud=vault`, an agent JWT would satisfy audience-only binding and inherit the redeem policy → plaintext. Bind `bound_subject`/`bound_claims` on a claim agents can never carry. ([Vault JWT/OIDC](https://developer.hashicorp.com/vault/docs/auth/jwt), [v1.17+ bound_audiences](https://support.hashicorp.com/hc/en-us/articles/36321323028755-Vault-v1-17-x-and-above-JWT-auth-login-requires-bound-audiences-on-the-role))
- **Prefer short-lived brokered host access over stored root passwords, via the SSH secrets engine in CA (signed-certificate) mode — but pin OpenBao ≥ 2.0.2 / Vault ≥ 1.17.6 and set `allow_empty_principals=false`.** CVE-2024-7594 (config-by-omission) signs "log in as anyone including root" certs when `valid_principals` is empty. Encode the safe config as a build-failing invariant. ([HCSEC-2024-20](https://discuss.hashicorp.com/t/hcsec-2024-20-vault-ssh-secrets-engine-configuration-did-not-restrict-valid-principals-by-default/70251), [NVD CVE-2024-7594](https://nvd.nist.gov/vuln/detail/cve-2024-7594))
- **The global kill switch does NOT halt an already-issued SSH cert.** A signed cert is validated cryptographically by the host, which never re-contacts Vault; short TTL (minutes, sized to one run) is the primary control, with enforced NTP a hard precondition and a KRL/CA-rotation runbook for true revoke. State this honestly in the SoD proof. ([Vault SSH engine](https://developer.hashicorp.com/vault/docs/secrets/ssh), [OpenSSH PROTOCOL.krl](https://github.com/openssh/openssh-portable/blob/master/PROTOCOL.krl))
- **Audit is fail-closed on auditability, HMAC-salted, and immutable only externally.** Adopt the audit-device model: HMAC-SHA256 every secret value (never `log_raw`), refuse to redeem if no sink can record it, but run **≥2 sinks of different types** so one outage can't DoS the Vault; ship a copy off-box to a WORM/SIEM the Vault host root cannot rewrite (Vault does not cryptographically tamper-proof its own log). ([Vault audit devices](https://developer.hashicorp.com/vault/docs/audit), [audit best practices](https://developer.hashicorp.com/vault/docs/audit/best-practices))
- **The agent MCP surface is a strict four-tool whitelist, enforced by OAuth RS scope checks — not by annotations.** `list_handles`, `describe_handle`, `request_release` (returns an opaque non-redeemable `release_id`), `release_status`; no `read`/`export`/`reveal`/`unwrap` tool is registered on the agent audience at all, and every redemption path is a structural `403 insufficient_scope` for agents. ([MCP 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization), [RFC 6750 §3.1](https://datatracker.ietf.org/doc/html/rfc6750#section-3.1))
- **Treat the container as crown jewels and pin to a patched line.** No standing root token (`operator generate-root` on demand); full production-hardening checklist (unprivileged user, TLS, mlock + no-swap, no core dumps, no wildcard policies); network isolation so only the Gateway, `auth` (JWKS), and the operator UI can reach it; re-pin every CVE against NVD/vendor advisories at build. ([production hardening](https://developer.hashicorp.com/vault/docs/concepts/production-hardening), [HCSEC-2025-22](https://support.hashicorp.com/hc/en-us/articles/43642320190611-HCSEC-2025-22-Multiple-Vulnerabilities-Impacting-HashiCorp-Vault-and-Vault-Enterprise))

## 1. Adopt vs build: OpenBao vs HashiCorp Vault vs rolling our own

**Findings.** This is the highest-leverage decision for the app, and it is not close. Rolling our own secrets store is strictly *more* dangerous than rolling our own auth: a subtle bug in encryption-at-rest, seal/unseal, or audit immutability is not a login inconvenience but total fleet compromise, with no external verifier to catch it. ADOPT is the default BUILD must beat, and the real question is which mature engine.

HashiCorp Vault is the reference implementation of everything this app needs — KV v2, dynamic/leased secrets with automatic revocation, a path-scoped ACL engine, pluggable audit devices, seal/unseal, an [SSH secrets engine](https://developer.hashicorp.com/vault/docs/secrets/ssh) supporting signed SSH certificates and one-time passwords, and [response wrapping](https://developer.hashicorp.com/vault/docs/concepts/response-wrapping) (a single-use cubbyhole token). The current line is Vault 2.0.3 (2026-06-17) alongside the maintained 1.x series; numbering jumped to 2.x at GA on 2026-04-14 ([release notes](https://developer.hashicorp.com/vault/docs/updates/release-notes)). The catch is licensing: since August 2023 Vault ships under the **Business Source License 1.1 (BUSL)**, not an OSI license, and the current [LICENSE file](https://github.com/hashicorp/vault/blob/main/LICENSE) now reads "IBM Corp" after IBM's acquisition of HashiCorp closed (Feb 2025). The grant permits internal/production use and bars only competing hosted/embedded offerings, converting to MPL 2.0 four years after each version's publication ([license FAQ](https://www.hashicorp.com/en/license-faq)).

> **Correction — the "BUSL scare" does not block a homelab.** A single operator running Vault to manage their own fleet is plainly internal, non-competitive use and is expressly permitted; "the license forbids us" is **not** a valid reason to reject Vault. The real reasons to prefer the alternative are license *hygiene and future-proofing*: the competitive-offering clause is vague and could bite if the suite is ever shipped to third parties; governance now sits with a single commercial steward (IBM) that has already relicensed once; and newer functionality lands in BUSL Enterprise, not the community build.

OpenBao is the community fork from **Vault 1.14.0, the last MPL-2.0 release before the BUSL change**, now an **OpenSSF Sandbox project under the Linux Foundation** (joined June 2025), MPL-2.0 with no competitive-use clause and no change date; current release **OpenBao 2.5.5 (2026-06-17)** on a monthly cadence ([openbao.org](https://openbao.org/), [GitHub releases](https://github.com/openbao/openbao/releases)). For the ~80% of functionality this app touches, OpenBao and Vault are functionally interchangeable (shared codebase), and OpenBao ships some things Enterprise-gates in Vault: **namespaces, PKCS#11/HSM auto-unseal, the Transform engine, and standby-node read scalability** ([2.5.x notes](https://openbao.org/community/release-notes/2-5-0/); [PKCS#11 seal docs](https://openbao.org/docs/configuration/seal/pkcs11/); parity cross-checked against a [secondary comparison](https://jorijn.com/en/blog/hashicorp-vault-vs-openbao/) flagged for build-time re-verification). The gaps versus Vault Enterprise — cross-cluster replication, Sentinel, FIPS 140-3 builds, HCP — are things a single-operator homelab does not need.

> **Correction — do not lean on the OSS namespaces/PKCS#11 "bonus" (adversarial adjustment).** These are real but *not load-bearing* for a homelab that needs neither multi-tenancy nor an HSM, and the "at parity" framing overstates them: OpenBao's namespaces are only "API-compatible with upstream," its PKCS#11 seal docs explicitly disclaim "no compatibility is expressed or implied" with Vault Enterprise and require externally-created key material, and the standalone **HSM distribution is being discontinued by v2.7.0** in favor of a plugin. Default to Shamir or on-prem Transit auto-unseal (§8), treat namespaces/HSM as out of scope, and never assume wire/storage compatibility with Vault Enterprise. ([PKCS#11 seal docs](https://openbao.org/docs/configuration/seal/pkcs11/), [2.6.x notes](https://openbao.org/community/release-notes/2-6-0/))

> **Load-bearing — adopting an engine does not discharge our security stage.** The property that only the Gateway identity can redeem a handle, and agents structurally never can, is enforced by **our** wrapper — the Vault app's API/MCP surface + ACL policies binding `vault:read-credential` to the Gateway's identity, with hand-off via response wrapping on the Gateway leg. The engine gives us crypto, storage, leasing, and audit plumbing for free; it does not give us the SoD proof. That proof is ours to write in Stage 5.

BUILD loses on every axis except "we control the code," which the wrapper approach already gives us without owning the crypto core. To build it yourself you must independently get all of the following right, forever: authenticated encryption-at-rest with a sound key hierarchy; the seal/unseal + secret-zero problem; a path-scoped policy engine that cannot be tricked into over-granting; tamper-evident append-only audit that survives a root-on-the-box attacker; rotation and lease/revocation lifecycle; and no-plaintext-in-logs across every code path. Each is a specialist subfield with its own CVE history, and this codebase's stated failure mode is "confident garbage" — exactly how you ship a store that *looks* encrypted but leaks.

> **Adversarial verdict — HOLDS_WITH_CAVEATS.** The core (adopt OpenBao, reject build, keep BUSL Vault as fallback) survives, and the "fork lags on security" attack backfired — OpenBao *led* the CVE-2025-54997/HCSEC-2025-14 audit-device-RCE fix. Six adjustments are load-bearing and folded into the recommendation:
> 1. **Pin the 2.5.x stable line (≥ 2.5.5); do NOT ship the 2.6.0 beta** (its namespace-sealing/workflows/KMS-plugin primitives are unstable — wrong footing for a Critical-infra secrets store). Prompt monthly patching is a Stage-5 requirement, not optional: the 2025-26 advisory stream clusters in auth methods/plugins (cert-auth renewal [CVE-2026-39388], OIDC UI XSS [CVE-2026-33758, fixed 2.5.2], AWS-auth bypass [CVE-2025-59048], LDAP MFA bypass, TOTP replay).
> 2. **Downgrade the namespaces/PKCS#11 claim** to "present in OSS but not needed" (see the §1 correction above): no Enterprise wire/storage compat, HSM distro EOL at 2.7.0.
> 3. **Fork-death contingency (state it honestly):** OpenBao is an OpenSSF *Sandbox*-tier project (lowest maturity) with a smaller ecosystem than OpenTofu. Mitigation: MPL-2.0 lets us self-fork/patch and storage is Vault-derived — **but migrating *back* to current BUSL Vault after using post-1.14-fork features is not a validated path**; do not treat the Vault fallback as a guaranteed one-click exit once OpenBao-specific features are in use.
> 4. **Minimum surface:** enable only KV v2 + the SSH secrets engine + exactly one Gateway auth method; deny `sys/audit/*` writes to non-root and leave every other engine/auth method disabled — directly shrinking the auth/plugin CVE exposure.
> 5. **Reframe the BUSL-Vault fallback** as "viable for internal homelab use but inherits IBM's future licensing discretion and loses the license-clean property" — a break-glass option, not co-equal.
> 6. **MPL-2.0 copyleft nuance:** file-level copyleft bites only if agent-corp is ever *distributed*; for a self-hosted single-operator deployment it is inert — say so rather than calling it flatly "license-clean." ([OpenSSF OpenBao](https://openssf.org/projects/openbao/), [OpenBao LICENSE](https://raw.githubusercontent.com/openbao/openbao/main/LICENSE), [HashiCorp license FAQ](https://www.hashicorp.com/en/license-faq))

**Recommendation.** **Adopt OpenBao on the 2.5.x stable line (≥ 2.5.5, NOT the 2.6.0 beta) as the storage/crypto/leasing/audit engine, and build the Vault app as a thin API/MCP wrapper over it that enforces the SoD inversion. Enable the minimum surface — KV v2 + the SSH secrets engine + one Gateway auth method — and commit to prompt monthly patching (a Stage-5 requirement given the auth/plugin CVE cadence). OpenBao removes the BUSL competitive-use ambiguity and single-vendor (IBM) stewardship risk at parity for our needs, and its team has demonstrably led, not lagged, security fixes; treat OSS namespaces/PKCS#11 as out-of-scope bonuses, not Enterprise-compatible primitives. Record the fork-maturity risk explicitly: OpenBao is OpenSSF-Sandbox tier, mitigated by MPL-2.0 self-fork rights, but migrating back to BUSL Vault post-divergence is not validated — so keep BUSL Vault only as a license-encumbered break-glass, not a guaranteed exit. BUILD is rejected outright: a bespoke store for the fleet's root credentials is the worst possible place for one operator's confident-garbage failure mode.**

## 2. Alternative secrets-store candidates vs this suite's five-axis bar

**Findings.** This suite has a sharp acceptance test most secrets managers were never built to pass. The Vault must (1) **broker**, not merely store — interpose a runtime redemption step it can gate and log; (2) enforce **gateway-only-redeem**; (3) prefer **dynamic / short-lived** host credentials; (4) produce a **complete, immutable redemption audit**; and (5) do all of this **self-hosted for a single operator** with no per-identity paywall on those features. The lighter candidates are judged strictly against those five axes against the OpenBao baseline.

**Infisical** is the strongest lightweight contender: MIT core, self-hosts on Postgres + Redis + Docker, machine identities as first-class principals authenticating via Universal Auth or standards-based OIDC with RBAC, and genuine Vault-style dynamic secrets plus short-lived signed **SSH certificates** — exactly the shape a fleet-patching Gateway wants ([github.com/infisical/infisical](https://github.com/infisical/infisical), [dynamic-secrets docs](https://infisical.com/docs/documentation/platform/dynamic-secrets/overview)).

> **Correction — "Infisical is MIT" does not mean the features we need are free.** The MIT grant excludes the `ee/` directory, which holds premium features requiring a paid license even self-hosted. Multiple sources indicate **dynamic secrets** and rotation fall under the paid Enterprise tier for self-hosted deployments, priced per *identity* (~$18/identity/mo, humans and machines both counting). So the exact capabilities that would let Infisical beat OpenBao — dynamic host creds, SSH-cert issuance, likely the full audit log — are the ones behind the license wall. Verify the exact free-vs-EE split at build; it is the whole decision.

**Bitwarden Secrets Manager** and **Vaultwarden** do not fit: Bitwarden SM offers machine accounts but self-hosting SM requires a paid Enterprise org, runs on MSSQL, and has **no dynamic-secret capability** ([bitwarden.com](https://bitwarden.com/products/secrets-manager/), [self-hosting announcement](https://bitwarden.com/blog/enterprise-self-hosting-for-bitwarden-secrets-manager/)); Vaultwarden's maintainer states SM's API is source-available-only, so RBAC/machine accounts are out of scope — eliminated outright ([vaultwarden #3368](https://github.com/dani-garcia/vaultwarden/discussions/3368)). **Doppler** launched On-prem on 2026-06-08 but gated to Enterprise orgs, covering only "core secrets management" with no claimed on-prem dynamic secrets — wrong shape for a solo local suite ([PR Newswire](https://www.prnewswire.com/news-releases/doppler-brings-secrets-management-on-prem-for-full-control-302794005.html)). **CyberArk Conjur OSS** has the best *authorization model* — YAML **policy-as-code** with `host`/`layer` identities and deny-by-default `grant`/`permit`, a natural fit for "only the Gateway may redeem" — but it is heavy (Ruby/Postgres), its dynamic-secret generation is understood to be Enterprise-tier (verify), and its roadmap is uncertain after CyberArk's acquisition into Palo Alto Networks in early 2026 ([conjur.org RBAC](https://www.conjur.org/solutions/rbac/), [conjur-oss-suite-release](https://github.com/cyberark/conjur-oss-suite-release)). **Keywhiz is dead** — Square's README recommends Vault and archived the repo 2023-11-22 ([github.com/square/keywhiz](https://github.com/square/keywhiz)).

> **Load-bearing — SOPS+age cannot be the Vault.** SOPS has no server and no runtime broker: decryption capability *is* possession of the age key, access control is resolved entirely client-side, and **no central system records that a redemption happened**. That structurally violates the no-plaintext-to-agents inversion, gateway-only-redeem, and immutable-access-audit requirements. SOPS+age is excellent for **bootstrapping** — e.g. git-committing the encrypted unseal seed or root config, consistent with "files are the source of truth" — but it cannot enforce gateway-only-redeem or produce a redemption audit ([github.com/getsops/sops](https://github.com/getsops/sops), [getsops.io](https://getsops.io/)).

**Recommendation.** **Stay with OpenBao/Vault as the engine; none of the lighter alternatives clears the five-axis bar. Rank the near-misses: (1) Infisical — adopt-worthy in shape, but only if the build-time check shows dynamic secrets + SSH-cert issuance are usable self-hosted without a per-identity EE license; if paywalled, it loses on cost/independence, not capability. (2) Conjur OSS — keep as the reference for the authorization model (policy-as-code host/layer grant/permit) even if we do not run it. Reject outright: Doppler, Bitwarden SM + Vaultwarden, and Keywhiz. Use SOPS+age only as a bootstrap/seed-encryption sidecar committed to git — never as the Vault itself.**

## 3. The gateway-only-redeem enforcement model

**Findings.** This is the Vault's central security property: the Gateway's machine identity MAY redeem a credential handle to plaintext; every agent identity MUST NOT, by any path. The enforcement point is a **path-scoped ACL on a deny-by-default backend, authenticated per-caller** — and, critically, response wrapping is *not* the access-control primitive.

OpenBao and Vault are deny-by-default: an empty policy grants nothing, and access requires a policy explicitly attaching a **capability** (`create`/`read`/`update`/`patch`/`delete`/`list`, plus `sudo`/`deny`) to a **path**. The load-bearing rule is precedence: **`deny` always wins**, "regardless of any other defined capabilities, including `sudo`," and the single most specific matching rule applies rather than accumulating up the tree ([Vault policies](https://developer.hashicorp.com/vault/docs/concepts/policies), [policies tutorial](https://developer.hashicorp.com/vault/tutorials/policies/policies), [OpenBao policies](https://openbao.org/docs/concepts/policies/)). Identity is modeled as **entities** with **aliases** and **groups**; templated policies can inject `identity.entity.id` into paths to scope each principal to its own subtree ([policy templating](https://developer.hashicorp.com/vault/tutorials/policies/policy-templating)). Writing the gateway-only redeem policy is therefore direct: bind the Gateway entity to a policy granting `read` on `secret/data/hosts/+/root`, grant agents nothing (deny-by-default already denies them), and add a belt-and-suspenders explicit `deny` on the credential subtree to the shared agent policy so no future accidental wildcard grant can re-open it.

The Gateway authenticates to redeem via **JWT/OIDC auth** (present a signed JWT minted by the suite's `auth` service; Vault validates the signature against the issuer's JWKS and a role maps the token to policies via `bound_audiences`/`bound_subject`/`bound_claims`), which is the cleaner fit — the same `auth` identity that mints the Gateway's scope is the thing Vault trusts, no second secret ([Vault JWT/OIDC](https://developer.hashicorp.com/vault/docs/auth/jwt)). **AppRole** (a `role_id`+`secret_id` pair) is the fallback when no workload-identity issuer exists, but reintroduces a secret-zero ([Vault AppRole](https://developer.hashicorp.com/vault/docs/auth/approle)).

> **Load-bearing correction — response wrapping does NOT enforce "only the Gateway may redeem."** A wrapping token is a **bearer** credential: the docs are explicit that "this endpoint can be used by using a wrapping token as the client token" — possession alone suffices, with no identity binding ([sys/wrapping/unwrap](https://developer.hashicorp.com/vault/api-docs/system/wrapping-unwrap)). You cannot hand an agent a wrapping token as its "handle." The correct decomposition: the agent's handle is a **powerless application-level reference** (e.g. `cred://hosts/nas-01/root`); the ACL on the underlying path is what forbids the agent; response wrapping is used only on the **Gateway-facing leg** for in-flight reference + tamper-evidence.

The near-perfect native SoD primitives — **Control Groups** (N named authorizers must approve before release) and **Sentinel** policy-as-code — are **Vault Enterprise / HCP only** and **absent from OpenBao**; OpenBao's only extra-policy logic is narrow CEL validation in specific plugins ([Control Groups](https://developer.hashicorp.com/vault/docs/enterprise/control-groups), [Sentinel](https://developer.hashicorp.com/vault/docs/enterprise/sentinel), [OpenBao #783](https://github.com/orgs/openbao/discussions/783)). On a self-hosted open-source stack, enforce the four-holder SoD agreement *in the Vault app's own redeem endpoint* and let the ACL layer be the hard floor on who can read the path at all.

> **Adversarial verdict — HOLDS_WITH_CAVEATS, and six conditions are load-bearing.** The core mechanism (path-scoped ACL + explicit `deny`; agent handle = powerless reference; wrapping only on the Gateway leg) is correct, but "no agent path reaches plaintext" holds *only* with all of the following, each of which Stage 5 must prove:
> 1. **Bind the redeem role on a Gateway-*unique* claim, never on `aud` alone.** `bound_audiences` is mandatory for JWT roles carrying an `aud` since Vault 1.17 and is checked even alongside `bound_claims`, but an audience is a *resource* id, not a *principal* id — if `auth` mints every machine token with `aud=vault`, an agent JWT satisfies audience-only binding, selects the redeem role, and inherits the redeem policy → plaintext. The role MUST additionally pin `bound_subject`/a Gateway-unique `bound_claims` + per-identity `user_claim`. If agents and the Gateway share one JWT mount (the natural single-issuer design), this binding is the only thing standing between an agent token and the redeem policy. This is blocked on `auth`'s final token shape — carry it in verify-at-build. ([v1.17+ bound_audiences](https://support.hashicorp.com/hc/en-us/articles/36321323028755-Vault-v1-17-x-and-above-JWT-auth-login-requires-bound-audiences-on-the-role), [vault#25791](https://github.com/hashicorp/vault/issues/25791))
> 2. **Identity is not unforgeable in the field — deny agents any write to `identity/*` and patch the backend.** CVE-2025-6037 (non-CA TLS-cert auth checked only the pinned public key, not the CN → CN impersonation to inherit another entity's policies), CVE-2025-5999 (root-namespace identity-entity access could self-assign the `root` policy via `" root"`/`"ROOT"` normalization — root bypasses deny-by-default), and HCSEC-2021-27 (entity-alias merge acquires another user's policies) all hit this SoD boundary. Deny every agent write to `identity/*` and auth-role config, key templated policies on immutable `identity.entity.id` (never attacker-settable alias metadata/names), and pin to the fixed line — **Vault 1.20.2 / 1.19.8 / 1.18.13 / 1.16.24** or the equivalent OpenBao release. ([Cyata](https://cyata.ai/blog/cracking-the-vault-how-we-found-zero-day-flaws-in-authentication-identity-and-authorization-in-hashicorp-vault/), [HCSEC-2025-22](https://support.hashicorp.com/hc/en-us/articles/43642320190611-HCSEC-2025-22-Multiple-Vulnerabilities-Impacting-HashiCorp-Vault-and-Vault-Enterprise), [HCSEC-2021-27](https://discuss.hashicorp.com/t/hcsec-2021-27-vault-merging-multiple-entity-aliases-for-the-same-mount-may-allow-privilege-escalation/30420))
> 3. **Strip the `default` policy from agent tokens.** The built-in `default` policy (on every token unless `-no-default-policy`) grants `update` on `sys/wrapping/unwrap` and full `cubbyhole/*`, so every agent *already has the capability* to unwrap; only non-possession stops it. One wrapping token leaked into a Board ticket, a markdown note, a log line, or an error payload is redeemable by any agent — and those surfaces are agent-readable by design. Issue agent tokens with `no_default_policy=true` and forbid any wrapping token from entering Board/Notes/Chat/logs. ([policy_store.go default policy](https://raw.githubusercontent.com/hashicorp/vault/main/vault/policy_store.go))
> 4. **Drop `secret_id_bound_cidrs` as an identity boundary.** CIDR binds by source IP, but Docker Compose containers share one subnet and the Caddy proxy rewrites the source IP, so a CIDR admitting the Gateway admits every agent container too. Rely on out-of-band possession of the Gateway `role_id`+`secret_id`. ([vault#12334](https://github.com/hashicorp/vault/issues/12334))
> 5. **The ACL enforces gateway-only-*read*, not the four-holder SoD.** The Gateway's policy grants direct `read`; the Board-approval + CMDB-in-window checks live only in the Vault app's redeem endpoint. Close the Gateway's direct-to-backend path (or move those checks somewhere the ACL cannot sidestep) so a confused/compromised Gateway token cannot read plaintext without Board + CMDB agreement.
> 6. **Confirm no agent container can reach a root/sudo token or the seal/auto-unseal material** (root and `sudo` override deny; `sys/raw` reads storage directly). No shared volumes/tokens with the seal or a root token. ([OpenBao GHSA-vf84-mxrq-crqc](https://github.com/openbao/openbao/security/advisories/GHSA-vf84-mxrq-crqc))

**Recommendation.** **Enforce gateway-only redemption with path-scoped ACL on a deny-by-default OpenBao backend; treat the agent handle as a powerless app-level reference, never a wrapping token. Attach a `read`-on-`secret/data/hosts/+/root` policy to the Gateway entity only, grant agents nothing plus a belt-and-suspenders explicit `deny`. Authenticate the Gateway via JWT/OIDC against `auth`, binding the redeem role on a Gateway-UNIQUE `sub`/`groups`/`bound_claims` (never `aud` alone). Do NOT phrase the property as an unconditional "no agent path reaches plaintext": it HOLDS only with all six conditions above — Gateway-unique claim binding, agent denial of `identity/*` writes + patched backend, `no_default_policy` agent tokens, no CIDR-as-identity, no Gateway direct-to-backend SoD bypass, and no agent reach to root/seal material. Do not depend on Sentinel or Control Groups (Enterprise-only, absent from OpenBao); enforce the four-holder agreement in the Vault app's redeem endpoint. Back it with a standing regression test: agent token → redeem → 403.**

## 4. Short-lived / dynamic credentials + the SSH secrets engine as host-access broker

**Findings.** Vault/OpenBao draw a hard line between *static* KV secrets (handed back verbatim, nothing expires, leaked-until-a-human-rotates) and *dynamic* secrets, which are generated per-consumer at read time and born with a **lease**: "all dynamic secrets in Vault are required to have a lease," TTL-bounded, auto-revoked on expiry, cascade-revoked when the parent token is revoked, and force-revocable per-lease or by prefix ([Vault leases](https://developer.hashicorp.com/vault/docs/concepts/lease)). This is what makes a stolen credential a *time-boxed* liability and gives the operator a "revoke everything the Gateway holds now" button.

The killer pattern for ~20 hosts is to **not store a root password at all**. The SSH secrets engine has two current modes ([Vault SSH engine](https://developer.hashicorp.com/vault/docs/secrets/ssh), [OpenBao SSH engine](https://openbao.org/docs/secrets/ssh/)): **signed SSH certificates (CA mode, recommended)** — Vault holds an SSH CA signing key, each host trusts it once via `TrustedUserCAKeys`, and at access time the Gateway submits an ephemeral public key to `sign/<role>` and gets back a short-lived cert scoped by `valid_principals`, `key_id` (stamped into the host's auth log for audit), and role `ttl`/`max_ttl`; no secret is stored on the host and the host never phones home ([OpenBao signed certs](https://openbao.org/docs/secrets/ssh/signed-ssh-certificates/)) — and **one-time passwords (OTP)**, which revoke-on-use but require the `vault-ssh-helper` PAM module on every host ([SSH-OTP tutorial](https://developer.hashicorp.com/vault/tutorials/secrets-management/ssh-otp)). The old **dynamic-keys** mode "has been removed in Vault 1.13" and never existed in OpenBao — do not use it.

> **Correction — "dynamic keys" is a trap term.** The intuitive-sounding "dynamic SSH keys" mode is the one that is gone. "Dynamic/brokered host access" here means **SSH-CA signed certificates**, technically a *signed* credential rather than a leased dynamic secret, but with the same short-lived auto-expiring property.

> **Adversarial verdict — HOLDS_WITH_CAVEATS; the SSH-broker direction is right but several caveats are load-bearing.**
> - **CVE-2024-7594 (the big one).** Until fixed, the SSH engine did not require `valid_principals` to be non-empty: a Gateway role created the obvious way (`allowed_users`, `key_type=ca`, a TTL, nothing else) mints a cert usable as **any user, including root, on every host trusting the CA** — NVD 8.8 High, affecting Vault CE 1.7.7–1.17.6 and OpenBao 0.1.0–2.0.2. **Note the researcher's dynamic-keys-removed pin (1.13) sits squarely inside this window.** Pin **OpenBao ≥ 2.0.2 / Vault ≥ 1.17.6**, set `allow_empty_principals=false`, and make it a build-failing invariant that every Gateway signing role carries a non-empty, ticket-templated `valid_principals` and never `default_user=root`/wildcard. ([HCSEC-2024-20](https://discuss.hashicorp.com/t/hcsec-2024-20-vault-ssh-secrets-engine-configuration-did-not-restrict-valid-principals-by-default/70251), [NVD CVE-2024-7594](https://nvd.nist.gov/vuln/detail/cve-2024-7594), [GHSA-jg74-mwgw-v6x3](https://github.com/advisories/GHSA-jg74-mwgw-v6x3))
> - **The kill switch cannot revoke an issued cert.** A signed cert is validated cryptographically by the host, which never re-contacts Vault; sealing Vault, revoking the Gateway token, or hitting the kill switch does **not** stop an in-flight session — only TTL expiry or an OpenSSH KRL pushed to all ~20 hosts does, and pushing a KRL during an incident itself needs host access. So "auto-expires" is true only *eventually*, and the mandated instant-halt property is **not** satisfied by SSH-CA alone. Size cert TTL to a single run (5–15 min) as the primary control and stand up a KRL/CA-rotation runbook. ([OpenSSH PROTOCOL.krl](https://github.com/openssh/openssh-portable/blob/master/PROTOCOL.krl))
> - **Clock skew silently extends the cert.** OpenSSH checks `ValidBefore` against the *host's* wall clock; a homelab box 20 min slow (no/broken NTP — common on appliances) honors a "15-minute" cert for ~35 real minutes. Require enforced/monitored NTP on all target hosts as a hard precondition. ([vault#3504](https://github.com/hashicorp/vault/issues/3504))
> - **Response-wrapping a static secret does not keep plaintext from an agent in the path.** Wrapping is single-use malfeasance *detection*, not prevention — whoever holds the token unwraps to plaintext. If the wrap token transits an agent, the agent simply unwraps to root plaintext. Wrap tokens travel Vault→Gateway only.
> - **Availability SPOF + crown-jewel shift.** A sealed Vault means zero host access for the continuous agents, with no offline break-glass because the CA signing key lives inside the sealed barrier and is non-exportable after generation ([Vault SSH API](https://developer.hashicorp.com/vault/api-docs/secret/ssh)). And the CA signing key — now the sole thing that matters — has **no lease and no TTL**; its compromise is fleet-wide and recovery requires CA rotation plus re-trusting `TrustedUserCAKeys` on all 20 hosts. This is a defensible trade but is emphatically **not** "the leaked thing auto-expires." Protect the CA key as a second barrier-grade crown jewel and provide an explicit out-of-container break-glass for host access.

Some secrets are irreducibly static (a NAS admin login, a switch enable password, third-party API keys on systems you cannot reconfigure). These live in KV with **no lease-based expiry**; soften with scheduled/post-redemption **rotation** plus short-TTL response-wrapping on the Gateway leg so even static plaintext is time-boxed in transit. Dynamic engines also raise the seal stakes: a sealed Vault "cannot decrypt its data… dynamic secrets engines cannot function until the barrier is unsealed," and auto-unseal recovery keys "cannot decrypt the root key" ([Vault seal](https://developer.hashicorp.com/vault/docs/concepts/seal)).

**Recommendation.** **Adopt "short-lived by default, static only where forced." Make SSH-CA signed certificates the canonical broker for all ~20 hosts' shell/root access (Vault holds a CA signing key, never a stored root password; the Gateway alone can call `sign/<role>` and mints ephemeral 5–15 min certs scoped by `valid_principals` and stamped `key_id=<ticket-id>`). Pin OpenBao ≥ 2.0.2 / Vault ≥ 1.17.6, set `allow_empty_principals=false`, and enforce non-empty ticket-templated `valid_principals` with no root/wildcard user as a build-failing invariant. Stop asserting the kill switch halts SSH-CA host access: TTL is the primary control, enforced NTP is a hard precondition, and a KRL/CA-rotation runbook covers true revoke; reserve OTP for the few hosts needing revoke-on-use. Never route a wrap token or redeemable handle through an agent. For irreducible static secrets, KV + scheduled rotation + short-TTL Gateway-leg wrapping. Treat the SSH CA signing key and the Shamir unseal-key set as the two crown jewels: keep unseal keys outside the container, provide an out-of-container host-access break-glass for a sealed Vault, and plan CA rotation via the SSH engine's multi-issuer support.**

## 5. Integration with the `auth` OIDC AS — who may request/redeem what, tied to SoD scopes

**Findings.** The Vault does not need a bespoke OIDC client: both Vault and OpenBao ship a **JWT/OIDC auth method** whose *JWT mode* (`role_type: "jwt"`) is exactly an OAuth resource-server primitive — it accepts a caller's already-minted JWT, validates it locally (no browser/redirect leg), and exchanges it for a short-lived internal token whose policies are decided by claim bindings ([OpenBao JWT/OIDC docs](https://openbao.org/docs/auth/jwt/)). Signatures are verified against `jwt_validation_pubkeys`, `jwks_url`, or `oidc_discovery_url`, with `bound_issuer` pinning `iss` to the `auth` AS — so Vault points at `auth`'s JWKS and validates tokens **offline**, the independent-resource-server posture the suite-wide "each app validates `aud`=self" rule wants ([OpenBao JWT/OIDC API](https://openbao.org/api-docs/auth/jwt/)). On a `jwt` role, `bound_audiences`/`bound_subject`/`bound_claims` gate acceptance, `user_claim` becomes the entity-alias name, and `token_policies` is the grant. The Gateway presents its `auth` token to `POST /auth/jwt/login` with `role=gateway-redeemer`; Vault checks `iss=auth`, `aud` includes `vault`, and a distinguishing Gateway claim, then issues a short-lived token bearing exactly the `redeem` policy. Agents never get this role: their `auth` roles carry zero credential scope, so their tokens fail the binding or match no redeem-capable role ([Vault JWT/OIDC API](https://developer.hashicorp.com/vault/api-docs/auth/jwt)).

> **Trust boundary — Vault must validate the JWT itself, not trust a forwarded header.** The proxy's `forward_auth` may gate the Vault UI at the edge, but the redeem path must not rest on a proxy-injected `X-User` header; Vault's JWT method structurally enforces this (the caller POSTs the raw JWT and Vault verifies the signature against `auth`'s JWKS). Given the 2026 forward-auth identity-injection CVE cluster documented in the `auth` research (Caddy CVE-2026-30851; Traefik `X-Forwarded-*` bypasses), relying on a forwarded header for the highest-value secret would be the worst place to do so.

> **Load-bearing — audience-binding alone does NOT enforce SoD; the grant does.** RFC 8707 audience restriction only stops a *Board*-audienced token from being replayed at Vault; it does not stop a dual-scoped identity minting a second Vault-audienced token. The real control is twofold and enforced at **both** ends: (a) `auth` never grants agents the `vault:read-credential` scope/role (issuance-time SSD), and (b) Vault's redeem role sets `bound_claims`/`bound_subject` so only the Gateway's identifying claim maps to the `redeem` policy. Phrase the property as "only the Gateway principal is grantable the redeem policy, verified by claim binding at the Vault role" — not "we audience-bind to Vault."

> **Gotcha — `bound_audiences` is *required* for `jwt` roles carrying an `aud`,** and is still evaluated even when `bound_claims` is set, so you cannot skip the audience match by expressing `aud` inside `bound_claims` ([vault#27343](https://github.com/hashicorp/vault/issues/27343)). Bind the audience explicitly to `vault` on the redeem role. Prefer the direct role→`token_policies` binding over external-group aliases for the security-critical grant (external groups need a pre-created `type=external` group + alias whose `canonical_id`=group id and mount accessor=JWT mount, or membership silently never attaches — [vault#6494](https://github.com/hashicorp/vault/issues/6494)); reserve `groups_claim` for human UI roles.

The four-holder agreement is **not** something stock Vault can evaluate — Vault has no knowledge of ticket approval state or maintenance windows. Its job is deliberately narrow: *given this `auth` token, is this the Gateway, and does its role grant `read` on this path?* Board-approval and CMDB-policy checks are enforced **upstream at the Gateway**; Vault is the fourth independent lock, not the arbiter of the other three. Two levers tighten this if wanted: **templated ACL policies** scoping `read` to the specific credential named in the token, and **RFC 9396 Rich Authorization Requests** — having `auth` mint the redeem token with an `authorization_details` object naming the credential path *and* the approved ticket id, so no approval → no redeemable token. Lever (2) is the cleanest way to make "no approval → no redeem" structural but depends on `auth`'s final model.

> **Depends on `auth`'s locked model — flag before Planning.** (i) Whether `auth` emits RFC 9068 `at+jwt` tokens Vault can validate via JWKS, and the exact claim carrying the SoD holder-scope (`scope` string vs `roles`/`groups` array) — the Vault role's `bound_claims` shape is downstream of that. (ii) Whether `auth` supports RFC 8707 resource indicators natively or (Keycloak-style) needs audience-mapper plumbing (the `auth` research flags Keycloak has no native RFC 8707 in v26). (iii) Whether `auth` can implement RAR per-ticket `authorization_details`, or ticket-approval enforcement stays entirely at the Gateway.

**Recommendation.** **Integrate the Vault as an independent OAuth resource server via the JWT/OIDC method in JWT mode, pointed at `auth`'s JWKS with `bound_issuer`=auth. Define exactly one redeem role binding `aud`=vault AND a distinguishing Gateway claim (`bound_subject`=the Gateway service account, or `bound_claims` requiring the `vault:read-credential` holder-scope), granting a single narrow `redeem` `token_policies` scoped (ideally templated) to the credential paths. Enforce SoD at BOTH ends — never grant agents the scope at `auth`, require the Gateway claim at the Vault role — because audience-binding alone is insufficient. Never front the redeem path with proxy header trust. Keep the four-holder check split: Board + CMDB gates live at the Gateway; Vault is the fourth independent lock answering only "is this the Gateway, for this credential." Prefer direct role→`token_policies` over external-group aliases. Where `auth` allows, bind the redeem token to the approved ticket via RFC 9396 RAR — but treat that as auth-dependent and keep Gateway-side enforcement as the baseline.**

## 6. Access audit — complete, immutable, no-plaintext-in-logs

**Findings.** The Vault's access-audit requirement maps almost exactly onto Vault/OpenBao's audit-device model, so that is the reference design. Audit devices "record all API requests and responses in detail" (except a few unauthenticated health/metrics endpoints), and the no-plaintext behavior is built in: by default a salted **keyed hash (HMAC-SHA256)** is applied to string values in both requests and responses "so that secrets aren't in plaintext within your audit logs," leaving the value visible only as `hmac-sha256:<hash>` ([Vault audit devices](https://developer.hashicorp.com/vault/docs/audit), [OpenBao audit devices](https://openbao.org/docs/audit/), [query audit logs](https://developer.hashicorp.com/vault/tutorials/monitoring/query-audit-device-logs)). Three device types exist — **file**, **syslog**, **socket** — and multiple can run at once ([file audit device](https://developer.hashicorp.com/vault/docs/audit/file)).

> **Load-bearing — the `log_raw` footgun.** Both projects expose `log_raw` (default `false`): "if enabled, logs the security sensitive information without hashing, in the raw format" — writing plaintext root credentials straight into the audit log, the exact failure the Vault exists to prevent. The rule is absolute: `log_raw` is never enabled anywhere, and `hmac_accessor` stays at its default `true`. ([/sys/audit API](https://developer.hashicorp.com/vault/api-docs/system/audit))

> **Gotcha — only strings are hashed.** Vault "does not hash non-string values, such as integers and booleans." Store and log every credential value as a string so the HMAC path always covers it.

The most important property is subtle. Vault "guarantees that it saves to at least one of the enabled devices"; if audit devices are enabled and Vault "cannot log information to at least one of the enabled devices, Vault refuses to service the corresponding API request." So the success threshold is **at least one** enabled device accepting the write — not all — but if *every* enabled device is blocked, Vault stops serving, making auditability a hard precondition for redemption.

> **Correction / availability tradeoff — a single audit device is a self-inflicted DoS.** Because a fully-unavailable sink halts Vault, HashiCorp's explicit guidance is to "enable at least two audit devices, of different types." Do not run one device; the fail-closed guarantee only helps if a second device can absorb the write. ([audit best practices](https://developer.hashicorp.com/vault/docs/audit/best-practices))

A complete redemption record must carry: **redeemer identity** (the Gateway machine identity, from `auth.display_name`/entity/`accessor` — the only holder of `vault:read-credential`), **handle** (`request.path`), **time**, **request id** (for correlation with the Gateway's audit and the Board ticket), **wrap/lease id**, and **outcome** (success vs deny). A permission-denied redemption attempt by an agent is itself a first-class audit event and must be captured — "an agent tried to redeem" is exactly the exfiltration signal this app exists to catch.

> **Correction — Vault does NOT cryptographically tamper-proof its audit log.** Nothing in the audit-device docs signs or hash-chains the file; append-only text on the Vault host is trivially editable by whoever has root there (and the container is the top-value target). "Immutable" must be achieved *externally*: forward at least one device off-box (TCP syslog — UDP silently drops oversized entries — or a local Unix socket), land it in an append-only/WORM store or SIEM the Vault host cannot rewrite, and optionally hash-chain records so any edit is detectable. These are our responsibility, not Vault features.

**Recommendation.** **Adopt the audit-device model wholesale with three non-negotiables. (1) No plaintext, ever: HMAC-SHA256 (salted) every secret value, store credential values as strings, forbid `log_raw` entirely. (2) Fail closed on auditability: a redemption succeeds only if the record is durably written — but run ≥2 sinks of different types (one local file for latency, one off-box) so a single sink outage cannot DoS the Vault. (3) Immutability is external: ship at least one copy off-box over TCP syslog/socket into an append-only/WORM store or SIEM the Vault cannot rewrite, and optionally hash-chain records. Each record carries redeemer identity, handle/path, timestamp, request id, wrap/lease id, and success/deny — and denied agent redemption attempts are logged as first-class exfiltration signals.**

## 7. The two-view inversion: near-empty agent MCP surface + rich human UI

**Findings.** The Vault deliberately inverts "two views over one state": the human UI is the rich surface (manage, rotate, audit), the agent MCP surface is near-empty by design. The organizing invariant is blunt — **no MCP tool on this server may ever emit a plaintext credential on any path, to any caller, including a fully-compromised agent.**

The MCP authorization model supplies the enforcement substrate: each MCP server "acts as an OAuth 2.1 resource server" that MUST validate that access tokens "were issued specifically for them as the intended audience, according to [RFC 8707]," and MUST implement RFC 9728 Protected Resource Metadata ([MCP 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization)). When a caller presents a token lacking the required scope, the server responds `403 Forbidden` with `WWW-Authenticate: Bearer error="insufficient_scope"` ([RFC 6750 §3.1](https://datatracker.ietf.org/doc/html/rfc6750#section-3.1)). That is the lever that keeps the surface near-empty: an agent's token has `aud=vault` but its scope set structurally never contains `vault:read-credential`, so any redemption path is a hard `403` for the agent and a `200` only for the Gateway. The safe agent tool list is short and side-effect-poor:

- **`vault.list_handles(ticket_id)`** — handles + non-secret metadata for the credentials a Board ticket legitimately needs; read-only, references never values.
- **`vault.describe_handle(handle)`** — non-secret metadata for one handle; read-only.
- **`vault.request_release(ticket_id, handle)`** — the one tool with a side effect; *stages* a release bound to the Gateway audience for that ticket and returns an opaque `release_id` (a reference the agent writes onto the Board ticket). It does **not** return plaintext and **not** a redeemable wrapping token.
- **`vault.release_status(release_id)`** — status only (`pending`/`redeemed`/`expired`/`revoked`).

Read-only-ness cannot live in a tool *annotation*: the spec states clients "MUST consider tool annotations to be untrusted unless they come from trusted servers," so `readOnlyHint` is decoration, never a control — enforcement is the RS scope check on every call ([MCP tools](https://modelcontextprotocol.io/docs/concepts/tools)).

> **Load-bearing — what must NOT exist on the agent surface, at all:** no `read_secret`/`get_secret`/`reveal`/`export`/`unwrap`/`set_secret`-that-echoes/`rotate`-that-returns-the-value, and no debug/admin method whose response body can contain a credential field. The near-empty surface is a *whitelist*: the four tools above are the entire agent-reachable set; every other operation is reachable only by the human UI or the Gateway identity. "We just won't document the read tool" is insufficient — it must not be registered on the MCP server the agent's audience can reach.

> **Correction — response wrapping belongs on the Gateway leg, never the agent leg.** The single-unwrap property cuts both ways: whoever holds the wrapping token can unwrap it. If the Vault handed the agent a wrapping token to "carry to the Gateway," the agent would hold a plaintext-equivalent and the inversion collapses. The agent surface returns only an opaque `release_id` that is not a redeemable cubbyhole token; wrapping (if used) is minted for and delivered to the Gateway out-of-band ([Vault response wrapping](https://developer.hashicorp.com/vault/docs/concepts/response-wrapping), [OpenBao response wrapping](https://openbao.org/docs/concepts/response-wrapping/)).

Redemption flows: the agent calls `request_release`, gets a `release_id`, writes it onto the ticket (its job is now done — it moved a reference, not a secret); the Gateway independently authenticates with its own audience-bound token, presents `(ticket_id, release_id)`, and the Vault **re-verifies preconditions itself** (Board approved, CMDB permits now) before releasing plaintext to the Gateway only, writing every redemption to the immutable access audit. The rich surface — create/import/version/rotate secrets, no-plaintext-in-logs confirmation, and the complete immutable access audit — lives entirely on the operator's authenticated session, because the operator, not the agent, is the trusted principal here.

The Stage-5 exit criterion is a set of **invariant tests** proving no MCP path returns plaintext: static surface enumeration (exactly four whitelisted tools; fail the build if any output schema has a secret field or if `read/export/unwrap/reveal` is registered on the agent surface); a runtime scope-denial sweep (every redemption path returns `403 insufficient_scope`, never `200`-with-plaintext); a reference-not-key test (the `release_id` is not a valid wrapping token); audience isolation (a `board`-audienced token is rejected); gateway-only redemption (non-Gateway principal denied + audited, Gateway succeeds + recorded); a compromised-agent exfiltration drill (dump everything the agent can read → zero plaintext, zero redeemable material); and a no-plaintext-egress log scan.

> **Gotcha — tie the redemption gate to `auth`'s final scope model.** This assumes `vault:read-credential` is a scope only the Gateway is ever issued. If `auth` lands on RAR/`authorization_details` or a central PDP for the "is this the Gateway, for this ticket, now?" decision, the *gate mechanism* changes but the *invariant* does not. Flag the exact scope/claim name and the PDP-vs-scope split for build-time confirmation.

**Recommendation.** **Ship the agent MCP surface as a strict four-tool whitelist — `list_handles`, `describe_handle`, `request_release` (opaque `release_id`, never plaintext, never a redeemable wrapping token), `release_status` — and register no read/export/reveal/unwrap tool on the agent audience at all. Enforce the inversion with standard OIDC/OAuth2 RS primitives: validate `aud=self` (RFC 8707), and `vault:read-credential` is a scope only the Gateway ever holds, so every redemption path is a structural `403` for agents. Redemption flows agent → `release_id` on the ticket → Gateway authenticates independently → Vault re-verifies Board + CMDB before releasing to the Gateway only; use response wrapping on the Vault↔Gateway leg only. Put the entire rich surface on the operator UI. Gate release on the seven invariant tests, treating "no MCP path returns plaintext even to a fully compromised agent" as the exit criterion, and confirm the exact redemption scope/claim once `auth`'s model locks.**

## 8. Top-value-target hardening + the "Vault alone causes no action" SoD proof

**Findings.** The container is the single highest-value target in the suite. Hardening has two jobs: keep the barrier closed against a container compromise, and produce the written SoD proof the Stage-7 Definition of Done demands.

A Vault-class store boots **sealed**: encryption is layered three deep (data → keyring encryption keys → root key → unseal key), and the process cannot read its keyring until unsealed ([Vault seal](https://developer.hashicorp.com/vault/docs/concepts/seal)). With the default Shamir seal the unseal key is split into **5 shares, threshold 3** ([OpenBao operator init](https://openbao.org/docs/commands/operator/init/)); any restart returns to sealed and demands a fresh quorum — painful for an unattended homelab. **Auto-unseal** wraps the root key with an external seal (cloud KMS, HSM/PKCS#11, or another OpenBao's Transit engine) called on boot, replacing Shamir shares with **recovery keys** (default 5/3); OpenBao's seal catalog is broad (AWS/GCP/Azure/AliCloud/OCI KMS, KMIP, PKCS#11, Static Key, Transit) ([OpenBao Transit seal](https://openbao.org/docs/configuration/seal/transit/)).

> **Load-bearing — recovery keys are NOT unseal keys.** Recovery keys "cannot decrypt the root key, and thus are not sufficient to unseal OpenBao if the Auto Unseal mechanism isn't working. They are purely an authorization mechanism." Auto-unseal makes the external seal a hard availability dependency. ([OpenBao seal](https://openbao.org/docs/concepts/seal/))

> **Gotcha — auto-unseal relocates "secret zero," it doesn't eliminate it.** Manual Shamir keeps secret zero as 3-of-5 human-held shares (offline, no runtime dependency, but no unattended restart). Cloud KMS moves it into a cloud IAM principal, tethering the airgap-friendly homelab to an internet KMS. **Transit auto-unseal keeps secret zero on-prem** but creates a chicken-and-egg chain: the outer "unsealer" OpenBao must itself be unsealed. The **Static-Key seal is convenience-only** (wrapping key sits plaintext-adjacent in config) and must not be used for the credential-holding Vault.

The initial root token is for first-boot setup only and should be revoked afterward; a new one is minted on demand with `operator generate-root` under a quorum of unseal/recovery-key holders, XOR'd with a one-time password so it is never transmitted in the clear — the system need never hold a standing root token ([production hardening](https://developer.hashicorp.com/vault/docs/concepts/production-hardening), [generate-root](https://developer.hashicorp.com/vault/docs/commands/operator/generate-root)). The official production-hardening checklist maps one-to-one onto this threat model: dedicated **unprivileged user**; **end-to-end TLS** including to storage; **disable swap** + **mlock** (needs `overlayfs2` + `IPC_LOCK` in a container); **disable core dumps**; **single tenancy**; **firewall** to essential peers only; **audit devices on**; **no seal creds in cleartext**; short TTLs and **no glob/wildcard policies**. For this suite the firewall item is architectural: the Vault must not be internet-exposed, and only the Gateway (redeem), `auth` (JWKS), and the operator UI may reach it — every agent identity is off the network path.

Recent advisories to re-pin at build (several surfaced via secondary aggregators — re-confirm against [NVD](https://nvd.nist.gov/), [HashiCorp bulletins](https://www.hashicorp.com/en/trust/security/vulnerability-management), [OpenBao advisories](https://github.com/openbao/openbao/security/advisories)): **Vault** — CVE-2026-5052 (PKI/ACME SSRF), CVE-2026-3605 (authenticated DoS via policy glob wildcards — reinforces "no wildcard policies"), CVE-2025-13357/6004/6003 (LDAP auth-bypass / lockout-bypass / MFA-bypass); **OpenBao** — GHSA-xp75-r577-cvhp and GHSA-vf84-mxrq-crqc (privileged operator may execute code / elevate token privileges — treat operator access as blast-radius), CVE-2026-40264/39388/39396. Also fold in the identity/SSH CVEs from §3–§4: CVE-2025-5999/6037, HCSEC-2021-27, and CVE-2024-7594 — pin to the fixed lines (Vault **1.20.2/1.19.8/1.18.13/1.16.24** or equivalent OpenBao, and OpenBao **≥ 2.0.2** for the SSH principals fix).

The Stage-7 SoD proof has two limbs. **(a) "Vault alone causes no real-world action"** — even a fully compromised Vault that hands out every plaintext secret cannot patch or reboot a host, because the Vault exposes **no execution surface**; a destructive action requires the Gateway to independently verify an approved Board ticket + CMDB in-window policy before it redeems and executes. The proof enumerates the Vault's entire API and shows none of it executes on a host or approves/policies anything. **(b) "Agent-only paths never reach plaintext"** — enumerate every plaintext-egress path and show each is closed to agents: the **redeem API** (gated on the Gateway-only `vault:read-credential` scope; agent token → 403, always), the **agent read surface** (handles only, no plaintext by any code path including error/echo), **storage/backup at rest** (closed by the barrier — ciphertext readable only through an unsealed process), **swap/core/logs** (mlock + no-swap, `LimitCORE=0`, no-plaintext-in-logs), and the **privileged operator/root token** (break-glass, not an agent path; no standing root token; operator access out-of-band). Every redemption is written to the immutable access audit so limb (b) is evidenced, not just asserted.

> **Auth-coupling caveat (verify at build).** Both limbs lean on `auth` binding the redemption scope to exactly one identity. Tie the proof to standard OIDC/OAuth2 primitives: the Gateway authenticates as a distinct machine principal (client-credentials), its token is audience-bound (RFC 8707) to the Vault and carries the redemption scope, and the Vault validates `aud=self` + scope against `auth`'s JWKS. If `auth` later adds a central PDP, route the "caller == Gateway" check there. Flag the exact scope name and token shape as auth-dependent.

**Recommendation.** **Wrap a hardened, patched OpenBao and treat the container as crown jewels. Keep no standing root token: revoke the initial one and mint per-use tokens via `operator generate-root` under quorum. For a homelab that must not depend on a cloud KMS, prefer Transit auto-unseal against a second minimal "unsealer" OpenBao so secret zero stays on-prem; escrow the 3-of-5 recovery shares offline with separate holders; never use the Static-Key seal for the credential store. Apply the full production-hardening checklist — unprivileged user, end-to-end TLS, mlock + swap disabled, core dumps disabled, single-tenant, audit devices on, no seal creds in cleartext, no wildcard policies — and enforce network isolation so only the Gateway, `auth` (JWKS), and the operator UI can reach the Vault. Ship the SoD proof as the two-limb written argument above, backed by the standing regression test: agent token → redeem → 403.**

## Recommendations for the vault-relevant decisions ARCHITECTURE.md deferred to research

1. **ADOPT vs BUILD → ADOPT OpenBao, 2.5.x stable line (≥ 2.5.5, MPL-2.0), NOT the 2.6.0 beta.** Do not build a bespoke barrier; do not ship BUSL Vault (its license permits homelab use but adds competitive-use ambiguity and IBM single-steward risk for zero functional gain). Run OpenBao as the crypto/storage/leasing/audit engine — minimum surface (KV v2 + SSH engine + one Gateway auth method), promptly patched — and build the Vault app as a thin wrapper that owns the SoD inversion. Verdict from the adversarial pass: HOLDS_WITH_CAVEATS — treat OSS namespaces/PKCS#11 as out-of-scope bonuses (no Enterprise compat; HSM distro EOL 2.7.0), record the OpenSSF-Sandbox fork-maturity risk (mitigated by MPL-2.0 self-fork rights, but back-migration to BUSL Vault post-divergence is unvalidated), and keep HashiCorp Vault only as a license-encumbered break-glass fallback, not a guaranteed exit.
2. **Gateway-only-redeem boundary → path-scoped ACL on a deny-by-default backend + app-endpoint enforcement, never response-wrapping-as-access-control.** Attach `read`-on-`secret/data/hosts/+/root` to the Gateway entity only; grant agents nothing + explicit `deny`. Authenticate the Gateway via JWT/OIDC bound on a Gateway-unique claim. The agent handle is a powerless app-level reference; response wrapping is used only on the Vault→Gateway leg. This boundary HOLDS only with the six §3 conditions (unique-claim binding, agent `identity/*` write denial + patched backend, `no_default_policy` agent tokens, no CIDR-as-identity, no Gateway direct-to-backend bypass, no agent reach to root/seal material).
3. **What the Vault stores → short-lived by default, static only where forced.** SSH-CA signed certificates (not stored root passwords) broker all ~20 hosts' shell/root access, with OpenBao ≥ 2.0.2 and `allow_empty_principals=false`; irreducible static secrets live in KV with rotation + short-TTL Gateway-leg wrapping. The crown jewels become the SSH CA signing key and the unseal-key set.
4. **Redemption authority split → Vault is the fourth independent lock, not the arbiter.** Board-approval + CMDB-in-window checks live upstream at the Gateway (or in the Vault app's redeem endpoint if it calls Board+CMDB at release time); the OpenBao ACL is only the hard floor on who can read the path. Enterprise Control Groups would do the four-way gate natively but are out of scope for an OpenBao build.
5. **auth integration → JWT/OIDC resource-server mode against `auth`'s JWKS,** one redeem role binding `aud`=vault + a Gateway-unique claim → a narrow `redeem` policy. Enforce SoD at both ends. **Planning is BLOCKED pending `auth`'s locked token model** (scope-vs-claim shape, native RFC 8707, RFC 9396 RAR feasibility).
6. **Audit → adopt the audit-device model:** HMAC-salted, `log_raw` forbidden, fail-closed on write with ≥2 sinks of different types, and external WORM/SIEM immutability (Vault does not tamper-proof its own log).
7. **Seal / secret-zero → operator policy decision, do not hardcode.** Prefer on-prem Transit auto-unseal for unattended restart without a cloud tether, or manual Shamir if the operator accepts unseal-on-restart; keep unseal/recovery shares outside the container; provide an out-of-container host-access break-glass for a sealed Vault.

## Open design questions to settle before Planning

1. **Planning is BLOCKED pending `auth`'s locked token model.** The exact scope/claim that gates redemption (`vault:read-credential` is a placeholder), whether the "is this the Gateway, for this ticket, now?" decision is a static scope, an RFC 9396 RAR grant, or a central PDP call, and whether the redeem role binds on `bound_subject` vs a `roles`/`groups` claim — all depend on `auth`. The invariant does not change; the gate mechanism does.
2. **SSH-CA vs OTP as the house default:** accept SSH-CA's cryptographic-expiry-only model (short TTL + enforced NTP + KRL/CA-rotation runbook, no revoke-on-use) or pay the `vault-ssh-helper`/PAM tax on all hosts for OTP's per-login revocation? What cert TTL fits a Gateway run, and does slow patch+reboot need mid-run renewal?
3. **Direct dynamic host creds vs brokered static?** If the model is truly short-lived signed certs, the dynamic-secrets engine is required; if it is brokering static root redeemed per-action with rotation, the field widens. This changes what the Vault actually stores.
4. **Seal/unseal for a continuously-running homelab:** manual Shamir (no unattended restart) vs on-prem Transit auto-unseal (needs a second unsealer OpenBao, whose own seal must not be Static-Key) vs cloud KMS (breaks airgap). Where do the recovery/unseal shares physically live, and what is the documented break-glass for regenerating a root token — and for host access when the Vault is sealed?
5. **Where the four-holder precondition check physically lives:** inside the Vault's redeem endpoint (Vault calls Board+CMDB — strong SoD, adds runtime coupling) or does the Gateway present signed approval/policy tokens the Vault verifies offline? This is the boundary between §3/§5 and the Gateway/SoD sections, and it determines whether a Gateway token alone can bypass the gate.
6. **Response wrapping on the Vault→Gateway leg — worth the complexity** for a single-operator homelab, or is mutual-TLS + short-lived direct release sufficient? Defense-in-depth vs simplicity.
7. **Agent-facing surface minimization:** does `request_release` need to succeed *before* approval (staging a pending release the operator/CMDB unlocks) or only after? Should `list_handles`/`describe_handle` expose rotation version markers at all (minimizing recon surface)? One `release_id` per handle or one bundle per ticket (bundling widens the blast radius of a single leaked reference)?
8. **Where the immutable off-box audit copy lives** so the Vault host root cannot rewrite it — a hardened log host, Drive with object-lock, or an external SIEM — and whether denied agent redemption attempts trigger an active escalation (chat/mission-control), not just a log line. Retention/rotation must ship to WORM before local deletion.
9. **Does the operator accept fail-closed-on-audit as a hard invariant** (a redemption is refused if it cannot be logged), trading availability of real-world patching for non-repudiation?
