# Stage 1 Research — `cmdb` Inventory + Policy Brain (Critical-infra)

This document is the Stage-1 research foundation for **`cmdb`**, the fleet inventory and — load-bearingly — the **policy brain** of the agent-corp suite. The CMDB answers the single question the Gateway must ask before it touches a host: *is this host allowed to be touched right now?* It is one of the four independent holders in the segregation-of-duties (SoD) chain — **Board holds approval, CMDB holds policy, Vault holds credentials, Gateway holds execution** — and it holds **no credentials and executes nothing**. Its job is to make the criticality tier, the maintenance window, and the auto-vs-ask decision *legible and queryable at decision time*, and to fail closed so that a correct plan on an out-of-window or wrong-tier host is refused here regardless of ticket or plan state. Because this is a Critical-infra app, every external fact is cited to a real URL and version/spec/endpoint claims that are fast-moving are flagged in a consolidated **Verify at build time** list rather than trusted from memory. Today is **~2026-07-01**; two live spec-drift facts frame the build: the current Wazuh line is **4.14.x** (4.14.6 shipped 2026-07-01; **5.0 is beta only** and must not be targeted), and the MCP authorization spec's current stable revision is **`2025-11-25`** with the **`2026-07-28` revision still a Release Candidate** roughly four weeks out.

## Executive summary

- **RESOLVED — Deferred Decision #1 (adopt vs build): BUILD a thin, opinionated policy service; do NOT adopt NetBox, iTop, or Ralph as the policy brain.** All three are mature but *inventory-first* systems; none models "may this host be touched right now" as a first-class fail-closed predicate, and wrapping one makes its own DB the de-facto source of truth — violating "markdown is the source of truth" and "two views, one state." The honest reason is **canonicality and SoD-independence, not audit** (NetBox in fact ships built-in change logging + journaling — do not claim adoption forfeits audit). ([NetBox Installation](https://netboxlabs.com/docs/netbox/installation/), [NetBox Change Logging](https://netboxlabs.com/docs/netbox/features/change-logging/))
- **RESOLVED — Deferred Decision #2 (policy-schema shape): a (criticality-tier × action-class) matrix over four operator-owned tiers, plus RFC 5545 RRULE+TZID+DURATION maintenance/freeze windows with deny-overrides precedence.** Approval is *not* a scalar per-host flag: reversibility is a property of the **action** (a reboot is irreversible even on a low-tier host), so the auto-vs-ask decision must key on (tier × action-class). ([NCSC tiers](https://www.ncsc.gov.uk/collection/secure-system-administration/risk-manage-administration-using-tiers), [itsm.tools](https://itsm.tools/change-enablement/), [RFC 5545](https://datatracker.ietf.org/doc/html/rfc5545))
- **SoD posture — the CMDB verdict is a *veto, not a trigger*.** A `permit`/`in_window` answer is *necessary but not sufficient*; a fully-compromised CMDB that returns `permit` for everything still causes zero destructive actions because it holds no ticket, no credentials, and no execution path. Conversely a CMDB `deny` is a hard veto no other holder can override except by an audited operator policy change. ([NIST SP 800-162](https://nvlpubs.nist.gov/nistpubs/specialpublications/nist.sp.800-162.pdf), [XACML 3.0 §7.2.2](https://docs.oasis-open.org/xacml/3.0/xacml-3.0-core-spec-os-en.html))
- **The SoD proof is NOT airtight as first drafted and must be hardened before Stage 5/7.** Two structural leaks survive adversarial review: (a) tier **auto-approve merges the Board-approval and CMDB-policy holders into one prior operator act**, so auto actions have three independent runtime checks, not four — destructive/irreversible classes must be structurally non-auto; and (b) the real bypass is the **agent-supplied `action_class` query input**, not the read-only stored policy — `action_class` MUST be derived by the Gateway from the specific vetted playbook, never accepted from the agent. ([XACML 3.0](https://docs.oasis-open.org/xacml/3.0/xacml-3.0-core-spec-os-en.html))
- **Inventory is a synced index; policy is git-backed markdown.** The host roster is *discovered* from the Wazuh server API `GET /agents` (port 55000) and is a rebuildable SQLite index keyed by a **stable operator-assigned host id** (never the recyclable Wazuh agent id); the criticality tier / window / approval-mode is operator-authored markdown + git. But inventory must be a **superset** — non-agent critical assets (NAS, hypervisors, network gear) that run no Wazuh agent still need policy. ([Wazuh listing agents](https://documentation.wazuh.com/current/user-manual/agent/agent-management/listing/listing.html), [Wazuh getting started](https://documentation.wazuh.com/current/user-manual/api/getting-started.html))
- **Fail-closed everywhere, and observably.** Unknown host, no policy on file, unreachable CMDB, malformed/unknown action-class, clock skew, and ambiguous/DST-unresolvable windows all **deny** (never "at most ask"); a missing affirmative `permit` is itself a deny. DST fold/gap on a window boundary must emit an escalation, never a silent no-op. ([XACML 3.0 §7.2.2](https://docs.oasis-open.org/xacml/3.0/xacml-3.0-core-spec-os-en.html), [PEP 495](https://peps.python.org/pep-0495/))
- **Read-only agent surface; verdict signed and re-queried at execution.** The agent MCP surface exposes zero mutation verbs (widening a window or lowering a tier would let an agent unlock its own destructive action); the centerpiece is an explainable `is_actionable_now(host, action_class?, at?)` verdict backed by one pure `evaluate()` shared verbatim with the Gateway's binding execution-time call and the operator dry-run screen. The verdict is **point-in-time, integrity-protected, and non-cacheable across the action**. ([MCP 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization), [RFC 8707](https://www.rfc-editor.org/rfc/rfc8707.html))
- **Planning is BLOCKED pending auth's locked identity model.** The "no agent role is ever mintable `cmdb:write-policy`" guarantee, the stable cross-app host-identity key, and the audience/scoping shape all depend on decisions owned by the `auth` component; CMDB Planning cannot lock until those are fixed.

## 1. Adopt an existing CMDB vs build a bespoke policy registry

**Findings.** The three obvious adopt candidates are all mature and actively maintained, and — decisively — all *inventory-first* systems, not policy engines. **NetBox** is at **v4.6.3 (2026-06-16)** and is DCIM/IPAM-first: its self-host footprint is Python 3.12–3.14 + **PostgreSQL 14+** + **Redis** (two logical DBs, tasks and caching) + a Gunicorn/uWSGI app server + an rq worker + a fronting HTTP server ([NetBox v4.6.3 release](https://github.com/netbox-community/netbox/releases/tag/v4.6.3), [NetBox Installation](https://netboxlabs.com/docs/netbox/installation/)). It exposes a REST API and a **read-only GraphQL API**, and models arbitrary per-object attributes via **custom fields**, **config contexts**, and **tags** ([NetBox GraphQL API](https://netboxlabs.com/docs/netbox/integrations/graphql-api/), [NetBox Custom Fields](https://netboxlabs.com/docs/netbox/customization/custom-fields/)). **iTop** (Combodo) is at **3.2.3-1 (PHP 8.4 compatible)**, a PHP/MySQL(MariaDB) ITIL suite with a native CI + relationship model, change-management tickets, and a REST/JSON API ([iTop releases](https://github.com/Combodo/iTop/releases), [iTop REST/JSON services](https://www.itophub.io/wiki/page?id=2_2_0:advancedtopics:rest_json)). **Ralph** (Allegro) is genuinely alive — CalVer releases roughly monthly, latest **20260609.1 (2026-06-09)** — a Django/Python + MySQL DCIM/asset system with a REST API ([Ralph releases](https://github.com/allegro/ralph/releases), [Ralph GitHub](https://github.com/allegro/ralph)). So maintenance status is not the discriminator; *fit to the CMDB's actual job* is.

The CMDB's job in this suite is narrow and load-bearing: be the **policy holder** in the segregation-of-duties chain and answer the Gateway's single decision-time question — is this host inside its maintenance window, and does its criticality tier auto-approve or force operator approval. Evaluated against that, the adopt candidates share disqualifying problems.

> **Load-bearing — none of these model "may this host be touched right now" natively.** NetBox has no maintenance-window primitive at all; you would represent tier/window/mode as **custom fields** and the Gateway would fetch them and do the in-window time math itself. iTop has change-management and CI states but its "maintenance window" lives in change tickets and SLA/coverage-window objects, not as a queryable per-host boolean the Gateway can trust. Ralph has no concept of it whatsoever. In every case the *authoritative* predicate is computed by the caller from raw fields — which means the policy decision does **not** actually live in the CMDB, defeating the point of it being an independent SoD holder.

The decision-time query ergonomics also cut against adoption. The Gateway needs a **single low-latency, fail-closed** call: one host in, an allow/deny + reason out. NetBox's read-only GraphQL can return custom-field values in one round trip ([NetBox GraphQL API](https://netboxlabs.com/docs/netbox/integrations/graphql-api/)), but it returns *data*, not a *decision*; iTop's REST/JSON is a single POST taking an operation + OQL query in a JSON envelope ([iTop REST/JSON services](https://www.itophub.io/wiki/page?id=2_2_0:advancedtopics:rest_json)), not a clean resource GET. A bespoke service can expose exactly `GET /policy/decision?host=<id>&at=<ts>` returning `{allow, tier, in_window, mode, reason}` with deny-by-default semantics baked into the service that *owns* the policy. On self-host weight, all three are over-scaled for ~20 hosts, one operator, and low-dozens of agents.

The one thing adoption genuinely buys is **free inventory CRUD, relationship modeling, and a polished UI**. But at 20 homelab hosts the inventory is small, and the fleet is *already enumerated* by Wazuh: `GET /agents` returns each agent's `id`, `name`, `ip`, `os`, `status`, `version`, `lastKeepAlive`, `dateAdd`, and `group_config_status`, filterable via `select=` ([Wazuh server API reference](https://documentation.wazuh.com/current/user-manual/api/reference.html), [Listing agents](https://documentation.wazuh.com/current/user-manual/agent/agent-management/listing/listing.html)); current Wazuh is the **4.14.x** line ([Wazuh 4.14.5 release notes](https://documentation.wazuh.com/current/release-notes/release-4-14-5.html)). What remains genuinely bespoke is the *policy* overlay (tier/window/mode). What I would **graft from NetBox**: its **config-context + tag precedence** idea (tier as a tag; policy resolves host-specific override > tier default > fleet default), typed/validated custom-field discipline, and clear REST/GraphQL read-model separation; from **iTop**, only the *conceptual* first-class recurring-window object.

The adversarial pass accepted the top-line "build" call but broke three supporting arguments and flagged a load-bearing data-model hazard; those corrections are folded in below and the recommendation is adjusted accordingly.

> **Correction — adopting does NOT forfeit audit; it forfeits *git* specifically.** NetBox ships built-in **Change Logging** (a serialized before/after snapshot per create/update/delete, with user, timestamp, and a request-UUID correlating all changes from one action) plus **Journaling** for human context ([NetBox Change Logging](https://netboxlabs.com/docs/netbox/features/change-logging/), [NetBox Journaling](https://netboxlabs.com/docs/netbox/features/journaling/)). The honest reason to prefer bespoke is **canonicality and SoD-independence** — the suite invariant is that markdown+git is the *canonical* source and any DB is a rebuildable index — not the false claim that a NetBox UI edit is "un-audited." State it that way.

> **Adversarial — the markdown-canonical + async SQLite index design has a fail-OPEN crack on the one predicate that must be fail-closed.** If the decision query reads a SQLite index that lags a git commit (operator closes a maintenance window in markdown, reindex hasn't run), the CMDB answers `allow` from stale policy — a fail-*open* error on exactly the "may this host be touched right now" question. A stale-but-valid index is **not** an error state, so "deny on any internal error" does not catch it. Fix: either (a) the decision service reads the canonical store directly (parse-on-read is trivial at ~20 hosts and eliminates index lag), or (b) invert the invariant application and make SQLite the *canonical* transactional store for the decision with a git-committed markdown *mirror* for human diff/audit. Do **not** ship "markdown canonical + async index feeding the gate."

> **Adversarial — the NetBox-plugin hybrid was strawmanned and must be costed, not assumed away.** The rejection claimed adoption forces the decision logic into the Gateway ("returns data, not a decision"). That is a design choice, not a constraint: NetBox supports **Custom Scripts / plugins** with REST-invocable endpoints ([NetBox Custom Scripts](https://netboxlabs.com/docs/netbox/customization/custom-scripts/)), so a plugin could expose `may_touch(host, at) -> {allow,...}` *inside* NetBox over its own custom fields and changelog. Reject the hybrid only on demonstrated grounds — binding a Critical-infra SoD policy holder to NetBox's Postgres+Redis+worker footprint and its release/CVE cadence, and coupling the decision to a large general-purpose codebase — not on the false "logic must live in the Gateway" claim.

> **Adversarial — Wazuh-as-sole-inventory-source under-covers the riskiest hosts.** `GET /agents` only returns hosts running an enrolled Wazuh agent. The reference scenario's high-criticality, rollback-required assets — the NAS, network gear, hypervisors, appliances — frequently do **not** run a Wazuh agent yet most need a tier and window. So inventory must be a **superset** (Wazuh ∪ operator-authored non-agent hosts), which restores a small amount of the inventory-CRUD value the section otherwise claims Wazuh "collapses." Say that plainly.

> **Correction — version currency (re-verify at build).** NetBox latest is **4.6.4 (2026-06-30)**, not 4.6.3, and its GraphQL remains read-only; Wazuh has shipped **4.14.6**, so 4.14.5 is not "current"; iTop **3.2.3 LTS (PHP 8.4)** is confirmed but community sources date 3.2.3 to ~April 2026 ([NetBox releases](https://github.com/netbox-community/netbox/releases), [Wazuh release notes index](https://documentation.wazuh.com/current/release-notes/index.html), [iTop community releases](https://www.itophub.io/wiki/page?id=latest%3Arelease%3Astart)). None change the conclusion; correct the "newest" claims.

**Recommendation (adjusted for the adversarial verdict — ACCEPT-WITH-CHANGES).** **Build a thin, opinionated policy registry that owns the fail-closed `may_touch(host, at) -> {allow, tier, in_window, mode, reason}` decision; do not adopt NetBox, iTop, or Ralph as the policy brain — but on corrected grounds.** Concretely: (a) **prefer bespoke for canonicality and SoD-independence, not audit** — drop the "adopting forfeits audit" claim. (b) **Resolve the canonical/index fail-open**: at ~20 hosts either read the canonical store directly on the decision path, or make the transactional store canonical and emit a git-committed markdown mirror; never let a stale async index answer the gate — and **argue the markdown-vs-DB canonicality choice explicitly** rather than defaulting to markdown (this is where Deferred Decision #2's schema seam is settled — see §5/Resolution). (c) **Fairly cost, then reject-with-reason, the NetBox-plugin hybrid** on footprint/CVE-cadence/coupling grounds. (d) **Inventory is a superset**: sync Wazuh `GET /agents` as one source but let the operator author rows for non-agent critical assets. (e) **Right-size honestly** — acknowledge the bespoke path's real surface (RRULE evaluator, markdown/git handling, MCP server, operator UI, and the Critical-infra hardening of all of it, forgoing NetBox's community scrutiny) and argue homelab scale still favors bespoke rather than pretending it is "a few SQLite tables." Graft NetBox's tag/config-context precedence and typed-custom-field discipline and iTop's first-class-window concept; take nothing from Ralph beyond confirming the tradeoff.

## 2. Sync the host list FROM Wazuh — the Wazuh inventory/server API

**Findings.** As of ~2026-07 the current Wazuh release line is **4.14.x**, not a 5.x: 4.14.6 shipped 1 July 2026, preceded by 4.14.5 (23 April 2026), 4.14.4 (17 March 2026), and 4.14.3 (11 February 2026) ([Wazuh 4.x release notes index](https://documentation.wazuh.com/current/release-notes/index-4x.html)). Wazuh 5.0 exists only as a beta and should not be targeted — **build against 4.14.** The surface we integrate with is the **Wazuh server API** (renamed from "manager API" in the 4.x docs), a RESTful, JWT-secured API listening on **port 55000** ([Wazuh server API — getting started](https://documentation.wazuh.com/current/user-manual/api/getting-started.html)). The CMDB *reads* the fleet from here and never writes back — it holds no credentials and executes nothing.

**Authentication.** Get a JWT via **`POST /security/user/authenticate`** with HTTP Basic credentials; append **`?raw=true`** for the bare token string. Pass it as `Authorization: Bearer <token>` on every subsequent call ([getting started](https://documentation.wazuh.com/current/user-manual/api/getting-started.html)). Default token lifetime is **900 seconds**, governed by `auth_token_exp_timeout`, changeable via **`PUT /security/config`** — and any change to the security config revokes all outstanding tokens ([Wazuh server API configuration](https://documentation.wazuh.com/current/user-manual/api/configuration.html)). RBAC is enforced from claims embedded in the token.

> **Load-bearing — the CMDB needs a scoped, read-only Wazuh account.** The default `wazuh:wazuh` credential is full-admin. Create a dedicated RBAC role granting only the read actions the sync uses (`agent:read`, `syscollector:read`, `group:read`) and nothing else, so a compromised CMDB cannot delete or reconfigure agents. The raw Wazuh credential is a secret (a Vault concern), not a config value ([RBAC reference](https://documentation.wazuh.com/current/user-manual/api/rbac/reference.html)).

**Enumerating the fleet.** **`GET /agents`** returns one record per registered agent. Fields include `id`, `name`, `ip`/`registerIP`, `os` (nested: `arch`, `name`, `platform`, `uname`, `version`), `status`, `status_code`, `group` (array), `lastKeepAlive`, `version`, `dateAdd`, `node_name`, `manager`, `mergedSum`/`configSum`, and `group_config_status` ([listing agents](https://documentation.wazuh.com/current/user-manual/agent/agent-management/listing/listing.html)). Connection state is the operational one: `active`, `disconnected`, `pending`, `never_connected`. `GET /agents/summary/status` gives a cheap fleet-wide count ([Wazuh server API reference](https://documentation.wazuh.com/current/user-manual/api/reference.html)).

> **Correction — the Wazuh agent `id` is not a stable host identity.** Agent ids (e.g. `001`) are assigned at registration and *reused* when an agent is removed and re-enrolled; a rebuilt host re-registers under a new id. Do **not** key the CMDB's fleet on the Wazuh agent id. Key on an operator-assigned stable host identity, and treat `{agent_id, name, registerIP, dateAdd}` as discovered attributes that can drift.

**Host facts (syscollector).** **`GET /syscollector/{agent_id}/<resource>`** gives hardware/OS/software facts. Confirmed resources: **`/os`**, **`/hardware`**, **`/packages`**, **`/ports`**, **`/processes`**, **`/netiface`**; the reference also documents `/netaddr`, `/netproto`, `/hotfixes` (verify spelling live) ([Wazuh server API reference](https://documentation.wazuh.com/current/user-manual/api/reference.html)). For the CMDB, `/os` + `/hardware` are the useful enrichment for eyeballing a host when assigning a tier; `/packages` is heavier and belongs to the recon/patching scenario.

> **Load-bearing — Wazuh knows the *facts*, never the *policy*.** Syscollector and `/agents` tell us OS, hardware, IP, group, and liveness. They cannot tell us a host's **criticality tier**, **maintenance window**, or **auto-vs-ask** mode — those are operator-authored policy the CMDB *adds on top*. The sync's job is strictly to keep inventory honest.

**Vulnerability data lives in the indexer, not this API.** Since the 4.8 Vulnerability Detection redesign (June 2024) there is no per-agent vulnerability endpoint on the server API; detected vulns are forwarded to the **Wazuh indexer** and stored in the **`wazuh-states-vulnerabilities-*`** index, queried over the indexer's OpenSearch API on **port 9200** ([4.8.0 release notes](https://documentation.wazuh.com/current/release-notes/release-4-8-0.html), [Wazuh indexer indices](https://documentation.wazuh.com/current/user-manual/wazuh-indexer/wazuh-indexer-indices.html)). **The CMDB should not touch this** — it belongs to the Gateway/recon path.

**Pagination, limits, cadence.** List endpoints share `limit` (default **500**, max **100000**), `offset`, `select`, `sort`, `search`, and `q` ([Wazuh REST API 4.14 reference, opennix mirror](https://opennix.org/en/docs/wazuh/development/wazuh-api-reference/)). At ~20 hosts `GET /agents` returns in a single page; a full syscollector enrichment cycle is ~20 sequential calls. The docs describe a configurable request throttle (`max_request_per_minute`) rather than a hard fleet limit; a **10–15 minute pull cadence** aligns with the 900 s token lifetime.

**Reconciling discovered hosts against operator policy.** The sync is a three-way reconcile between what Wazuh reports now, the CMDB's last-known mirror, and operator-authored policy: **new in Wazuh, absent in CMDB** → create an *un-policied* inventory record surfaced as "needs tiering," not touchable until the operator assigns tier/window (fail-closed); **in CMDB, gone from Wazuh** → mark stale/quarantined, never silently delete the operator's policy; **drift** (OS/IP/group changed) → update discovered facts, leave policy untouched, log the diff. The *facts* mirror is a rebuildable SQLite index; the *policy* is git-backed markdown the reconcile reads but never overwrites.

> **Adversarial — the group-anchor idea contradicts fail-closed and is an enrollment-attack surface.** The section both suggests a Wazuh group can seed a "sane default" tier/window for new hosts *and* asserts new hosts land at the most-restrictive default. These conflict. Worse, `group` is set at agent enrollment and is influenceable; an attacker or runaway agent who sets a new agent's group to e.g. `scratch` could inherit a permissive/auto tier on a host the operator never reviewed. The group anchor must be a **UI-only suggestion requiring explicit operator commit**; the new-host default must be fail-closed **regardless of group**.

> **Adversarial — the (Wazuh agent ↔ policy host) identity-binding seam is unspecified and security-relevant.** The design correctly keys on a stable operator-assigned id, but never says *how* a discovered agent is matched to it. Matching on `name`/`registerIP` (both attacker-controllable at enrollment) means re-enrolling a machine under a low-tier host's name/IP can bind a lax policy to an attacker's box, or rebind a high-value host's policy record. Every rebind must be an **operator-confirmed, git-audited event**, never an automatic fuzzy match.

> **Correction — inventory is a superset (see §1).** Because non-agent critical assets (NAS, hypervisors, network gear) run no Wazuh agent yet still need policy, the operator must be able to author inventory rows for them. Wazuh is one discovery source, not the sole roster.

**Recommendation (adjusted).** Treat the Wazuh server API v4.14 (port 55000) as a **read-only discovery feed**, polled every 10–15 minutes by a dedicated RBAC-scoped read-only account: `POST /security/user/authenticate?raw=true` → one `GET /agents` (with `select` to trim fields) → per-agent `GET /syscollector/{id}/os` and `/hardware` → `GET /groups`. Persist as a **rebuildable SQLite facts-mirror keyed by a stable operator-assigned host id** (never the recyclable Wazuh agent id), reconciling add/remove/drift against **git-backed markdown policy** the sync reads but never mutates. Make inventory a **superset** (Wazuh ∪ operator-authored non-agent hosts). Treat Wazuh groups as a **UI-only tiering suggestion requiring explicit operator commit**, never auto-applied policy; make every (agent ↔ policy host) bind/rebind an operator-confirmed, git-audited event. Fail closed: any host without operator policy is untouchable by default. Explicitly **exclude vulnerability data** — it moved to the indexer's `wazuh-states-vulnerabilities-*` index (port 9200) in 4.8 and is the Gateway's concern.

## 3. Policy schema, part 1 — criticality tiers + the auto-approve-vs-always-ask switch

**Findings.** The CMDB's approval half of the policy schema maps cleanly onto two decades of ITIL change practice. ITIL splits every change into three models — **standard** (pre-authorized, low-risk, repeatable, needs *no* fresh approval per execution), **normal** (risk-assessed case-by-case), and **emergency** (expedited, with post-implementation review) ([IT Process Wiki — Change Management](https://wiki.en.it-processmaps.com/index.php/Change_Management), [itsm.tools — Change Enablement](https://itsm.tools/change-enablement/)). The load-bearing concept is the **standard change**: in ITIL 4 "pre-authorized" means *the procedure* was risk-assessed and authorized once, so each execution "does not require operational oversight by a change authority" ([itsm.tools](https://itsm.tools/change-enablement/)) — precisely the "tier can auto-approve" switch the Gateway reads. ITIL 4 also decentralizes the approver into a **change authority** that varies by change type. In agent-corp terms: the operator is the change authority, and the CMDB tier policy *is* the operator's standing pre-authorization.

Criticality tiering is the other axis, and real practice converges on a small ordinal scale. ITIL treats **criticality as a first-class CI attribute** ([Cloudaware — CMDB CI](https://cloudaware.com/blog/cmdb-ci/)). ServiceNow ships a 4-value business-criticality enum ([ServiceNow — define criticality](https://www.servicenow.com/docs/bundle/xanadu-it-operations-management/page/product/service-mapping/task/define-criticality-for-business-services.html)). A representative four-tier standard runs *Tier 1 Mission Critical → Tier 4 Administrative Non-Critical* ([Service Criticality Tiers, S. Samy](https://www.linkedin.com/pulse/service-criticality-tiers-standard-architecture-sherif-samy)). Crucially, NCSC's tiered-administration model makes the same move for *security control strength*: **Tier 0** is "the root of trust that all other administration relies upon" with break-glass-only access, descending to a "constrained set of functions" where compromise is "undesirable… but not catastrophic," with "stricter protections increase with tier criticality" ([NCSC — risk-manage administration using tiers](https://www.ncsc.gov.uk/collection/secure-system-administration/risk-manage-administration-using-tiers)). NCSC's framing is the right one to borrow, because the CMDB's tier drives an *approval/blast-radius* gate, not a DR target.

> **Correction — criticality tier ≠ availability/DR tier.** Most "Tier 0/1/2/3" taxonomies size RTO/RPO, HA redundancy, and SLA ([Service Criticality Tiers](https://www.linkedin.com/pulse/service-criticality-tiers-standard-architecture-sherif-samy)). The CMDB reuses the *label* but the semantic is "how much operator scrutiny must a destructive action here carry" (the NCSC control-strength sense). A host can be low-availability yet still forced to always-ask if it shares a package set with a canary.

On whether auto-vs-ask should be a pure function of tier: **no — it must key on (tier × action-class).** A standard change is pre-authorized *per well-understood procedure*, not per host. Reversibility is a property of the *action*: a package update can be canaried and rolled back, but a **reboot is not reversible even on a "low" tier host**. Google SRE's change-safety model is built on this axis — "if change is isolated AND covered by automated canary AND reversible, it should be auto-approved," with gated rollout and automatic rollback ([Google SRE — Canarying Releases](https://sre.google/workbook/canarying-releases/)). The correct shape is a small **(tier × action-class) matrix** whose cells hold `auto | ask`, the tier supplying the *default row*, with per-host overrides.

> **Load-bearing — "auto-approve" is not the agent approving its own work; it is the operator pre-approving a *class* ahead of time.** In ITIL the standard-change *procedure* is authorized once by a change authority; only then do executions skip fresh approval ([itsm.tools](https://itsm.tools/change-enablement/)). The operator authors the CMDB tier policy (a human act, ahead of time); the agent never flips its own Board ticket to `approved`. At execution the Gateway asks the CMDB "is this (host, action) pre-authorized *by policy*?" — an operator-authored fact the agent cannot mint. The failure mode to prevent is letting the agent *edit* policy — hence the agent surface is read-only.

> **Correction — auto-approve is only valid for actions in a vetted catalog.** A cell may be `auto` *only* if the action-class corresponds to a pre-vetted, parameterized Gateway playbook (the ITIL "well-known procedure" / SRE "canaried + reversible" precondition). Any action-class **not** in the standard-change catalog resolves to `ask` regardless of tier. Catalog membership is a join between CMDB policy and the Gateway's playbook registry.

For a ~20-host homelab with one operator, **four tiers is the ceiling, three is defensible**: **tier0 irreplaceable/root-of-trust** (always ask, every class — includes the NAS, hypervisor, and the hosts running the SoD chain itself: auth, Vault, Gateway, CMDB) → **tier1 critical service** → **tier2 standard** → **tier3 disposable/stateless (canary-eligible)**. A minimal action-class enum carrying the reversibility signal: `package_update`, `config_change`, `service_restart`, `reboot`, `kernel_update`, `destructive`. Resolution fails safe to `ask`: `effective_mode(host, action_class) = host.override[action_class] ?? tier_default[host.tier][action_class] ?? "ask"`, and forces `ask` if `action_class ∉ standard_change_catalog`. The CMDB answer composes with the sibling window decision: `may_act_now = approval_satisfied(tier, action_class) AND in_window(host)`.

The adversarial pass accepted the two-axis model but found the SoD proof over-claimed on two structural points; both are folded in and the recommendation adjusted.

> **Adversarial — the `action_class` trust boundary is the real SoD leak, and it is unspecified (highest severity).** The (tier × action_class) matrix is only as trustworthy as the mapping from a concrete command to an `action_class`. If the Gateway forwards the *agent's asserted* class, a runaway agent labels a `reboot` as `package_update`, hits an `auto` cell, and bypasses the whole gate without writing a single policy field. Read-only policy is necessary but **not sufficient** — the leak is the query input, not the stored table. Invariant: **`action_class` MUST be derived by the Gateway from the specific vetted catalog playbook it is about to execute (playbook → fixed action_class binding), never accepted from the agent's request.** A cell may be `auto` only if the playbook exists in the standard-change catalog **and declares a rollback path**.

> **Adversarial — honest SoD accounting for auto: four holders collapse to three at runtime.** ARCHITECTURE ties tier auto-approve to clearing the Board's `awaiting_approval` ("by operator OR by CMDB tier policy"), and agents can create/claim their own Board tickets. For an `auto` cell the *same* operator-authored tier policy satisfies **both** the Board-approval holder and the CMDB-policy holder — two of four supposedly-independent holders reduce to one prior operator decision. Auto actions therefore have **three** independent runtime checks (standing-policy, Vault creds, Gateway executor), not four. State this explicitly in the security stage and justify why 3-of-4 is tolerable for the specific auto-eligible classes — do not assert the four-holder chain is unchanged.

> **Adversarial — `package_update` is misclassified as routinely-reversible.** Kernel/glibc/init upgrades, bad maintainer postinstall scripts, and dependency cascades can brick a host or require an implicit reboot; the section's own reversibility argument applies to a kernel or libc package. Split kernel/bootloader/init/libc into their own high-gate class, **or** make auto-eligibility a property of the specific vetted playbook + proven rollback path, not of the coarse `action_class`.

> **Adversarial — host-identity aliasing is an input-side leak.** `effective_mode` keys on `host → tier`; if one physical machine is reachable under multiple identifiers (hostname vs IP vs alias) not canonicalized to one CI, an agent can query the lenient alias to get a more permissive tier. **One physical host = one canonical CI; resolve all aliases before tier lookup** as an explicit gate precondition.

**Recommendation (adjusted for the adversarial verdict — ACCEPT-WITH-CHANGES).** Model approval as a **(criticality-tier × action-class) matrix**, not a scalar per-host flag. Adopt **four operator-owned tiers** (tier0 irreplaceable/root-of-trust including the SoD-chain hosts → tier3 disposable/stateless) using NCSC control-strength semantics, plus a **six-value action-class enum**. The tier supplies a default `{action_class → auto|ask}` row; per-host `override[action_class]` handles exceptions; resolution **fails safe to `ask`** for any unknown host/action/tier or any action-class outside the vetted Gateway standard-change **catalog**, and a cell is `auto`-eligible only if its playbook exists in the catalog **and declares a rollback path**. Make the **`action_class` derived by the Gateway from the vetted playbook, never accepted from the agent** — this is the actual SoD proof, and read-only stored policy is necessary but not sufficient. **Do not trust `package_update` as blanket-reversible** — split kernel/bootloader/init/libc into a high-gate class or tie auto-eligibility to the playbook + rollback path. Require **host-identity canonicalization** (one host = one CI) before tier lookup. Persist **tiers + approval modes + catalog as git-audited policy** (source of truth) with a rebuildable DB index for live facts; the agent surface is **read-only** and all mutation is operator-only. State honestly that tier auto-approve **merges the Board and CMDB holders into one prior operator act** (three runtime checks, not four) and justify it for the specific auto-eligible classes; the emergency-change path is an operator-only break-glass with post-hoc review.

## 4. Maintenance-window representation + the "in-window NOW?" computation

**Findings.** A maintenance window is *policy*, not convention, so its representation must be unambiguous, diffable, and computable to a single yes/no at an instant. The industry-standard grammar is the iCalendar **RECUR** value type (RRULE) of RFC 5545 §3.3.10: `;`-separated rule parts where `FREQ` is REQUIRED and appears once, `INTERVAL` sets the multiplier, `BYDAY`/`BYHOUR`/`BYMINUTE` pin the wall-clock slot, and `UNTIL`/`COUNT` bound it (mutually exclusive) ([RFC 5545](https://datatracker.ietf.org/doc/html/rfc5545), [dateutil rrule](https://dateutil.readthedocs.io/en/stable/rrule.html)). A "Sunday 02:00–04:00" window is `FREQ=WEEKLY;BYDAY=SU;BYHOUR=2;BYMINUTE=0` + a `DURATION` of 2h. RRULE beats the alternatives: **cron** has no duration, no `UNTIL`, no standardized timezone binding, and implementation-defined DST behavior ([croniter](https://github.com/pallets-eco/croniter)); a **hand-rolled day+start+duration** model fails on the first "last Sunday of the month" (`BYDAY=-1SU`, `BYSETPOS`) an operator wants.

**Timezone/DST is the load-bearing correctness problem.** RFC 5545 defines three DATE-TIME forms: **#1 floating/local**, **#2 UTC** (`...Z`), and **#3 local-time-with-TZID** (`TZID=America/New_York:19970714T133000`) ([icalendar.org §3.3.5](https://icalendar.org/iCalendar-RFC-5545/3-3-5-date-time.html)). A maintenance window is emphatically **Form #3**: the operator means "02:00 *in the host's civil time*," a fixed wall-clock slot whose *UTC* value shifts twice a year. **Store the window as local wall-clock + explicit IANA zone id and compute at query time — never precompute the window into UTC and store that**, which drifts an hour after every DST transition and goes stale whenever IANA tzdata rules change ([Ganssle, pytz footgun](https://blog.ganssle.io/articles/2018/03/pytz-fastest-footgun.html), [Python zoneinfo](https://docs.python.org/3/library/zoneinfo.html)).

> **Load-bearing — the two DST edge instants the "in-window now?" function must handle.** Fall-back repeats an hour (a **fold** → *ambiguous*); spring-forward skips an hour (a **gap** → *nonexistent*). Python encodes this via PEP 495's `fold` attribute ([PEP 495](https://peps.python.org/pep-0495/), [zoneinfo](https://docs.python.org/3/library/zoneinfo.html)). A 02:00–04:00 window on a US spring-forward night is the canonical trap. The CMDB must pick and document a deterministic policy for both cases and, for a safety gate, resolve conservatively (**fail-closed**): treat a query instant whose local mapping is ambiguous or nonexistent as *not* cleanly in-window.

> **Correction — RFC 5545 does not define fold/gap disambiguation; do not cite it for that.** The RFC specifies the *representation*; the "which 01:30 did you mean" resolution is supplied by the runtime (PEP 495 / `zoneinfo`). What the RFC *does* mandate: **`UNTIL` must be UTC (`...Z`) whenever `DTSTART` carries a TZID** — dateutil raises on a mismatch ([RFC 5545 §3.3.10](https://datatracker.ietf.org/doc/html/rfc5545), [nextcloud/calendar #2675](https://github.com/nextcloud/calendar/issues/2675)). So a window's recurrence anchor is local-with-TZID but its end-of-recurrence bound is UTC.

> **Load-bearing — dateutil.rrule does NOT do timezone math.** It increments wall-clock and carries `dtstart`'s tzinfo forward; expanding directly against UTC (or an aware `dtstart` without re-normalizing) drifts after a DST boundary. Correct pattern: **expand in the host's own IANA zone, then convert each occurrence to UTC through `zoneinfo`** ([collective/icalendar #162](https://github.com/collective/icalendar/issues/162)).

**Blackout/change-freeze windows are the inverse and need explicit precedence.** ITIL distinguishes *maintenance windows* from *freeze/blackout windows* ([Atlassian change windows](https://support.atlassian.com/jira-service-management-cloud/docs/what-are-change-windows/)). Same RRULE+TZID+DURATION shape, inverted sense. Precedence must be **deny-overrides**: an active freeze refuses even if an allow-window is open — the same algebra chosen for `auth`'s Cedar (`forbid` over `permit`): `in_window(T) := any(allow covers T) AND NOT any(freeze covers T)`.

**Boundary semantics — the CMDB answers a point, but a change takes time.** Use **half-open intervals `[start, start+DURATION)`**. "In-window now?" gates *entry*, not completion. Whether the whole patch-and-reboot fits before close is a **Gateway/Planning concern**, but the CMDB must supply the raw material: return not just a boolean but **`window_closes_at`** (and `seconds_remaining`). A small operator-configurable **grace/lead-time** ("don't start within N minutes of close") is window policy.

**Markdown-vs-DB for windows.** Window definitions live as YAML frontmatter (RRULE string + TZID + DURATION) in the per-host policy markdown; the DB is a rebuildable index. **Library landscape (mid-2026):** `python-dateutil` 2.9.0.post0 ships `rrule`/`rruleset` ([PyPI](https://pypi.org/project/python-dateutil/)); `croniter` 6.2.2 (pallets-eco) — solid but wrong grammar (no duration) ([PyPI](https://pypi.org/project/croniter/)); `recurring-ical-events` 3.8.2 sits above `icalendar`+`dateutil` if real `.ics` ingestion is ever needed ([PyPI](https://pypi.org/project/recurring-ical-events/)); stdlib **`zoneinfo`** (3.9+) is the IANA-backed, PEP-495-aware engine for the instant math ([zoneinfo docs](https://docs.python.org/3/library/zoneinfo.html)).

The adversarial pass found the section *named* the DST cases but did not *close* them in the concrete schema, and introduced an internal contradiction; both are load-bearing and folded in.

> **Adversarial — internal contradiction: the materialized-occurrence cache re-introduces the forbidden UTC-precomputed table.** The formula says "compute at query time, never precompute the window into UTC," but the markdown-vs-DB paragraph says the DB "materializes upcoming occurrences for fast in-window-now lookups." A materialized upcoming-occurrence table **is** a UTC-precomputed table one level removed: it goes stale across a DST transition, a tzdata update, or a markdown edit whose index has not rebuilt — and a safety gate trusting a stale cache returns a wrong `allow`. At 20 hosts RRULE expansion is microseconds; there is no performance case for it. **Resolve `in_window(host, T)` LIVE from RRULE+TZID at query time**; the DB may index inventory but must not be the authority for the gate.

> **Adversarial — `duration: "PT2H"` is a cited RFC 5545 correctness bug.** RFC 5545 §3.3.6 makes hours/minutes/seconds **accurate (fixed-elapsed) durations**; only weeks/days are nominal/DST-adjusting. `start + PT2H` = start_utc + exactly 7200 s, so on a fall-back night a "02:00–04:00 LOCAL" window (operator wall-clock intent) has its computed close land at wall-clock 03:00 or 05:00, not 04:00 — the window silently shrinks or extends by an hour twice a year, and `window_closes_at`/`seconds_remaining` are wrong on exactly the DST nights the section claims to handle. To mean wall-clock 04:00 the **END must be a second local-time anchor (or a nominal duration)**, and fold/gap must be closed for the window's own START and END, not just the query instant T.

> **Adversarial — fail-closed on a DST gap must be OBSERVABLE.** If an allow-window's start lands in the spring-forward gap and the code fail-closes, the window never opens that night; if it is the only window, a critical patch is silently deferred with no signal — realizing the "silent staleness / never-terminate" failure mode the architecture warns against. Fail-closed on the gate must **emit an escalation/log** ("window unresolvable due to DST gap, did not open"), never a silent no-op.

> **Adversarial — TOCTOU: the point-in-time answer needs a validity bound, and "must fit in window" is orphaned.** There is a gap between the CMDB answer and the Gateway acting; near an edge the answer is stale by redemption time. The answer needs a stated **as-of/validity timestamp** and the Gateway must **re-check at act time**. "Whole change must fit before close" also requires an action-DURATION estimate defined nowhere — either define where it comes from (playbook catalog) or drop the claim it is enforceable. `grace/lead-time` is **window policy** and belongs in the CMDB answer, not smeared into the executor.

> **Adversarial — "RRULE subset" must be an enforced allowlist, or the YAGNI counter wins.** Adopting RFC 5545 is fine only if the accepted grammar is a validated allowlist (`FREQ` WEEKLY/DAILY/MONTHLY + `INTERVAL` + `BYDAY`/`BYHOUR`/`BYMINUTE` + bounded `UNTIL`). Accept unrestricted RRULE and an operator can write `BYSETPOS`/`RDATE`/`EXDATE`/`COUNT`-with-`UNTIL` combinations the fast and live paths may evaluate differently and whose DST behavior the mitigations do not cover. **Reject unsupported rule parts at schema level.**

> **Adversarial — specify the full freeze/break-glass precedence lattice.** deny-overrides is defined for allow-vs-freeze, but break-glass is said to "waive the window" without saying whether it waives a **freeze** (which often protects the highest-risk periods). Decide and log the three-level lattice (**allow < freeze < break-glass**), and require break-glass-over-freeze to carry a **distinct, louder operator confirmation** than break-glass over a merely-closed window.

**Recommendation (adjusted for the adversarial verdict — ACCEPT-WITH-CHANGES).** Represent every window as an **RFC 5545 RRULE subset (`FREQ`+`INTERVAL`+`BY*`) + explicit IANA `TZID` + explicit end**, stored as local wall-clock (Form #3) in git-audited policy markdown; never cron, never a UTC-precomputed table, and **never a materialized-occurrence cache on the gate path** — resolve `in_window(host, T)` **live** at query time (microseconds at 20 hosts). Fix the **duration encoding**: represent the window END as a second local-time anchor (or a nominal duration) so it means wall-clock, and close fold/gap for START and END, not just T. Compute by: (1) expand the RRULE in the host's own IANA zone with `dateutil.rrule`, (2) normalize each occurrence and `T` to UTC via `zoneinfo`, (3) test half-open membership `[start, end)`, (4) apply **deny-overrides** so any active `freeze` beats any `allow`. Handle DST **fold/gap explicitly and fail-closed, and emit an observable escalation** when a window is unresolvable. Return a **struct with an as-of/validity timestamp**: `{ verdict, in_window, window_id, window_closes_at, seconds_remaining, active_freeze?, grace, evaluated_at, valid_until, tzid }`, and require the Gateway to **re-check at act time** (kill the TOCTOU); make `grace/lead-time` a CMDB window-policy field, and either source the action-DURATION estimate from the playbook catalog or drop the "must-fit-before-close" enforceability claim. **Enforce the RRULE subset with a schema-level allowlist** and specify the **allow < freeze < break-glass** lattice with a louder confirmation for break-glass-over-freeze. Use `zoneinfo`+`dateutil` in v1 (ship `tzdata` on slim/Windows containers); reach for `recurring-ical-events` only if `.ics` ingestion becomes a requirement.

## 5. Decision-time query + the CMDB's place in the segregation-of-duties chain

**Findings.** The CMDB's core job maps onto the NIST ABAC vocabulary `auth` already adopted: the CMDB is a **Policy Decision Point (PDP)** and the Gateway is the **Policy Enforcement Point (PEP)** that calls it before touching a host. NIST SP 800-162 defines the PEP as the component that "enforces policy decisions in response to a request from a subject," while "the access control decisions are made by the PDP," alongside a PIP (attributes) and PAP (authoring) ([NIST SP 800-162](https://nvlpubs.nist.gov/nistpubs/specialpublications/nist.sp.800-162.pdf)). The CMDB is a PDP with a deliberately narrow attribute set — it decides on `(host, action-class, environment=time)`, **not** principal identity (that is `auth`/Board's half).

**The query shape.** Model the request as the AuthZEN/XACML access-evaluation tuple, trimmed. The OpenID **Authorization API** ("AuthZEN") formalizes a *subject, action, resource, context* 4-tuple with "the simplest form of a response is simply a boolean representing a Decision" plus optional context/obligations ([OpenID AuthZEN Authorization API 1.0 draft-01](https://openid.net/specs/authorization-api-1_0-01.html)). The CMDB's concrete contract — **Request (Gateway → CMDB):** `host_id` (required, stable key), `action_class` (required), `now` (advisory; the CMDB evaluates against *its own* authoritative clock), `ticket_ref` (optional, **audit correlation only — never an authorization input**); the *subject* is intentionally absent. **Response (CMDB → Gateway):** a single machine-usable **`verdict ∈ {deny, ask, permit}`** backed by evidence: `in_window`, `window_id`, `window_opens_at`/`window_closes_at`, `tier`, `approval_mode ∈ {auto, ask}`, `decision_id`, `evaluated_at`, `policy_version` (git commit hash), `valid_until`, `reason[]`. The verdict is deterministic: `not in_window → deny`; `in_window ∧ auto → permit`; `in_window ∧ ask → ask` (timing permitted but the tier forces the Board's operator-approval gate). `ask` is *not* an approval.

**Fail-closed semantics (Critical-infra).** XACML 3.0 gives the citable pattern: a PDP returns `Permit`, `Deny`, `NotApplicable`, or `Indeterminate` (error), and a **deny-biased PEP (§7.2.2)** "treats both Indeterminate and NotApplicable as access denial" ([XACML 3.0 core spec §7.2.2, §7.17](https://docs.oasis-open.org/xacml/3.0/xacml-3.0-core-spec-os-en.html)). The Gateway MUST be deny-biased:

| Condition | XACML analogue | CMDB/Gateway result |
|---|---|---|
| Unknown `host_id` | NotApplicable | `deny` (`no_such_host`) |
| Host known, **no policy on file** | NotApplicable | `deny` (`no_policy`) |
| **CMDB unreachable** | (no response) | Gateway denies — absence of affirmative `permit` = deny |
| Unknown/malformed `action_class` | Indeterminate | `deny` (`bad_action_class`) |
| Clock skew / boundary ambiguity | — | `deny` — boundaries closed; single authoritative clock |
| Overlapping/ambiguous windows | Indeterminate | `deny` (most-restrictive-wins) |

The consequence: the Gateway needs a **positive, fresh `permit`**; anything else — including inability to reach the CMDB — is a deny. This is why the CMDB need not be highly available to be *safe* (its downtime fails closed), though it gates throughput.

> **Load-bearing — the CMDB verdict is a veto, not a trigger.** A `permit`/`in_window` response is **necessary but not sufficient**. On `permit`, the Gateway still cannot act unless *independently* (1) the Board holds an approved ticket in the right state, (2) the Vault releases the host's credentials **to the Gateway only**, and (3) the Gateway holds the per-host mutex. A **fully-compromised CMDB returning `permit` for every host and action still causes zero destructive actions** — it holds no ticket, no creds, no execution path. Conversely a CMDB `deny` is a **hard veto** no other holder overrides. This asymmetry — can veto, cannot trigger — *is* the SoD property for the policy holder (invoice-approver-can't-cut-the-check).

> **Load-bearing — the CMDB holds no lock.** Resource-level locking (ARCHITECTURE §5) is real but is **not** the CMDB's — the per-host mutex belongs to the **Board's atomic claim** and/or the **Gateway's execution mutex**. Putting the lock in the CMDB would make the PDP stateful and couple policy evaluation to execution serialization. Keep the CMDB a stateless read PDP; pin mutex ownership in Planning so it is neither double-owned nor dropped.

> **Correction — the verdict is point-in-time; do not let the Gateway cache `in_window`.** Policy *definitions* (tier, window schedule, mode) are slow-changing and may be replicated; the *verdict* — `in_window` NOW — is time-sensitive and MUST NOT be cached across the action. A window open at 02:59 is closed at 03:01. Evaluate at the **moment of action** against the CMDB's authoritative clock; carry `valid_until = window_closes_at` (capped small); refuse to *begin* any action that cannot *finish* before close. Re-query, don't remember (mirrors the RFC 7662 introspection rule `auth` adopted: for destructive paths, do not cache the decision).

**Where authoritative policy lives.** Separate *inventory* from *policy*: **inventory is a synced index** (Wazuh-discovered, DB-shaped); **policy IS operator-authored knowledge → git-backed markdown** (tier, windows, mode, keyed by the stable host id; `policy_version` in every verdict is the commit hash, so any decision is replayable); **the query DB is a rebuildable JOIN** of the two, with reconciliation edge cases failing closed (host in Wazuh with no policy → `no_policy → deny`; policy for a host absent from Wazuh → flag orphan).

The adversarial blast-radius review accepted the veto-not-trigger framing and fail-closed matrix but proved the SoD property is **not airtight as written**; the hardening is folded in and the recommendation adjusted.

> **Adversarial (highest severity) — the "compromised CMDB causes zero actions" walkthrough is NOT airtight for auto-approve tiers.** ARCHITECTURE §3 delegates approval "by operator OR by CMDB tier policy," so a CMDB that reports `approval_mode=auto` (via compromise *or* plain misconfig) removes the human gate; combined with agents being able to create and claim their own Board tickets, the surviving holders collapse to Vault(creds)+Gateway(executor), both acting on the compromised signal — an agent-unilateral destructive path with no human in the loop. The claim "a compromised CMDB returning permit for everything causes zero destructive actions" is **false whenever any destructive/irreversible class is auto-approvable**. Fix: make destructive/irreversible classes (reboot, wipe, anything on tier-0/NAS) **structurally non-auto via a schema rule the operator cannot override**, and hold that floor at the Board/Gateway independently of the CMDB.

> **Adversarial — verdict integrity is unspecified.** The whole fail-closed property rests on "absence of an affirmative permit = deny," but the verdict is never required to be integrity-protected. An unsigned HTTP-200 permit is forgeable by a cache, proxy error page, MITM, or a Gateway `try/except: log; continue` fail-open. The CMDB cannot enforce its own veto — it depends entirely on the Gateway being a disciplined deny-biased PEP. Close it with a **signed, audience-bound, short-lived verdict token carrying `decision_id`/nonce and hard expiry that the Gateway MUST cryptographically validate**, so "no verdict" is unforgeable-absence.

> **Correction — hard-deny the no-policy and unknown-host cases; delete "at most ask."** The matrix's "no policy → deny **or at most ask**" is a fail-open crack: `ask` means "policy permits the timing," but a no-policy host has no policy to vouch for the timing, so an `ask` + Board approval + creds would execute on a host the CMDB knows nothing about. It also contradicts §6 (newly-synced host defaults to not-actionable). **No-policy = deny, uniformly.**

> **Adversarial — "stateless PDP" vs complete-audit tension.** Critical-infra Stage 5/7 requires a complete per-command audit including the decision log; if the CMDB persists nothing there is no CMDB-side record. Scope "stateless" to the decision **logic**; every issued verdict must still be appended to an **append-only audit log** (a deliberate side effect), ideally mirrored at both CMDB and Gateway.

> **Adversarial — the authoritative clock is a single point of trust.** A wholesale CMDB clock skew (NTP drift or attack) opens windows that should be closed. Require a **trusted/monotonic time source with drift monitoring**; and note `valid_until` compared against the *Gateway's* clock quietly reintroduces cross-clock skew — treat it as a conservative deadline the Gateway re-queries against, never a cross-clock computation.

> **Adversarial — policy-tightening is not a stop button; and the CMDB can corrupt the human holder.** The CMDB holds no execution and cannot revoke an in-flight action — a post-redemption policy change protects only the *next* decision; only the Gateway/auth kill switch stops an in-flight one. Separately, the CMDB is the operator's *only* window into policy (fleet UI, dry-run, even the git-history screen are CMDB-rendered), so a compromised CMDB can **lie to the operator** and corrupt their Board-approval decision. The git trail is ground truth only if verified **out-of-band** (operator/external checker reading `git log` directly).

> **Adversarial — defense-in-depth on the write path; and break-glass needs elevated controls.** The "no agent mutates policy" guarantee leans on an *assumed* auth SSD property; the `apply_policy_change` endpoint still exists behind the proxy. The CMDB write endpoint must **reject any agent-class principal by construction (principal-class check), not merely test for absence of a scope**. Break-glass (emergency window / tier downgrade) is the single highest-value mutation and must carry stronger controls than a routine edit: re-auth, schema-enforced time-boxed auto-expiry, and a distinct audit channel.

**Recommendation (adjusted for the adversarial verdict — ACCEPT-WITH-CHANGES).** Model the CMDB as a stateless-*logic*, side-effect-*audited* **PDP** the Gateway (a **deny-biased PEP**) queries at the moment of action with `(host_id, action_class, now)`, returning one tri-state **`verdict ∈ {deny, ask, permit}`** plus evidence — the AuthZEN shape trimmed of subject. Enforce the fail-closed matrix with **no-policy and unknown-host as hard `deny` (delete "at most ask")**. Make destructive/irreversible classes **structurally non-auto** via a schema rule held at the Board/Gateway independently of the CMDB, so a compromised or misconfigured CMDB cannot remove the human gate — and restate the Stage-5/7 proof honestly: a compromised CMDB cannot *trigger* an action, but it *can* remove the timing veto and (absent the auto floor) the human gate, so the airtight property holds **only** once destructive classes are non-auto and verdicts are signed. Ship **integrity-protected, audience-bound, short-lived signed verdicts** the Gateway cryptographically validates. Keep the verdict **point-in-time and non-cacheable across the action** (re-query at redemption; refuse actions that cannot finish before `window_closes_at`); replicate only slow-changing definitions. Persist an **append-only decision log** at CMDB and Gateway. Use a **trusted/monotonic clock with drift monitoring**. Put **authoritative policy in git-backed operator-authored markdown**, Wazuh inventory as a synced index, the decision DB as a rebuildable JOIN. Keep the **per-host mutex out of the CMDB**. Make the **write path reject agent-class principals by construction**; give break-glass re-auth + schema-enforced auto-expiry + a distinct audit channel; and state plainly that CMDB policy change protects only the *next* decision and that the git trail must be verifiable out-of-band.

## 6. CMDB agent MCP surface (query policy) + human fleet/policy UI — two views over one state

**Findings.** The CMDB's two surfaces are asymmetric by design: the agent MCP surface is a **read-only projection of policy**, and the human UI is the **only writer**. Both sit over one shared state and — the load-bearing detail — both, plus the Gateway's execution-time check, resolve every "may this host be touched now?" question through **one pure evaluator** in the core service, so all three callers get byte-identical answers. This is the concrete form of "two views, one state": one decision function, thin adapters, neither surface downstream of the other.

*MCP surface shape.* Each app's MCP server is an OAuth 2.1 **Resource Server** validating audience-bound tokens locally against the auth AS's JWKS, advertising its AS via RFC 9728 Protected Resource Metadata, and on a scope failure returning **HTTP 403 with `WWW-Authenticate: Bearer error="insufficient_scope"`** (401 = unauthenticated, 400 = malformed) per RFC 6750 §3.1 as adopted by the MCP authorization spec ([MCP Authorization 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization), [RFC 9728](https://datatracker.ietf.org/doc/html/rfc9728), [RFC 6750 §3.1](https://datatracker.ietf.org/doc/html/rfc6750#section-3.1)). Audience binding (RFC 8707) ensures a CMDB token cannot be replayed at the Gateway ([RFC 8707](https://www.rfc-editor.org/rfc/rfc8707.html)).

> **Load-bearing — pin the spec revision.** As of ~2026-07 the **current stable MCP authorization revision is `2025-11-25`**; the `2026-07-28` revision is still a **Release Candidate** (locked 2026-05-21, publishing 2026-07-28) and MUST NOT be assumed live at build ([MCP 2026-07-28 Release Candidate](https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/)). Its most relevant hardening for the CMDB's single-AS/many-RS topology is **mandatory RFC 9207 `iss` validation** to kill mix-up attacks; adopt it now regardless of which revision ships ([RFC 9207](https://www.rfc-editor.org/rfc/rfc9207.html)).

*The decision query is the whole point.* The centerpiece tool is **`is_actionable_now(host, action_class?, at?)`** — the composite "is this host allowed to be touched right now?" — returning a **structured, explainable verdict**, not a bare boolean: `{ in_window, approval_mode: "auto"|"ask", tier, next_window_open, next_window_close, evaluated_at, timezone, reasons: [...] }`. `reasons[]` + `evaluated_at` serve three masters: agents write grounded notes (recon principle), the Gateway gets a machine-checkable verdict, the operator's dry-run screen shows *why* a host was refused. The `at?` parameter makes the same tool a zero-mutation planning probe.

> **Correction — CMDB reports approval *mode*, not approval *state*.** `approval_mode` says the tier *auto-approves* or *forces operator approval*; it must NOT claim the host *is approved* — that is the **Board's** state. The Gateway combines CMDB(policy: in-window + mode) with Board(approval granted?) with Vault(creds) — four independent holders. If the CMDB ever answered "approved," it would collapse two of four holders and break SoD.

> **Load-bearing — the verdict must be re-evaluated at execution time, never cached from planning.** An agent's planning-time `is_actionable_now(..., at=future)` is advisory scheduling; the **binding** check is the Gateway's own call at the instant of execution, because a window can close between plan and execute. The Gateway MUST re-query and MUST NOT trust an agent-relayed verdict.

*The read/write boundary is security-critical.* The agent surface exposes **zero** mutation verbs — no `set_tier`, `widen_window`, `set_approval_mode`, or `add_host` exists on the agent-scoped server. This is not merely a scope check but a **structural absence**, backed by the auth-layer invariant that **no agent role is ever mintable a `cmdb:write-policy` scope**. If an agent could widen its own window or lower a tier from "ask" to "auto," it would **unlock its own destructive action**. Policy mutation is an operator action only.

**Recommended agent MCP tool list** (all read-only, side-effect-free, deterministic; scope `cmdb:read`, audience `cmdb`):

| Tool | Returns | Purpose |
|---|---|---|
| **`is_actionable_now(host, action_class?, at?)`** | structured verdict | **THE decision query** — the Gateway's binding call + the recon/planning probe |
| `is_in_window(host, at?)` | `{in_window, next_open, next_close, timezone}` | narrow window check; `at?` enables scheduling |
| `get_tier(host)` | `{tier, approval_mode, description}` | criticality tier + what it implies |
| `get_maintenance_windows(host)` | recurring rules + one-off exceptions/blackouts | so planning can schedule batches |
| `get_host_policy(host)` | full policy object | one-shot fetch of a host's whole policy |
| `list_fleet(filter?)` | host summaries | recon fleet enumeration; filter by tier/tag/window-state |
| `get_host(host)` | inventory/identity facts | inventory lookup, distinct from policy |

*Human UI.* Mature open-source CMDBs converge on a **list → detail → custom-fields → change-log → journal** information architecture. NetBox records every create/update/delete as pre/post-change JSON snapshots plus who/when and a request-UUID, surfaced as a global Change Log and inline, exported at `/api/extras/object-changes/` ([NetBox Change Logging](https://netboxlabs.com/docs/netbox/features/change-logging/)); it separately offers **Journaling** — persistent human notes with info/success/warning/danger kinds ([NetBox Journaling](https://netboxlabs.com/docs/netbox/features/journaling/)). iTop models the fleet as CIs with typed relationships and customizable fields ([iTop Config-Mgmt datamodel](https://www.itophub.io/wiki/page?id=latest%3Adatamodel%3Aitop-config-mgmt)). Borrow the *shape*, not the heft.

**Recommended operator screens:** (1) **Fleet inventory list** — one table of ~20 hosts (hostname, tier badge, live window-state, mode, Wazuh agent status, tags); (2) **Host detail / policy editor** — Wazuh-synced identity facts + the only place policy is written; (3) **Tier catalog** — define the reusable tiers; (4) **Sync from Wazuh** — reconcile inventory, flag drift, **populate inventory only, never policy**; (5) **Policy-change history** — per-host/global diff rendered from the git trail; (6) **Verdict dry-run / "explain this verdict"** — runs the same evaluator so operator trust is grounded in the real code path.

> **Load-bearing — split markdown-truth from DB-index along the policy/telemetry seam.** **POLICY** (tier, window, mode) is human-authored, low-churn, and security-load-bearing — markdown-as-truth + git makes it diffable, reversible, blame-able, and yields the change-history screen for free. **INVENTORY telemetry synced from Wazuh** is machine-generated, high-churn, re-derivable — its canonical source is **Wazuh**, in the rebuildable SQLite index. So: policy = markdown+git canonical; synced inventory = index; the SQLite query index over both is a pure rebuildable projection.

The adversarial blast-radius review (shared with §5) accepted the read-only-surface and shared-evaluator design but flagged two default-state cracks and the write-path/identity seams; folded in below.

> **Adversarial — the new-host default must be a synthetic always-deny sentinel, not an editable tier pointer.** "Most-restrictive default (highest tier + no window)" described as a *tier the host references* is unsafe: if the default-for-new-hosts is a pointer into the editable tier catalog, an operator edit or bug can point it at an `auto` tier, silently making every newly-discovered host actionable. The new-host state must be a **synthetic `unpolicied`/always-deny sentinel that is not a real tier and cannot be edited into an actionable one**. (Reconcile with §5: no-policy = deny, uniformly.)

> **Adversarial — sync populates inventory only, and identity binding is operator-confirmed.** Screen #4 must never write policy; a newly-discovered host lands at the always-deny sentinel until the operator authors policy. The (Wazuh agent ↔ policy host) match/rebind is an **operator-confirmed, audited event**, never an automatic name/IP fuzzy match (see §2).

> **Adversarial — attribution is only as good as the committer identity.** If the operator UI commits as a single service account, git author = "cmdb-bot" and the real actor is known only via the app's session log — the audit story becomes app-log-dependent, not git-native as advertised. Record **operator identity + auth session id in every commit** (author or trailer, à la NetBox's request-UUID). With one operator today this is moot but must be designed in.

> **Correction — the write path enforces principal class structurally.** `apply_policy_change(operator, host, change) -> commit` authz-checks the operator identity, writes markdown + git commit, then rebuilds the index; it must **reject any agent-class principal by construction**, not merely rely on absence of a write scope, so the guarantee does not rest solely on auth's SSD config. Every read hits the rebuildable index, never git directly.

**Recommendation (adjusted for the adversarial verdict — ACCEPT-WITH-CHANGES).** Ship the seven-tool read-only agent surface with `is_actionable_now` as the explainable-verdict centerpiece, all backed by one pure `evaluate()` shared verbatim with the Gateway's binding execution-time call and the operator dry-run screen — that shared evaluator *is* the two-views-one-state guarantee. Give agents only `cmdb:read` (audience `cmdb`), never any policy-write scope, and **omit every mutation verb from the agent server entirely**; make the write path **reject agent-class principals by construction** and record operator identity + session id in every commit. Make policy markdown-canonical + git-backed and Wazuh-synced inventory a rebuildable index; **default newly-synced hosts to a synthetic always-deny sentinel** (not an editable tier) until the operator authors policy, and keep **sync inventory-only**. Build the six operator screens right-sized to one operator + ~20 hosts, borrowing NetBox's change-log/journal and iTop's CI/custom-field patterns without the enterprise weight. **Pin the MCP authorization revision to `2025-11-25`** at build (with `2026-07-28`/RFC 9207 `iss` validation as the imminent upgrade), and require the Gateway to **re-query the verdict at execution time** rather than trust any agent-relayed answer.

## Resolution of the deferred decisions

**(a) Adopt an existing CMDB vs build a bespoke policy registry → BUILD (accept-with-changes).** Do not adopt NetBox, iTop, or Ralph as the policy brain. None models the fail-closed "may this host be touched right now" predicate natively; wrapping one makes its own DB the de-facto source of truth, violating "markdown is the source of truth" and "two views, one state." The correct justification is **canonicality and SoD-independence, not audit** — NetBox demonstrably ships change logging + journaling, so the honest cost of adoption is *git-canonicality* and coupling a Critical-infra policy holder to a heavy general-purpose codebase and its CVE cadence, not "no audit." The NetBox-plugin hybrid (`may_touch` exposed as a custom script over NetBox inventory + changelog) is rejected on those demonstrated footprint/coupling grounds, not on the false "logic must live in the Gateway" claim. Inventory is a **superset** (Wazuh `GET /agents` ∪ operator-authored non-agent assets), and the build must be right-sized honestly (an RRULE evaluator, markdown/git handling, MCP server, and operator UI, all Critical-infra-hardened) — homelab scale still favors bespoke.

**(b) Policy-schema shape → a (criticality-tier × action-class) matrix over four operator-owned NCSC-control-strength tiers, plus RFC 5545 RRULE+TZID+explicit-end maintenance/freeze windows with deny-overrides precedence (accept-with-changes).** Approval is not a scalar per-host flag because reversibility is a property of the *action*. Six-value action-class enum; tier supplies the default row; per-host overrides; resolution fails safe to `ask`, and `auto` is valid only for a vetted catalog playbook that declares a rollback path. **The canonical-vs-index seam is settled as: policy (tier/window/mode/catalog) = git-backed markdown canonical; Wazuh-synced inventory = rebuildable index; the decision store must not answer the gate from a stale async index — resolve `in_window` live** (this closes the fail-open crack the adversarial pass found and is the explicit argument for which side of the markdown invariant policy belongs on). Windows are stored as local wall-clock Form #3 with the **END as a second local-time anchor (not an exact `PT2H`)** and DST fold/gap closed for START, END, and the query instant, failing closed *observably*. Two structural hardenings are mandatory before the schema is considered locked: **`action_class` is Gateway-derived from the vetted playbook (never agent-supplied)**, and **destructive/irreversible classes are structurally non-auto**, held at the Board/Gateway independently of the CMDB — because tier auto-approve otherwise merges the Board-approval and CMDB-policy holders into one prior operator act (three runtime checks, not four).

## Open design questions to settle before Planning

> **Planning is BLOCKED pending auth's locked identity model.** The stable cross-app host-identity key, the "no agent role is ever mintable `cmdb:write-policy`" SSD guarantee, the audience/scoping shape for the CMDB Resource Server, and the Gateway-vs-agent caller authz all depend on decisions owned by the `auth` component. CMDB Planning cannot lock its data model or surfaces until the auth model is fixed.

1. **Canonical stable host identity** across the whole suite (CMDB, Board host-lock, Gateway per-host mutex), given Wazuh agent ids are recyclable — a cross-app decision, and how a discovered Wazuh agent is *matched/rebound* to that id (operator-confirmed, audited).
2. **Source-of-truth granularity & precedence:** one markdown policy file per host, or per-tier defaults + per-host override files, with precedence (host > tier > fleet default) pinned; and where the standard-change **catalog** physically lives (CMDB, Gateway, or shared registry) and who vets it.
3. **Canonical/index coherency for the gate:** read the canonical store directly on the decision path, or make the transactional store canonical with a git markdown mirror — never a stale async index answering `in_window`.
4. **Tier count & override semantics:** four tiers vs three; may a per-host override only *tighten* (auto→ask), or also *loosen* (ask→auto), and what confirmation ceremony gates a loosening?
5. **CMDB/Board boundary for approval:** does the CMDB return a tri-state `{deny, ask, permit}`, or only `{tier, in_window}` with the Board composing the approval requirement — so policy stays a single SoD holder and `auto` never short-circuits the ticket/claim/verify steps or external verification?
6. **Per-host mutex ownership** (Board atomic-claim vs Gateway execution mutex vs both layered) and how the two avoid split-brain — confirmed NOT the CMDB, but ownership must be pinned.
7. **Availability strategy:** the CMDB fails closed (safe) when down but halts all destructive throughput — is an HA/redundant CMDB required, or per-host cached last-known *definitions* (never a cached `in_window` verdict), at homelab scale?
8. **New-host fail-closed surfacing:** does "untouchable pending tiering" surface as a Board escalation/needs-review or purely a CMDB-UI "needs tiering" queue; and how is a host that vanished from Wazuh but still carries policy treated at Gateway decision time?
9. **Window edge policy:** the default when a host has NO window (always-refuse, recommended); one-off retro exceptions modeled as bounded RRULE/EXDATE vs a first-class one-shot freeze record; whether to expose the next N upcoming windows for batch scheduling; and the grace/lead-time default + enforcement point.
10. **Break-glass mechanics & precedence:** the exact audited operator action that overrides a `deny` or a freeze (temporary emergency window vs tier downgrade), the **allow < freeze < break-glass** lattice with a louder confirmation for break-glass-over-freeze, schema-enforced auto-expiry, and how the mandatory post-hoc review ties back to the Board ticket — without becoming an agent-triggerable SoD bypass.
11. **Verdict evidence format:** does `is_actionable_now`/the Gateway RPC return a signed/attestable verdict token the Gateway logs as evidence, or does the Gateway re-derive locally — decide the audit-evidence and signing/attestation format and whether the decision must prove *which* CMDB + `policy_version` issued a `permit`.
12. **Retro feedback loop:** do tier/action-class cells need per-cell decay or retro escalation (a host that hung on reboot last cycle auto-escalates its `reboot` cell to `ask`) to close the ARCHITECTURE §7 loop?
13. **Clock integrity:** how the CMDB's authoritative clock is established and monitored (NTP discipline, monotonic source, skew alarms), given clock correctness is load-bearing and skew is a fail-closed condition.
14. **Emergency global freeze:** does the operator need an ad-hoc "touch nothing" switch in the CMDB, distinct from the global kill switch at the Gateway?
