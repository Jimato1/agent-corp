# Stage 1 Research — `gateway` Execution Gateway (Critical-infra)

This document is the Stage-1 research foundation for **`gateway`**, "the hands" — the *only* component in the entire agent-corp suite that executes commands on real hosts, and therefore the chokepoint through which every destructive action must pass. Its whole design assumes it will eventually face a **compromised or runaway agent** and must fail safe anyway: the agent names an approved plan and gets no shell, and no single component or agent-only path can reach execution. This research fixes the load-bearing primitives — how the agent's high-level `execute_approved_plan` call maps to vetted parameterized steps, how patch/reboot/rollback stays safe, how the per-host mutex and the four-holder segregation chain are enforced, how Wazuh confirms remediation externally, how the per-command audit is made tamper-evident, and how the kill switch physically bites here. Every external fact is cited to a real URL; because this is Critical-infra and several load-bearing specs are fast-moving (the MCP revision, ansible-core's templating trust model, Wazuh's post-4.8 vulnerability module, Rundeck/apt version floors), version- and revision-specific items are flagged in a consolidated **Verify at build time** list rather than trusted from memory. Today is **2026-07-01**; note the MCP `2026-07-28` revision is still a Release Candidate as of this date, so the build targets `2025-11-25` semantics.

## Executive summary

- **Brokered execution, no shell.** The agent surface exposes only `execute_approved_plan(ticket_id, host_id)`; the Gateway maps the named plan to a hardcoded, admin-authored playbook and passes only a schema-validated parameter dict. Build the executor on **ansible-runner as an embedded library** — but the playbook-only property lives entirely in the wrapper, because `ansible_runner.run()` is a superset API whose `module`/`module_args` (ad-hoc shell) and `cmdline` (raw CLI) are arbitrary-exec escape hatches the wrapper must never forward. ([ansible-runner python interface](https://docs.ansible.com/projects/runner/en/latest/python_interface/))
- **The real injection surface is Jinja2 templating on the control host, not the module call.** Pre-2.19 ansible-core implicitly templated variables (running on the Gateway itself as the ansible user), a CVE-worthy RCE class; **ansible-core 2.19 inverts the trust model** so only trusted-source strings are templated. Pin ≥2.19 and mark caller inputs `!unsafe`. ([ansible-core 2.19 porting guide](https://docs.ansible.com/projects/ansible/latest/porting_guides/porting_guide_core_2.19.html))
- **apt has NO transactional rollback; dnf does — this asymmetry shapes rollback per-distro.** `dnf history undo/rollback` reverses a transaction, but Red Hat does not support it for kernel/glibc/selinux and it fails if old RPMs left the mirrors; apt/dpkg has no equivalent, so on Debian/Ubuntu a **pre-patch filesystem/block snapshot is the only reliable rollback**. Where no in-band rollback exists, escalate — never blind-retry. ([DNF history — RHEL docs](https://docs.redhat.com/en/documentation/red_hat_enterprise_linux/9/html/managing_software_with_the_dnf_tool/assembly_handling-package-management-history_managing-software-with-the-dnf-tool))
- **The per-host mutex is a correctness/security property, acquired before dispatch.** apt/apt-get fails *immediately* on lock contention by default rather than queuing, and the dpkg lock covers only apt operations — a reboot, service restart, or config edit collides without ever touching it — so serialize at claim time with an application-level lock, not dpkg's. For a single Gateway process, an in-process async mutex keyed by `host_id` backed by a **Postgres advisory lock** (auto-released on connection death → free crash recovery, no TTL) is correct; **TTL locks (Redis/Redlock) are disqualified** for a correctness lock a long patch can outrun. ([PostgreSQL advisory locks](https://www.postgresql.org/docs/current/explicit-locking.html), [Kleppmann — distributed locking](https://martin.kleppmann.com/2016/02/08/how-to-do-distributed-locking.html))
- **The Gateway completes a four-holder Dynamic-SoD chain and trusts the agent for none of it.** In strict, fail-closed order: verify an **approved** ticket bound to this exact (host, plan-hash) against live **Board** state → verify **CMDB** policy permits this host *right now* (tier + maintenance window) → redeem a **Vault** response-wrapped single-use handle only the Gateway can unwrap → acquire the **per-host mutex** and hold it across the run. Any agent-supplied "approved=true" / policy verdict / credential is a textbook confused-deputy attack and must be structurally impossible. ([HashiCorp response-wrapping](https://developer.hashicorp.com/vault/docs/concepts/response-wrapping), [MCP authorization](https://modelcontextprotocol.io/specification/draft/basic/authorization))
- **Done is confirmed by Wazuh, read-only, by document *disappearance*.** The post-4.8 vulnerability module deletes a vuln's document from `wazuh-states-vulnerabilities-*` once solved (there is no persistent "Solved" record there), emitting only a transient `data.vulnerability.status:Solved` alert that does *not* fire for offline/restart/initial-sync changes. Confirm remediation by absence of the (CVE, agent) doc after the host's next Syscollector scan; treat the alert as best-effort corroboration. The connector never executes. ([Wazuh — how vuln detection works](https://documentation.wazuh.com/current/user-manual/capabilities/vulnerability-detection/how-it-works.html), [wazuh/wazuh #24290](https://github.com/wazuh/wazuh/issues/24290))
- **Audit is an append-only SHA-256 hash chain anchored off-box.** A hash chain proves internal consistency only; an attacker with full store access can rebuild a consistent alternate chain, so the chain HEAD MUST be periodically signed and pushed somewhere the container cannot rewrite. Capture via ansible-runner's per-task event stream, store in a segregated-privilege append-only Postgres table (the app role must never be superuser/owner), mirror to a git audit repo, and anchor the signed HEAD to the operator. ([Crosby & Wallach — tamper-evident logging (USENIX '09)](https://static.usenix.org/event/sec09/tech/full_papers/crosby.pdf))
- **The kill switch halts new dispatch without corrupting an in-flight transaction.** Model `execute_approved_plan` as an MCP long-running **Task** (returns a handle immediately; the agent polls); the switch cancels at safe task boundaries via `cancel_callback` and refuses new redemptions rather than SIGKILLing a live dpkg/apt run — so run duration must be bounded so a hung task cannot ignore the switch indefinitely. ([MCP Tasks 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/utilities/tasks))
- **The MCP surface is deliberately thin and audience-bound.** Exactly one write tool with a two-field, `additionalProperties:false` input schema plus read-only status/health tools; absence of any shell/credential tool from `tools/list` *is* the enforcement. The server is an OAuth 2.1 Resource Server that MUST validate the token and that its own canonical URI is in the `aud` claim — a valid token only reaches the door; Board+CMDB+Vault+mutex still gate the run. ([MCP server tools](https://modelcontextprotocol.io/specification/draft/server/tools), [MCP authorization](https://modelcontextprotocol.io/specification/draft/basic/authorization))

## 1. Brokered execution (the agent gets no shell)

**Findings.** The correct primitive is: the agent names a plan id, and the Gateway alone maps it to a vetted, parameterized unit of work with no code path for the agent to supply a command string. **ansible-runner** is the strongest embeddable fit — a stable Python library wrapping Ansible, invoked via `ansible_runner.interface.run(...)` (synchronous) or `run_async(...)`; the named unit is `playbook='site.yml'` resolved under a fixed `private_data_dir/project`, with runtime parameters passed as a schema-validated `extravars={...}` dict. The Gateway's guarantee comes from what it does **not** forward: `run()` also accepts `module=`+`module_args=` (ad-hoc mode, e.g. `module='shell'`) and `cmdline=` (raw Ansible CLI) — the arbitrary-execution escape hatches. A safe broker hardcodes `private_data_dir` and the playbook name (or picks from an allowlist keyed by plan id), passes only the validated `extravars`, and never surfaces `module`/`module_args`/`cmdline` to the caller. ([ansible-runner python interface](https://docs.ansible.com/projects/runner/en/latest/python_interface/), [ansible-runner module docs](https://docs.ansible.com/projects/runner/en/latest/ansible_runner/), [PyPI ansible-runner](https://pypi.org/project/ansible-runner/))

> **Load-bearing gotcha — the injection surface is Jinja2 templating on the CONTROL host, not the module invocation.** Historically ansible-core implicitly treated strings as Jinja templates; missed propagation of the "unsafe" wrapper around untrusted strings caused numerous CVE-worthy RCEs, because templating executes on the control host (the Gateway) as the ansible user. **ansible-core 2.19 inverts the previous trust model: only strings loaded from a trusted source (playbooks/roles/vars files) are eligible to be rendered as templates**; external sources are not. Defensive practice regardless of version: mark caller-supplied values with the YAML `!unsafe` tag, prefer the `command` module over `shell`, apply the `|quote` filter to any variable reaching a shell, and bind `become` (privilege escalation) to the CMDB policy check — not to the plan the agent named. ([ansible-core 2.19 porting guide](https://docs.ansible.com/projects/ansible/latest/porting_guides/porting_guide_core_2.19.html), [Ansible advanced syntax / !unsafe](https://docs.ansible.com/projects/ansible/latest/playbook_guide/playbooks_advanced_syntax.html))

ansible-runner also offers per-run job isolation (`process_isolation=True` with `process_isolation_executable`, `container_image`, `container_volume_mounts`) giving per-run filesystem isolation and teardown. *(The specific isolation internals — new mount namespace rooted on a tmpfs invisible to the host, auto-clean on last-process exit, `ansible_runner_<job id>` container naming — and the `podman` default were **not** confirmed against the fetched docs; see Verify-at-build.)* ([ansible-runner python interface](https://docs.ansible.com/projects/runner/en/latest/python_interface/))

Two alternative brokers are good design references but each stands up a second server (extra container/attack surface against the single-container requirement):
- **Rundeck** enforces "named jobs only, no ad-hoc" purely through ACL policy: distinct `job` and `adhoc` resource types; grant `[read,run]` on `job`, never grant `run` on `adhoc`; deny rules evaluate first. Its residual risk is job **option** values flowing into shell steps — an advisory (affecting Rundeck 3.4.1–5.19.0, **fixed in 5.20.0**) where quoting was applied *after* `${option.name}` substitution let metacharacters bypass escaping; mitigate with regex/enum-validated options and `$RD_OPTION_NAME` (escaped) over inline `${option.name}`. *(This is a vendor security advisory that does not carry a formal CVE-YYYY-NNNNN id; the 5.20.0 release notes date it **April 2, 2026**, not the advisory's imprecise "March 2026".)* ([Rundeck authorization](https://docs.rundeck.com/docs/administration/security/authorization.html), [Rundeck option-escaping advisory](https://docs.rundeck.com/docs/history/cves/2025-07-option-escaping.html), [Rundeck 5.20.0 release notes](https://docs.rundeck.com/docs/history/5_x/version-5.20.0.html))
- **salt-ssh** gives agentless parameterized execution needing only sshd on the target; expose only `state.sls`/`state.apply` of allowlisted SLS and the Reactor's fixed event→SLS mapping, and never expose `cmd.run` or `salt-ssh -r` (raw), both of which run arbitrary shell. ([salt-ssh](https://docs.saltproject.io/salt/user-guide/en/latest/topics/salt-ssh.html), [Salt Reactor](https://docs.saltproject.io/en/latest/topics/reactor/index.html), [Salt security](https://docs.saltproject.io/salt/user-guide/en/latest/topics/security.html))

**Host-side last line of defense:** SSH restricted keys guarantee a stolen Gateway key still cannot get a shell. In `authorized_keys`, options precede the key: `restrict,command="/usr/local/bin/gw-exec.sh" ssh-ed25519 AAAA...`. `command=` forces that program regardless of the client's request (exposed as `$SSH_ORIGINAL_COMMAND` but not executed); `restrict` disables pty, agent/X11/TCP forwarding and user-rc. Critical caveat: a wrapper that passes `$SSH_ORIGINAL_COMMAND` to a shell re-opens arbitrary execution — it must ignore or strictly allowlist it. *(The load-bearing OpenSSH facts should be cited to the authoritative sshd(8) AUTHORIZED_KEYS FILE FORMAT man page, not only the secondary tutorial used in raw research.)* Note also that a strict forced command that ignores `$SSH_ORIGINAL_COMMAND` is **incompatible with Ansible**, which ships its own Python modules and invokes them with varying command lines — see the deferred-decision section. ([restrict authorized_keys (secondary)](https://www.simplified.guide/ssh/restrict-authorized-keys-command), [OpenSSH sshd(8) AUTHORIZED_KEYS FILE FORMAT](https://man.openbsd.org/sshd#AUTHORIZED_KEYS_FILE_FORMAT), [Fabric #358 — pty vs no-pty](https://github.com/fabric/fabric/issues/358))

**Recommendation.** Build the executor on **ansible-runner as an embedded Python library**, layering SSH restricted keys underneath as host-side defense-in-depth. The MCP surface exposes only `execute_approved_plan(ticket_id, host_id)`; internally the Gateway maps the approved plan id to a hardcoded admin-authored playbook in a fixed `private_data_dir` and calls `run(playbook=<mapped>, extravars=<schema-validated>, private_data_dir=<fixed>, process_isolation=True)`, never forwarding `module`/`module_args`/`cmdline`. Treat every extravar as untrusted (per-plan schema/enum/regex allowlist), pin ansible-core ≥2.19 and/or mark inputs `!unsafe`, prefer `command` over `shell`, and `|quote` anything reaching a shell. Rundeck and salt-ssh are design references, not adopted components, because they require a standing server against the single-container Critical-infra requirement.

## 2. Exec-safety (patch / reboot / health / rollback / unreachable-escalate)

**Findings.** Encode two distro-specific, parameterized patch playbooks — no free-form shell to agents.

*Debian/Ubuntu:*
- Use **`apt-get`, never `apt`**, for scripted patching — apt(8)'s own man page states its CLI "may change behavior between versions" and directs script authors to `apt-get`/`apt-cache` for backward compatibility and stable exit codes. ([apt(8) man page](https://manpages.debian.org/bookworm/apt/apt.8.en.html))
- Canonical non-interactive invocation: `DEBIAN_FRONTEND=noninteractive apt-get -y -o Dpkg::Options::="--force-confdef" -o Dpkg::Options::="--force-confold" --only-upgrade upgrade`, scoped to the security pocket / CMDB allowlist and honoring `apt-mark hold`. *(Precision correction: `DEBIAN_FRONTEND=noninteractive` does not typically "hang" on a dpkg conffile prompt — a non-interactive frontend auto-selects a default. The reason to also pass `--force-confdef`/`--force-confold` is to make that choice explicit and **deterministic**, not to prevent a stall. Cite the option semantics to the **dpkg(1) man page**, the authoritative source, not the debconf/apt pages.)* ([dpkg(1) man page](https://man7.org/linux/man-pages/man8/dpkg.1.html))
- `unattended-upgrades` is the distro-blessed security-only mechanism (allowlist `Allowed-Origins`, `Package-Blacklist` regex, reboot control), but it is a scheduled daemon; a Gateway that executes on demand and captures per-command output will drive `apt-get` directly while borrowing its allowlist semantics. *(Its default `Allowed-Origins` is broader than the `-security` pocket — it also includes `${distro_id}:${distro_codename}` and ESM origins; state the full default when borrowing it.)* ([Ubuntu automatic updates](https://ubuntu.com/server/docs/how-to/software/automatic-updates/))
- Reboot detection via the `/var/run/reboot-required` marker. *(The marker is created by `notify-reboot-required` from `update-notifier-common`, default on **Ubuntu** but **not** on bare Debian — so the Gateway must not assume it exists on Debian; consider `needrestart` as a cross-distro fallback. This attribution is **not** supported by the linux-audit page cited in raw research; re-cite to the Debian package pages at build.)*

*RHEL/Fedora/Rocky/Oracle:*
- Prefer **`dnf upgrade-minimal --security -y`** (smallest version bump that fixes the issue → lowest blast radius); enumerate first with `dnf updateinfo list updates security`; scope further with `--cve`/`--advisory`. ([dnf(8) man page](https://www.man7.org/linux/man-pages/man8/dnf.8.html), [Fedora — security-only dnf updates](https://fedoramagazine.org/how-to-install-only-security-and-bugfixes-updates-with-dnf/))
- Reboot detection via `dnf needs-restarting -r` (exit 1 = reboot advised, 0 = not); `-s` lists services to restart without reboot. ([dnf needs-restarting](https://dnf-plugins-core.readthedocs.io/en/latest/needs_restarting.html))

**Rollback asymmetry (load-bearing):** `dnf` has real transactional rollback (`dnf history undo <id>` / `rollback <id>` off `/var/lib/dnf/history.sqlite`); **apt/dpkg has nothing equivalent** — the only app-level "undo" is manually pinning a prior version (`apt-get install pkg=oldversion`), which is not transactional and needs the old `.deb` still cached/mirrored. So on Debian/Ubuntu the **only reliable rollback is a pre-patch filesystem/block snapshot** (Btrfs/LVM/ZFS, or Timeshift via a dpkg pre-invoke hook). And even on RHEL, **Red Hat does not support** `dnf history undo/rollback` for kernel, glibc, or selinux/selinux-policy-* (and it fails if old RPMs left the enabled repos) — treat it as best-effort for leaf packages and fall back to a snapshot for those core packages. *(Cite the DNF "Handling package management history" chapter — reachable and authoritative — rather than the 403'd security-updates chapter or the secondary blogs used in raw research.)* ([DNF history — RHEL docs](https://docs.redhat.com/en/documentation/red_hat_enterprise_linux/9/html/managing_software_with_the_dnf_tool/assembly_handling-package-management-history_managing-software-with-the-dnf-tool))

**Post-action health check (fixed sequence):** bounded wait-for-SSH → `systemctl is-system-running` is `running` (not `degraded`) → `systemctl --failed` empty AND the specific critical services `is-active` → reboot marker cleared (Debian `/var/run/reboot-required` gone; RHEL `dnf needs-restarting -r` exits 0).

> **Do NOT judge health from `systemctl status` exit code** — there is a known systemd behavior where `status` can return 0 on a degraded system. Use `systemctl is-system-running` (whose exit code reflects state: `running`⇒0, `degraded`⇒non-zero) with `--wait`, and enumerate `systemctl --failed`. ([systemctl(1) man page](https://manpages.debian.org/testing/systemd/systemctl.1.en.html), [systemd #38042](https://github.com/systemd/systemd/issues/38042))

**Rollback-or-escalate & unreachable-host rules:** on a failed health check, attempt **exactly one bounded rollback** (snapshot restore or `dnf history undo`); if no rollback path exists or it also fails, file `needs_review` — never blind-retry the patch. Reboot waits must be **bounded**: poll SSH with a hard deadline; a host that does not return within the deadline is an immediate `needs_review` escalation (a non-returning host may only be recoverable via out-of-band snapshot/console — a human decision), matching the CLAUDE.md / ARCHITECTURE.md mandates. The Gateway stops at "commands ran + host healthy"; it must **not** self-declare the vulnerability fixed — that is Wazuh's independent confirmation (§5). ([oneuptime — dnf history rollback](https://oneuptime.com/blog/post/2026-03-04-rollback-package-updates-dnf-history-rhel-9/view), [Btrfs snapshot rollback](https://oneuptime.com/blog/post/2026-03-02-roll-back-system-changes-btrfs-snapshots-ubuntu/view))

**Recommendation.** Ship two parameterized playbooks. Debian/Ubuntu: **snapshot-before-patch is mandatory** (no transactional rollback), then `apt-get` (never `apt`) with `DEBIAN_FRONTEND=noninteractive` + `--force-confdef`/`--force-confold` for deterministic conffile handling, scoped to the security allowlist and honoring holds; detect reboot via `/var/run/reboot-required` (don't assume it on bare Debian). RHEL family: `dnf upgrade-minimal --security -y`, enumerate first, reboot via `needs-restarting -r`, `dnf history undo` for leaf packages but snapshot for kernel/glibc/selinux. After any patch/reboot run the fixed health sequence (`is-system-running` ≠ degraded, never `status` exit code). On failure: exactly one bounded rollback then escalate, or escalate immediately if no rollback path; on no-SSH-return within deadline, immediate `needs_review`. CMDB records per-host snapshot capability; hosts with no in-band rollback route to ask-tier/manual.

## 3. Per-host mutex (the real-world resource lock)

**Findings.** dpkg/apt use multiple filesystem locks — the modern top-level one is `/var/lib/dpkg/lock-frontend` (held for the whole apt invocation), plus `/var/lib/dpkg/lock`, `/var/lib/apt/lists/lock`, `/var/cache/apt/archives/lock`. Contention fails with an error naming the holding PID; deleting lock files is discouraged (can break the package DB). ([sinjakli — apt locks](https://blog.sinjakli.co.uk/2021/10/25/waiting-for-apt-locks-without-the-hacky-bash-scripts/))

> **apt/apt-get FAILS IMMEDIATELY on lock contention by default — it does not queue.** A design that assumes "the second run will just wait for the dpkg lock" is wrong: it errors out, possibly after the Gateway already redeemed credentials and started a step. There is an opt-in wait (`apt-get -o DPkg::Lock::Timeout=<seconds>`; `-1` waits forever), but even then the dpkg lock only covers dpkg/apt operations — a reboot, service restart, non-apt package tool, config edit, or health-check window can collide with an in-flight apt run **without ever touching the dpkg lock**. So the per-host mutex must cover the whole "touching this host" window (strictly broader than the dpkg lock), and must be acquired at **claim time, before dispatch and before any credential redemption**. The dpkg lock is a last-line in-host safety net, not the coordination primitive. ([sinjakli — apt locks](https://blog.sinjakli.co.uk/2021/10/25/waiting-for-apt-locks-without-the-hacky-bash-scripts/), [dev.to — dpkg lock contention](https://dev.to/yulin/resolving-dpkg-lock-contention-in-ubuntu-deployments-5fkk))

**PostgreSQL advisory locks** are a native mutual-exclusion primitive keyed by a 64-bit integer: `pg_try_advisory_lock(key)` is non-blocking and returns boolean (reject the second attempt immediately rather than queue). **Crash-recovery property:** if the Postgres backend connection dies (Gateway crash, network drop, container kill), Postgres releases that session's advisory locks at session end — so a crash-with-lock-held self-heals with **no TTL, no fencing token, no stuck-lock janitor**. Map `host_id`→bigint key and hold the lock on a dedicated connection for the execution's duration. ([PostgreSQL advisory locks](https://www.postgresql.org/docs/current/explicit-locking.html), [PostgreSQL admin functions](https://www.postgresql.org/docs/current/functions-admin.html))

> **Two Postgres gotchas.** (1) Session-level advisory locks break under transaction-pooling (PgBouncer transaction mode): the lock lives on a physical backend, but pooled statements can land on different backends, so you can "hold" a lock on a connection you no longer own and your unlock can miss — hold the lock on a **dedicated, non-pooled, session-lifetime connection**. (2) `pg_advisory_xact_lock` (transaction-level) auto-releases at COMMIT/ROLLBACK — convenient but useless for a multi-minute patch spanning many statements; use the session-level `pg_try_advisory_lock` and hold the connection. ([PostgreSQL admin functions](https://www.postgresql.org/docs/current/functions-admin.html), [rclayton — Postgres advisory locks](https://rclayton.silvrback.com/distributed-locking-with-postgres-advisory-locks))

**Redis/Redlock is disqualified here.** A correct single-node Redis lock is `SET key token NX PX <ttl>` (atomic; never `SETNX`+`EXPIRE` as two commands, and release must be a Lua compare-and-delete, never plain `DEL`), but the TTL is a guess: **if a patch/reboot outruns the TTL, Redis auto-expires the lock and a second execution acquires it — two Gateways then believe they own the host**, the exact double-execution a Critical-infra chokepoint must prevent. Kleppmann's analysis: TTL/lease locks are fine for *efficiency* but unsafe for *correctness* under process pauses, clock jumps, and network delays; the safe fix is a monotonic **fencing token** the resource rejects if stale, and **Redlock cannot generate fencing tokens** (its value is random, not monotonic) — for correctness locks, use a proper consensus system, not Redlock. ([Redis distributed locks](https://redis.io/docs/latest/develop/clients/patterns/distributed-locks/), [Kleppmann — distributed locking](https://martin.kleppmann.com/2016/02/08/how-to-do-distributed-locking.html))

> **The in-process mutex is authoritative ONLY while there is exactly one Gateway process** — which is the segregation-of-duties chokepoint, so that holds by design. But the moment the Gateway is scaled to 2+ replicas (or blue/green during deploy), per-replica memory locks silently allow two host-touches. Document this HA landmine loudly at the mutex, and promote the Postgres advisory lock to the authoritative cross-replica mutex if HA ever arrives (adding a monotonic fencing token, cheaply via a Postgres sequence per host, if defending against process-pause split-brain).

**Recommendation.** For a single Gateway process, use the simplest fail-safe primitive: an **in-process async mutex keyed by `host_id`** acquired non-blocking (`tryAcquire`, reject the second execution), backed by a **Postgres advisory lock** (`pg_try_advisory_lock(hash(host_id))` on a dedicated non-pooled connection) as the durable cross-process second layer *and* the audit/visibility record of which host is locked. Belt-and-suspenders on the host: pass `-o DPkg::Lock::Timeout=<n>` so a stray `unattended-upgrades` makes the Gateway wait rather than fail mid-plan. Acquire the mutex **before** brokering Vault credentials or dispatching; release only after the post-action health check completes (or on escalation). Do NOT use Redis/Redlock — this is a correctness lock, and TTL/clock/fencing hazards buy nothing over in-process + Postgres for a single Gateway. If HA ever arrives, promote the Postgres lock and add fencing tokens; a consensus system (etcd/ZooKeeper) is the Kleppmann-endorsed path if you outgrow Postgres.

## 4. Segregation-of-duties chain

**Findings.** Separation of duties is a formally standardized RBAC control (NIST RBAC, adopted as **ANSI/INCITS 359**) with two variants: **Static SoD** (enforced at role-assignment time) and **Dynamic SoD** (constrains which capabilities may be *activated together at a single point in time*). The Gateway's rule — Board-approval, CMDB-policy, Vault-credential, and mutex must **all** hold at the instant of execution — is a **DSD-style constraint** checked at execute time; no single actor (never an agent) may simultaneously assemble enough capability for a destructive action. This is the invoice-approver-can't-cut-the-check control: four checks, each **independently sourced from its owning service** — do not let any one service (or the agent) vouch for another's piece. ([NIST RBAC FAQ](https://csrc.nist.gov/projects/role-based-access-control/faqs), [ANSI INCITS 359 spec](https://profsandhu.com/journals/tissec/ANSI+INCITS+359-2004.pdf))

The Gateway is a **Policy Enforcement Point (PEP)** that calls authoritative **Policy Decision Points** at run time and enforces permit/deny — it must not embed or cache the decision. The Board (is-this-(host,plan)-approved) and CMDB (is-this-host-actionable *now*: tier approval satisfied + inside maintenance window) are the PDPs/PIPs it queries at the moment of execution. Because approval and maintenance-window state are time-varying, the decision **MUST be evaluated at execution time against live state**, never trusted from a stale token or an agent-passed flag. *(A synchronous per-execution check is the right design on its own merits; a specific XACML latency figure asserted in raw research was unsupported by any cited source and is dropped.)* ([XACML overview](https://en.wikipedia.org/wiki/XACML), [PEP/PDP/PAP/PIP](https://medium.com/identity-beyond-borders/the-four-horsemen-of-authorization-the-p-ps-pep-pdp-pap-and-pip-42717e445ce7))

**Secret brokering:** the Gateway should redeem a short-TTL, single-use **handle** that resolves to plaintext only for the Gateway — Vault's **response-wrapping / secure-introduction** pattern: Vault inserts the response into the cubbyhole of a single-use token; only a single party can ever unwrap it (a token that fails to unwrap = a malfeasance signal / security incident), the client can inspect origin before unwrapping, and the wrapping token's lifetime is separate from (and shorter than) the wrapped secret. The handle travels through the untrusted path (the agent); only the Gateway unwraps. Prefer **dynamic, short-lived, per-host** credentials with automatic lease expiry/revocation, minting a per-execution credential whose lease is explicitly revoked at run end — the Gateway holds no long-term secrets. ([HashiCorp response-wrapping](https://developer.hashicorp.com/vault/docs/concepts/response-wrapping), [Vault cubbyhole / secure introduction](https://www.hashicorp.com/en/blog/cubbyhole-authentication-principles), [Vault lease concepts](https://developer.hashicorp.com/vault/docs/concepts/lease))

> **The single most important defect to avoid: the Gateway trusting the orchestrating agent's claims.** The agent may name a ticket and host, but must NEVER supply approval status, policy verdicts, tier/window facts, or credentials — that is a textbook **confused-deputy** attack. The OAuth defense is **audience binding**: a resource server MUST validate that a token was issued specifically for it (RFC 9068 `aud`; the MCP spec restates this as a MUST) and MUST NOT pass a client-supplied token through. So the approval must be scoped/audience-bound to **this exact (host_id, plan-hash)** so it cannot be replayed against a different host or a mutated plan; verify plan integrity via a hash so the plan executed equals the plan approved. *(Precision correction: RFC 8707 is SHOULD-level for audience restriction — "SHOULD audience-restrict issued access tokens"; the MUST-level audience-validation rule comes from RFC 9068 and the MCP authorization spec, which are the correct citations for the MUST.)* ([RFC 9068 §4](https://www.rfc-editor.org/rfc/rfc9068.html), [MCP authorization](https://modelcontextprotocol.io/specification/2025-06-18/basic/authorization), [RFC 8707](https://www.rfc-editor.org/rfc/rfc8707.html), [MCP confused-deputy (flowhunt)](https://www.flowhunt.io/blog/mcp-authentication-authorization-oauth-confused-deputy/))

**Ordering, fail-closed:** verify approval (Board) → verify policy-now (CMDB tier+window) → redeem creds (Vault handle) → acquire per-host mutex → only then run. If any earlier check fails or errors (Board/CMDB/Vault unreachable), refuse to execute — never default-allow, never blind-retry; unreachable target host → immediate escalation. The mutex is held last across the entire run so approval/policy cannot expire mid-execution and no two runs touch one host; bound run duration so a run cannot spill outside the window. ([NIST RBAC FAQ](https://csrc.nist.gov/projects/role-based-access-control/faqs), [Vault lease concepts](https://developer.hashicorp.com/vault/docs/concepts/lease))

**Recommendation.** Model the Gateway as a **fail-closed PEP enforcing a Dynamic-SoD constraint at execution time**, in strict order and independently sourcing each fact from its owning service, trusting the agent for none: (1) Board — an approved ticket bound to this exact (host_id, plan-hash), re-checked against live state (optionally also verify a Board-signed, audience-bound approval token as a tamper-evident capability); (2) CMDB — policy permits this host *right now* (tier + window), evaluated live; (3) Vault — redeem a response-wrapped single-use handle, preferably a short-TTL per-host dynamic secret whose lease is revoked at run end, treating a non-unwrappable/already-unwrapped handle as a security incident; (4) acquire and hold the per-host mutex across the run. Only then execute. The agent surface names only a ticket/host and supplies no approval assertions, policy verdicts, or credentials — this must be structurally impossible.

## 5. Wazuh API (external verification, read-only)

**Findings.** Current stable Wazuh is the **4.14.x** line; **as of today (2026-07-01) the latest stable is 4.14.6, released 1 July 2026** *(the raw research said 4.14.5 / 23 Apr 2026 — bump the concrete version)*, with **5.0 still in beta** (not production; brings clustering-by-default, Filebeat removal, and an overhauled RBAC — gate 5.0 support behind a version probe). Build against 4.14.x semantics. ([Wazuh 4.x release notes index](https://documentation.wazuh.com/current/release-notes/index-4x.html), [Wazuh GitHub releases](https://github.com/wazuh/wazuh/releases))

There are **two distinct read APIs**: (a) the **Wazuh Server RESTful API** on **:55000** (JWT: `POST /security/user/authenticate?raw=true` → `Authorization: Bearer`) for agent liveness/metadata and Syscollector inventory; (b) the **Wazuh Indexer API** on **:9200** (OpenSearch `_search`, HTTP basic auth) where vulnerability **state** now lives. **Wazuh 4.8.0 refactored Vulnerability Detection and removed the old Server-API endpoints** — deprecated in 4.7.0, removed in 4.8.0: `PUT /vulnerability`, `GET /vulnerability/{agent_id}`, `GET /vulnerability/{agent_id}/last_scan`, `GET /vulnerability/{agent_id}/summary/{field}` *(these are the doc's actual names; raw research listed them imprecisely)*. Query posture instead via `GET /wazuh-states-vulnerabilities*/_search`. *(The Indexer-API "JWT also supported" claim was not shown by the fetched pages — only HTTP basic was demonstrated; treat JWT-on-indexer as Verify-at-build.)* ([Wazuh Server API getting started](https://documentation.wazuh.com/current/user-manual/api/getting-started.html), [Wazuh Indexer API getting started](https://documentation.wazuh.com/current/user-manual/indexer-api/getting-started.html), [Wazuh 4.8.0 release notes](https://documentation.wazuh.com/current/release-notes/release-4-8-0.html))

Read posture by CVE + agent with a bool filter on `vulnerability.id` + `agent.id`; documented index fields include `vulnerability.id`, `vulnerability.severity`, `vulnerability.score.base`, `vulnerability.detected_at`, `agent.id`/`agent.name`, `package.name`/`package.version`, `host.os.*`. A single query caps at 10,000 hits (`index.max_result_window`, the standard OpenSearch default — confirm on the live cluster); paginate for fleet sweeps. ([Wazuh Indexer API use case](https://documentation.wazuh.com/current/user-manual/indexer-api/use-case.html), [Wazuh indexer indices](https://documentation.wazuh.com/current/user-manual/wazuh-indexer/wazuh-indexer-indices.html))

> **CRITICAL remediation-confirmation semantic — confirm by DISAPPEARANCE, not a status flip.** The post-4.8 states index holds **only active** vulnerabilities and **deletes** the document once a vuln is solved ("once resolved, there is no track of it ever being present"). There is **no persistent "Solved" record** in the states index — the active→solved transition is emitted only as a transient **alert** in `wazuh-alerts-*` with `data.vulnerability.status: Solved`. A connector that polls the states index for `status==Solved` will never see it. Recommended logic: before patch, record the doc exists for (CVE, agent); post-patch, poll and treat **absence** of that (`vulnerability.id`, `agent.id`) doc — after a completed post-patch scan — as remediation, optionally corroborated by a Solved alert. *(Cite the how-it-works page and issue #24290; the FAQ.html URL used in raw research now returns a moved/redirect stub.)* ([Wazuh — how vuln detection works](https://documentation.wazuh.com/current/user-manual/capabilities/vulnerability-detection/how-it-works.html), [wazuh/wazuh #24290](https://github.com/wazuh/wazuh/issues/24290))

> **The Solved alert is best-effort, not a gate.** It fires **only** when the package change is captured during a *scheduled* Syscollector scan; offline changes, agent restarts, and initial inventory syncs update state **without** emitting a Solved alert. A connector that waits for the alert can hang on legitimately-patched hosts — so states-index disappearance (with a confirmed post-change scan) is the PRIMARY signal, the alert only corroboration. This satisfies "done confirmed externally" while avoiding a false "never terminates" failure. ([Wazuh — how vuln detection works](https://documentation.wazuh.com/current/user-manual/capabilities/vulnerability-detection/how-it-works.html))

Scans are inventory-driven off **Syscollector** (agent-side, default ~1h interval plus `scan_on_start`): the vuln module re-evaluates whenever Syscollector reports inventory, so post-patch the doc will not vanish until the host's **next** Syscollector package scan completes and is processed — poll with backoff sized to that interval rather than expecting instant confirmation. ([Wazuh — configuring scans](https://documentation.wazuh.com/current/user-manual/capabilities/vulnerability-detection/configuring-scans.html))

The connector is **strictly read-only** — Wazuh detects and tracks remediation but never patches; execution stays in the Gateway. Enforce at the boundary: scope both API credentials to READ-ONLY via Wazuh RBAC (GET/`_search`, agent-read), never write/execute/index-management, and never expose any Wazuh write path to agents. ([Wazuh Indexer API getting started](https://documentation.wazuh.com/current/user-manual/indexer-api/getting-started.html))

**Recommendation.** Build the read-only connector against **Wazuh 4.14.x** (not 5.0 beta) with two clients: the **Indexer API (:9200)** as the PRIMARY posture source (bool filter `vulnerability.id` + `agent.id` over `wazuh-states-vulnerabilities-*`) and the **Server API (:55000)** for agent liveness/Syscollector metadata. Confirm remediation by **disappearance** of the (CVE, agent) document after the host's next Syscollector scan — not by a status field — treating any `data.vulnerability.status:Solved` alert as best-effort corroboration. Poll with backoff sized to the Syscollector interval; scope both credentials read-only via Wazuh RBAC; probe server version at startup and branch/refuse on 5.x.

## 6. Immutable audit

**Findings.** **ansible-runner already emits a complete per-command/per-task capture** as on-disk artifacts you can hash directly: `artifacts/<id>/` with `job_events/` (one JSON per event: `uuid`, sequential `counter`, `event`, ISO-8601 `created`, `event_data`, and the stdout slice with `start_line`/`end_line`), full `stdout`, `rc`, and `status`. This gives command, output, exit code, and timing per step without a custom capture loop — hash-chain the event stream as it arrives via the `event_handler` callback. *(The per-task `res` sub-field names — rc/stdout/stderr/delta — and the exact `status` literal ("successful" vs "success") are version-ambiguous across the docs; confirm against the pinned release before matching on them — see Verify-at-build.)* ([ansible-runner intro/artifacts](https://docs.ansible.com/projects/runner/en/stable/intro/), [ansible-runner reference](https://ansible.readthedocs.io/projects/runner/en/stable/ansible_runner/))

> If capturing at the SSH layer instead (paramiko, for non-Ansible steps), you must read stdout AND stderr separately and call `recv_exit_status` — but `recv_exit_status` blocks until exit and **deadlocks if you await it before draining both streams** once output exceeds the Transport's default channel window (default `default_window_size` = 2 MiB). Interleave reads of both streams before/while awaiting exit status. This is why the higher-level ansible-runner artifact stream is the safer capture substrate. ([paramiko Channel API](https://docs.paramiko.org/en/stable/api/channel.html), [paramiko #1778](https://github.com/paramiko/paramiko/issues/1778))

**Hash-chaining** (`record_hash = SHA-256(canonical(fields) || previous_hash)`) is the core tamper-evidence primitive — the same construction as git, Certificate Transparency, and blockchains — so any modification/deletion/reordering is detectable by re-walking the chain. ([Crosby & Wallach — tamper-evident logging (USENIX '09)](https://static.usenix.org/event/sec09/tech/full_papers/crosby.pdf))

> **A hash chain proves internal consistency ONLY.** An attacker with full store access can rebuild a self-consistent alternate chain, so without an externally-held/signed chain HEAD you cannot detect wholesale rewrite. **Anchoring the HEAD off-box is mandatory, not optional** — sign the current HEAD on an interval with a forward-secure/off-box key and push it to the operator (mission-control). systemd-journald **Forward Secure Sealing** is the reference model (a sealing key regenerated non-reversibly every interval — default 15 min — with the verification key kept off the machine), but note its limitation: FSS only **detects** tampering, cannot recover altered data, and cannot protect entries written after compromise but before the next seal. *(Cite the systemd journalctl(1)/journald.conf(5) man pages for the 15-min default, not the secondary netdata page.)* ([journalctl(1) man page](https://man7.org/linux/man-pages/man1/journalctl.1.html), [Sigstore Rekor — transparency log](https://docs.sigstore.dev/logging/overview/))

**Append-only substrate options, ranked for one self-hosted container:**
- **Postgres least-privilege append-only:** audit table owned by a role the Gateway app is NOT; app role granted only INSERT+SELECT; REVOKE UPDATE/DELETE/TRUNCATE. **Documented hard limit: in-database auditing CANNOT protect against a superuser or the table owner** — so the Gateway's DB role must never be superuser/owner, or the control is theatre. ([PostgreSQL REVOKE](https://www.postgresql.org/docs/current/sql-revoke.html), [PG audit-trigger wiki](https://wiki.postgresql.org/wiki/Audit_trigger_91plus))
- **Git commit chaining** (aligns with the repo-wide git-backed-audit mandate): every commit hashes tree + parent-commit hash + metadata → a parent-linked hash chain. Two caveats: default object format is SHA-1 (SHAttered produced a real collision; git ships collision-detecting SHA-1DC since 2.13, and `--object-format=sha256` exists but most forges still don't support SHA-256 repos as of early 2026, limiting mirroring); and local history can be rewritten (rebase/force-push) — tamper-evidence holds only if a trusted external party retains the true HEAD. Use signed commits and record HEAD off-box. ([git SHA-256 status (LWN)](https://lwn.net/Articles/898522/))
- **chattr +a** on the raw log volume for cheap local WORM — but a container root can clear it unless applied from the host and the container lacks `CAP_LINUX_IMMUTABLE`. **S3 Object Lock** is over-engineered for a homelab and only **COMPLIANCE** mode (with versioning) is true WORM — GOVERNANCE mode is bypassable via `s3:BypassGovernanceRetention`. ([chattr (secondary)](https://linuxize.com/post/chattr-command-in-linux/), [S3 Object Lock](https://docs.aws.amazon.com/AmazonS3/latest/userguide/object-lock.html))

**Per-command forensic record fields** map to **NIST SP 800-53 Rev 5 AU-3** (what/when/where/source/outcome/identity) plus the Gateway's SoD-proof fields: `record_id` + monotonic `sequence_no`; `ticket_id` + Board approval reference; `host_id` + Gateway instance id; the actor identity that invoked `execute_approved_plan`; the CMDB policy snapshot (tier, in-window) at execution time; the credential **HANDLE** used (never plaintext); mutex acquire/release markers; the exact command/playbook + parameters run; full stdout; full stderr; exit code; start/end timestamps + duration; post-action health result; external-verification linkage (Wazuh check id); and the tamper-evidence trailer `previous_hash` + `record_hash` + signature. The record records outcome as "commands ran + host healthy" — it deliberately does **not** self-declare task "done". *(The specific supporting-content enumeration attributed to AU-3(1) in raw research derives from Rev 4 supplemental guidance, not Rev 5 AU-3(1); cite the official Rev 5 OSCAL/CSRC PDF, and consider AU-9/AU-10 for the signing/non-repudiation requirement.)* ([NIST SP 800-53 Rev 5 AU-3](https://csf.tools/reference/nist-sp-800-53/r5/au/au-3/), [NIST SP 800-53 Rev 5 OSCAL PDF](https://csrc.nist.gov/CSRC/media/Projects/risk-management/800-53%20Downloads/800-53r5/SP_800-53_v5_1-derived-OSCAL.pdf))

**Recommendation.** Use a layered append-only-plus-hash-chain design, not any single mechanism: (1) **capture** via ansible-runner artifacts (fall back to paramiko only for non-Ansible steps, draining stdout+stderr before `recv_exit_status`); (2) **record** each command as an AU-3-complete JSON row (credential HANDLE only, `prev_hash` + `record_hash` + Ed25519 signature) — a SHA-256 hash chain; (3) **store** in a Postgres segregated-privilege append-only table (app role never superuser/owner); (4) **mirror** into a dedicated git audit repo with signed commits (prefer `--object-format=sha256` where GitHub mirroring isn't needed, else rely on SHA-1DC); (5) **anchor** the signed chain HEAD off-box to mission-control on an interval (mandatory — hash chaining proves consistency, only external anchoring proves freshness); (6) optionally set `chattr +a` from the host with the container denied `CAP_LINUX_IMMUTABLE`. Skip S3 Object Lock unless real S3/MinIO is already present, and then only COMPLIANCE mode.

## 7. Kill-switch chokepoint

**Findings.** The kill switch lives on the human surface and must **physically bite at the Gateway chokepoint** — halting the Gateway halts all destructive action suite-wide, regardless of what agents believe. Modeling `execute_approved_plan` as a long-running **MCP Task** makes the switch clean: the call returns a task handle immediately (status `working`) and the agent polls, so the switch can cancel in-flight tasks and refuse new dispatch without holding an RPC open through a multi-minute patch. ansible-runner supports cancellation via a `cancel_callback`, which stops at safe task boundaries rather than SIGKILLing a live process. ([MCP Tasks 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/utilities/tasks))

> **The switch honors cancellation only at task boundaries — and that is correct.** A play already inside the apt/dnf task completes that task (preventing dpkg/apt corruption) rather than being killed mid-transaction; the switch halts **new** dispatch and cancels at the next boundary. The consequence: the switch is not instantaneous, so **run duration must be bounded** so a hung task cannot ignore the switch indefinitely. This is the deliberate tradeoff — halt new work + clean cancellation over a corrupting hard-kill.

The kill switch must also interact cleanly with the per-host mutex and credential leases: halting the Gateway should release/abandon in-flight per-host locks (the Postgres advisory lock frees on connection close — free clean recovery, no stuck lock blocking post-kill recovery) and revoke outstanding Vault leases. Verify the whole halt path in Stage 7 (demonstrate the switch stops an in-flight run). ([PostgreSQL advisory locks](https://www.postgresql.org/docs/current/explicit-locking.html), [Vault lease concepts](https://developer.hashicorp.com/vault/docs/concepts/lease))

**Recommendation.** Wire the kill switch to a **hard server-side halt**: (1) refuse all new `execute_approved_plan` dispatch and new Vault redemptions; (2) cancel in-flight runs at safe task boundaries via ansible-runner's `cancel_callback` (never SIGKILL a live package transaction); (3) bound maximum run duration so a hung task cannot outlast the switch; (4) let Postgres advisory locks free on connection close so no stuck host-lock blocks recovery, and revoke outstanding leases. The kill-switch halt and mutex acquire/release should be first-class chained records in the audit log. Prove the halt stops an in-flight run in Stage 7.

## 8. Thin MCP surface + operator monitor

**Findings.** The current MCP final revision is **`2025-11-25`**; **`2026-07-28` is a Release Candidate as of 2026-07-01** (locked 2026-05-21, ships 2026-07-28) carrying breaking changes (stateless core removing the initialize handshake and `Mcp-Session-Id`; an Extensions framework; Tasks becoming an extension; authorization hardening). Build to `2025-11-25` semantics and keep the RC diffs as a planned migration. ([MCP 2026-07-28 RC](https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/), [MCP 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25))

The thin agent surface is **exactly one write tool** `execute_approved_plan` with a tight input schema — `{"type":"object","properties":{"ticket_id":{"type":"string"},"host_id":{"type":"string"}},"required":["ticket_id","host_id"],"additionalProperties":false}` — plus read-only status/health tools. Because the agent can only invoke tools returned by `tools/list`, **never registering a shell/command/credential tool means there is no MCP path to raw execution** — the agent names an approved plan; the Gateway maps it internally. Servers MUST validate all tool inputs; annotations are untrusted metadata. ([MCP server tools](https://modelcontextprotocol.io/specification/draft/server/tools))

> **The input schema is the injection boundary.** `additionalProperties:false` + `required:[ticket_id,host_id]` stops an agent smuggling extra fields (e.g. an injected command string). But the thin surface only holds if you ALSO never register a lower-level tool — there is no MCP flag that "hides" a registered tool from a scoped agent beyond scope-filtered `tools/list`.

The Gateway MCP server is an **OAuth 2.1 Resource Server**: it MUST validate the bearer token, MUST verify its own canonical URI is in the token `aud`, and MUST publish RFC 9728 Protected Resource Metadata; scope a custom `gateway:execute` enforced both in middleware and inside the tool handler. Use the SDKs' built-in bearer middleware (TS `requireBearerAuth`+`mcpAuthMetadataRouter`; Python FastMCP `AuthSettings`+`TokenVerifier`) — do **not** hand-roll token validation. The platform **`auth`** app is the Authorization Server minting short-lived audience-bound tokens; the Gateway only validates. **A valid token is necessary but NOT sufficient** — the Board/CMDB/Vault/mutex chain still gates the run. ([MCP authorization](https://modelcontextprotocol.io/specification/draft/basic/authorization), [MCP security tutorial](https://modelcontextprotocol.io/docs/tutorials/security/authorization))

`execute_approved_plan` is long-running → model it as an **MCP Task**: return a handle in status `working` immediately; the agent polls `tasks/get` (honoring `pollInterval`) through patch→reboot→Wazuh-verify rather than blocking the RPC. Status enum: `working` / `input_required` / `completed` / `failed` / `cancelled`. ([MCP Tasks 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/utilities/tasks))

> **Task method surface CHANGED between revisions — pin to your SDK's revision.** `2025-11-25` has `tasks/get` + `tasks/result` (blocking) + `tasks/list` + `tasks/cancel`; **`2026-07-28` (SEP-2663) makes Tasks an extension and REMOVES `tasks/result` and `tasks/list`, adding `tasks/update` with server-directed handles and `resultType:"task"`.** Hedge by exposing your own stable `get_execution_status(run_id)` read tool so the agent contract survives the migration. *(Anchor this redesign to the primary source, PR #2663, not the secondary blog used in raw research.)* ([MCP PR #2663](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2663))

> **Task access MUST be auth-context-bound — a security property.** Task IDs are the primary access mechanism, so the Gateway MUST reject `tasks/get`/`tasks/result`/`tasks/cancel` for tasks outside the caller's authorization context and generate high-entropy IDs — otherwise Agent B could poll/cancel Agent A's host run, a cross-host isolation break. *(Note: `tasks/update` is a 2026-07-28-only method; it is not part of the 2025-11-25 reject-set.)* ([MCP Tasks 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/utilities/tasks))

**Operator execution monitor:** the MCP spec supplies the streaming transport (Streamable-HTTP SSE), the task-state model, and audit-logging SHOULDs — **not** the UI itself (that is our design decision, per the two-views-one-state invariant). A good surface: per-host cards keyed by `host_id` (state = task status enum, current run/ticket, mutex holder), live streaming stdout/stderr per command, an immutable append-only per-command audit view as the source of truth (distinct from the ephemeral SSE view), and the kill switch as a hard server-side halt. Prefer a **separate operator-facing SSE/WebSocket over the same audit store**, not the agent's MCP stream. Never log Authorization headers/tokens (no-secrets-in-logs), which matters because the monitor streams logs. ([MCP Tasks 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/utilities/tasks))

> **Two distinct "done" concepts.** The Gateway task's terminal `completed` means "commands ran + host healthy" — it must NOT self-report task success in the product sense; external verification (Wazuh active→solved, §5) is a separate confirmation. Keep the two distinct in the UI and the ticket lifecycle.

**Recommendation.** Build against `2025-11-25` today, structured for a mechanical `2026-07-28` migration. Register exactly one write tool `execute_approved_plan` (`additionalProperties:false`, two required fields) plus read-only `get_execution_status(run_id)` and `get_host_health(host_id)`; never register any shell/command/credential tool. Make the server an OAuth 2.1 Resource Server via SDK bearer middleware — validate the token, verify the Gateway's canonical URI is in `aud`, publish RFC 9728 metadata, require `gateway:execute` in middleware and handler — with `auth` as the Authorization Server and the four-holder chain still gating the run. Model execution as an auth-context-bound MCP Task with a stable `get_execution_status` shim; emit progress notifications. Back the operator monitor with the immutable audit store over a separate human channel, and wire the kill switch to a hard server-side halt proven in Stage 7. Do not conflate the task's `completed` with externally-verified "done".

## Recommendation on the deferred decision: execution mechanism

**Winner — a hybrid: ansible-runner as the embedded, playbook-only executor, hardened by *relaxed* host-side SSH controls (not strict forced-command).**

The Gateway runs `ansible-runner` in-process as the plan engine, and on every host layers `restrict` options **minus** `command=` (i.e. `no-pty`, no agent/X11/port-forwarding, `from=<gateway-ip>`) **plus** a sudoers allowlist limiting the Gateway user to only the vetted privileged patch commands, backed by short-lived Vault-issued SSH certificates. This preserves the real guarantee of the forced-command approach — a stolen Gateway key/cert cannot open a tunnel or sudo anything outside the patch set, and expires quickly — while remaining compatible with how Ansible actually runs.

**Why ansible-runner (option A) wins.** It keeps execution in one self-hosted container (unlike Rundeck/Salt, which need a standing server = extra container and attack surface); it gives idempotent, vetted modules for the whole patch/reboot/health/rollback flow (apt/dnf modules, the reboot module's native bounded reconnect, `block`/`rescue` for rollback-or-escalate); a structured per-task event stream that is **richer audit** than raw command strings; and cancellation at safe task boundaries (`cancel_callback`) that halts new dispatch without SIGKILLing an in-flight dpkg. Its real weakness — that `run()` is a superset API and the true injection channel is control-host Jinja2 templating — is containable via the hardcoded plan-id→signed-playbook allowlist, schema/enum/regex-validated `extravars` only, ansible-core ≥2.19 + `!unsafe`, and per-run process isolation.

**Alternatives rejected:**
- **(B) constrained command templates over SSH (paramiko/Fabric):** exec runs through the target login shell, so safety rests on hand-written `shlex.quote` on every parameter — the exact quoting-after-substitution bug behind the Rundeck 3.4.1–5.19.0 advisory — and forces re-implementing the entire EXEC-SAFETY tree (conffile flags, `upgrade-minimal --security`, distro-branched reboot detection, `is-system-running`, snapshot/rollback) as bespoke security-critical code. Maximum injection surface in the most dangerous logic; loses Ansible's structured audit. Rejected as primary.
- **(C) strict forced-command restricted keys, as the PRIMARY mechanism:** decisive finding — a forced command that ignores `$SSH_ORIGINAL_COMMAND` **cannot serve Ansible's dynamic module shipping**, so C-strict and A cannot coexist; and as a standalone engine it pushes parameterized-plan logic into per-host bash wrappers across ~20 hosts (the worst place for security-critical parameterization) and fragments the central audit trail. Kept only in **relaxed** form as host-side defense-in-depth.
- **Rundeck (job/adhoc ACL split):** good design reference, but a second server, and still had the option-injection advisory fixed only in 5.20.0.
- **salt-ssh (vetted-SLS + Reactor):** valid agentless pattern, but a master/reactor model with its own escape hatches (`cmd.run`, `salt-ssh -r`) to suppress; ansible-runner-as-library stays in-process without a second service.
- **Pure A with no host-side control:** leaves a compromised Gateway process (or a control-host template RCE that escapes isolation) holding a key that can open a shell/tunnel on every host — the relaxed-C layer is required to bound blast radius at the host boundary.

**Red-team attacks answered:**
- *Agent smuggles code via a plan parameter* → schema/enum/regex-validated `extravars` only, ansible-core ≥2.19 inverted trust + `!unsafe`, never forwarding `module`/`module_args`/`cmdline`; the agent names a plan id mapping to a hardcoded admin-signed playbook.
- *Agent claims "approved=true" or passes a credential (confused deputy)* → structurally impossible; the surface is only `execute_approved_plan(ticket_id, host_id)`. Approval is re-verified against live Board state bound to host+plan-hash, CMDB policy/window checked at execute time, Vault plaintext resolves only inside the Gateway via a response-wrapped single-use handle.
- *Gateway key/cert stolen to open a shell/tunnel across the fleet* → relaxed-C: `restrict` (no-pty/no-forwarding) + `from=<gateway-ip>` + sudoers allowlist + short-TTL Vault SSH certs, so a leaked cert can neither tunnel nor sudo outside the vetted commands and expires quickly.
- *Kill switch thrown mid-patch corrupts a dpkg transaction* → halt NEW dispatch + `cancel_callback` at task boundaries (not SIGKILL); the in-flight package task finishes atomically, bounded run duration prevents indefinite ignore.
- *Approval/window revoked before execution, or token replayed against another host* → re-evaluate Board+CMDB at execute time (no cached verdict), bind approval to the exact plan-hash+host, hold the per-host mutex across the run.
- *Strict forced-command assumed to co-exist with Ansible and silently breaks module execution (or is loosened to pipe `$SSH_ORIGINAL_COMMAND` to a shell, reopening arbitrary exec)* → explicitly reject C-strict under A; adopt relaxed-C, which is Ansible-compatible and never pipes `SSH_ORIGINAL_COMMAND` to a shell.

**Residual risks (accepted, with mitigations):**
- The playbook-only property lives **entirely in the wrapper** — `run()` still accepts `module`/`module_args`/`cmdline`. Any future refactor that builds `run()` kwargs from agent-influenced data reopens arbitrary exec. Mitigate with a hardcoded allowlist and unit tests asserting those kwargs are never populated.
- Control-host Jinja2 templating RCE (CVE-2021-3583 class) must be **empirically tested** for extravars passed specifically via ansible-runner under the pinned ansible-core, with and without `!unsafe`.
- `process_isolation` adds operational weight (podman-in-container often wants elevated privileges — ironic for the scariest container; bwrap is lighter but its semantics must be validated against the container base image).
- `become` is orthogonal to no-shell — a vetted playbook can still escalate; bind `become`/tier to the live CMDB check and the Board-approved plan-hash, not the plan name the agent supplies.
- apt has no transactional rollback and `dnf history undo` is unsupported for kernel/glibc/selinux — some hosts have no in-band rollback path and MUST be ask-tier/manual; CMDB records per-host snapshot capability.
- The kill switch honors cancellation only at task boundaries (correct, but not instantaneous) — bound run duration.
- Audit fidelity is per-task/module, not literal shell strings — persist append-only + hash-chained, and reviewers must accept module-level granularity.
- Relaxed-C weakens the strict "can never get a shell" guarantee (the ansible user CAN run commands, just not tunnel/sudo outside the allowlist) — the sudoers allowlist becomes the real host-side SoD boundary and must be maintained accurately across the fleet.

## Open design questions to settle before Planning

> **PLANNING IS BLOCKED pending `auth`'s locked identity/token model.** The Gateway's MCP-surface authorization and its per-holder token audiences depend directly on decisions `auth` has flagged as open (its Stage-1 §"Open design questions": IdP selection bake-off; per-agent signing-key lifecycle; **multi-audience token acquisition across the SoD chain** — how one agent obtains four per-resource tokens for Board+CMDB+Vault+Gateway without any token spanning approve+execute; the kill-switch token strategy; and whether `aud` is per-resource-specific). Until `auth` locks the identity model, the Gateway cannot finalize how it validates `gateway:execute`, how the Board approval is bound/signed, or how Vault handles are audience-scoped. Do not enter Stage-2 Planning on the Gateway's authorization surface before that dependency resolves.

1. **Embed vs shell-out for ansible-runner:** in-process (simplest single container, but templating runs in the Gateway's own address space) vs a podman container per run (stronger blast-radius containment at cost of complexity).
2. **Where the plan-id → playbook/SLS mapping lives and who signs it:** it must be admin-authored and outside agent write to preserve SoD — likely Board/CMDB-owned, not Gateway-local config; and are per-plan `extravars` schemas part of the "approved plan" artifact the Board holds (so approval covers the exact parameter allowlist, not just the playbook)?
3. **SSH transport role:** relaxed-C restricted key as belt-and-suspenders beneath ansible-runner's own SSH (recommended), vs paramiko-under-wrapper as primary transport; and Vault SSH **signed certificates** vs key/password dynamic secrets for host exec.
4. **Snapshot responsibility:** Gateway-invoked per host, or a CMDB-declared precondition (hosts with no snapshot capability ⇒ ask-tier/manual)? And the bounded wait-for-SSH / health-check timeout policy per criticality tier (CMDB-driven).
5. **Mutex semantics:** reject-immediately (the Board holds ordering; the Gateway is not a queue — recommended) vs queue; strictly per-`host_id` vs per-`(host_id, resource)`; and whether to add a monotonic fencing token now (cheap Postgres sequence) as HA future-proofing.
6. **Execution-window contract:** if the CMDB window closes mid-run, does the Gateway abort-and-roll-back in place or finish the current atomic step?
7. **Confirmation policy:** is states-index doc-absence alone sufficient evidence of "solved," or require doc-absence AND a corroborating Solved alert when available; and how long to poll (tie the timeout to the host's actual Syscollector interval read from CMDB/agent config) before escalating to `needs_review`.
8. **Audit substrate:** Postgres append-only table primary + git mirror, vs git repo primary with SQLite index; checkpoint/anchor interval and off-box destination (mission-control? operator? second host?); whether kill-switch and mutex events are first-class chained records (recommended); inline full stdout/stderr vs content-addressed blob reference (large patch output bloats the chain); and a signed, verifiable rotation/archival scheme preserving chain continuity.
9. **Kill-switch terminal semantics:** does a killed run report `cancelled` or `failed` in the ticket lifecycle, and does the switch cancel in-flight MCP tasks in addition to halting the executor (belt-and-suspenders recommended)?
10. **Task read tool:** raw MCP `tasks/get` vs a Gateway-owned `get_execution_status(run_id)` wrapper (recommended — decouples the agent contract from the 2025-11-25→2026-07-28 task-method churn).
